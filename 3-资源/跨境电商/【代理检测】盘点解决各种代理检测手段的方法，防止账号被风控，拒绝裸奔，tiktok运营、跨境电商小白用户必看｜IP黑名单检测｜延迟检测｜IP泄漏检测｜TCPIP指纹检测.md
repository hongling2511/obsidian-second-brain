## 一句话总结
系统盘点四类代理检测原理（IP、网络行为、配置、系统环境）及防风控绕过与终极远程桌面解决方法。

## 目标受众与前置知识
- **适合谁看**：tiktok运营、跨境电商小白用户及需要防止账号被风控、拒绝“裸奔”的代理使用者。
- **前置知识**：[[代理]]、[[IP地址]]、[[TCP]]、[[UDP]]、[[DNS解析]]、[[WebRTC]]、[[vps]]

## 核心观点
1. **代理检测包含四大基本维度**：不仅查IP本身，还查网络行为、代理配置和系统环境。
   - 为什么重要：多数人只关注IP是否干净，忽略其他维度导致被高风控网站检测。
   - 依据：视频展示网页的严格检测状态，除IP库对比外，还包含延迟检测、TCP/IP指纹等非IP维度的验证手段。
2. **网络协议延迟差值可暴露代理行为**：[[TCP]]与[[websocket]]连接延迟差距过大会被判定为代理。
   - 为什么重要：揭示了物理距离与节点选择对隐蔽性的决定性影响。
   - 依据：讲者演示经美国节点访问美国网站，TCP延迟在70毫秒徘徊，WS最低延迟为250毫秒（讲者原话），因相差过大被标红；换用香港节点后差值缩小即成功绕过。
3. **本地代理配置不当会直接泄露真实网络身份**：[[WebRTC]]通信默认通过UDP传输，易越过仅代理TCP的配置暴露本机的真实IP。
   - 为什么重要：即使节点IP非常干净，本地流量漏发也会立刻被目标网站判定为使用代理。
   - 依据：检测工具显示用户进行UDP通信的IP是16结尾，而TCP通信IP是12结尾（屏幕展示/讲者原话），两者不匹配被判定为代理。
4. **终极防检测方案是使用远程桌面**：直接操控安装了桌面系统的境外VPS可完美避开所有本地特征检测。
   - 为什么重要：为无法解决复杂系统特征（如iOS无对应节点）或嫌麻烦的小白用户提供了最彻底的降维打击方案。
   - 依据：讲者说明远程操作等同于在当地实体机直接上网，不存在本地到代理节点的流量转发过程，自然不产生代理特征。

## 概念关系
- [[hosting]] → [[代理检测]]：如果购买的[[vps]]其IP对应的公司主营业务是数据中心托管（hosting），即使IP不在黑名单，也会被判定为使用了代理。
- [[物理距离]] → [[延迟差值]]：本地电脑到代理节点距离越近（如使用香港节点），TCP与WS的延迟差值越小，越容易绕过网络行为异常检测。
- [[操作系统]] ↔ [[TCP/IP指纹]]：不同操作系统（如[[Windows]]与[[Linux]]）建立TCP连接的默认参数（TTL、滑动窗口等）不同，网站借此对比HTTP头判断环境是否造假。

## 详细笔记
### 第一类：代理IP自身问题检测
- **黑名单（blocklist）检测**：节点IP存在于网站调用的IP数据库黑名单中。由于机场节点、便宜[[vps]]、伪住宅IP被多人滥用（发垃圾邮件、钓鱼诈骗等），会被各大公司（如[[ipinfo]]）拉黑。
  - **涉及实体**：检测项包括是否为大中型VPN商家（如[[Express VPN]]、[[vpsofsock]]等）、是否为[[Tor]]节点。
  - **影响**：对风控严格的网站如[[tiktok]]、[[facebook]]、[[paypal]]会导致账号出问题。
  - **解决方案**：换一个没被拉黑的干净IP。
- **数据中心（Data center）检测**：即使IP不脏，只要商家的业务类型为[[hosting]]，就说明该IP由数据中心托管，极易获取，会被标记。
  - **解决方案**：寻找类型为business或isp的商家购买[[vps]]。

### 第二类：网络行为异常检测
- **延迟异常检测**：网站检测[[TCP]]连接和[[websocket]] (ws) 连接的延迟差。
  - **演示数据**：无代理直连美国网站假设皆为150毫秒。使用美国节点后，TCP记录节点到网站耗时（如5毫秒），WS记录全程耗时（本地到美国150+节点到网站15=165毫秒），差值过大被标记为代理。讲者实际演示美国节点TCP 70毫秒，WS 250毫秒（讲者原话）。
  - **解决方案**：缩短差值。如改用香港节点，香港到美国TCP 211毫秒，本地到香港20毫秒，总WS延迟233毫秒（讲者原话），差值小成功绕过。
- **DNS解析行为检测**：网站随机访问不存在的域名。
  - **正常/异常对比**：正常会立即返回空解析报错；存在代理时可能等待3秒后报错超时（讲者原话）。
  - **工具提及**：[[小火箭]]配置模式会导致此项超时，代理模式则正常。

### 第三类：代理配置不当问题（泄漏本地或上游IP）
- **WebRTC泄露**：[[WebRTC]]默认通过[[UDP]]传输。若系统代理模式或浏览器插件模式只将[[TCP]]交给节点，[[UDP]]直连就会暴露真实IP。
  - **演示数据**：检测出UDP通信IP（16结尾）与TCP通信IP（12结尾）不一致（讲者原话），被标红。
  - **解决方案**：安装浏览器插件屏蔽WebRTC通信；或使用软路由/[[tun]]模式全局接管流量。
- **DNS泄露**：代理检测网站发现访客和节点均在美国，却由中国DNS提供商处理请求，明显不合理。前端源码中有此检测项，但当前网页暂未实施。

### 第四类：系统环境不符问题
- **TCP/IP指纹（tcp/ip fingerprint）检测**：各操作系统建立TCP连接的默认参数（如序列号、[[flag]]标志位、滑动窗口大小、[[ttl]]默认值等）不同。
  - **演示案例**：讲者本地使用[[Mac]]系统，但TCP参数计算出节点是[[chromeos]]（[[chromum]]系统概率最高）被标红。
  - **解决方案**：通过插件修改浏览器的[[user agent]]匹配目标系统；或本端与节点端使用相同系统（如本端[[Windows]]/[[Linux]]连对应系统的节点）。注：[[iOS]]系统无解，因为无法购买iOS的vps。
- **时区问题**：本机时区（[[上海]]）与节点IP所在地（[[香港]]）虽同属UTC+8但非同时区，导致被标记。
  - **解决方案**：使用浏览器插件、修改系统时区与IP一致，或使用指纹浏览器。
- **终极防风控方案**：讲者`不良林`针对无法完全隐藏的代理特征，建议给VPS安装[[Windows]]桌面操作系统，通过远程桌面连接操控。此法直接在远端生成流量，不产生任何代理特征。

## 步骤拆解
1. **解决 IP 黑名单与类型检测**
   - 具体操作描述：查询代理 [[IP]] 是否在业务黑名单中，并检查服务商类型。若是数据中心托管（hosting）类型，需更换服务商，购买公司类型为 business 或 isp 的 [[VPS]] 节点 [2, 8]。
   - 关键代码/命令：未明确提及。
   - 注意事项：廉价的 [[VPS]] 自建节点、机场节点以及伪住宅 [[IP]] 容易因为他人的滥用行为（如批量注册、发垃圾邮件）而被列入各大 [[IP]] 数据库黑名单 [2, 9]。

2. **绕过网络延迟检测**
   - 具体操作描述：缩小建立 [[TCP]] 连接与建立 [[ws]] 连接的延迟差值。通过改变节点和网站之间的物理距离，使用靠近本地物理位置的节点（例如人在中国访问美国网站，使用香港节点而非美国节点） [3, 10]。
   - 关键代码/命令：未明确提及。
   - 注意事项：正常直连情况下 [[TCP]] 和 [[ws]] 延迟相近，若使用代理，本地到节点再到目标网站的 [[ws]] 总耗时与节点到目标网站的 [[TCP]] 耗时差距过大，会触发代理检测 [3]。

3. **修复本地 IP 泄露 ([[WebRTC]] 泄露)**
   - 具体操作描述：避免 [[WebRTC]] 通过直连发送 [[UDP]] 数据。可以通过安装浏览器插件屏蔽 [[WebRTC]] 通信，或者使用 [[软路由]] 或 [[TUN模式]] 接管系统全局流量，将 [[UDP]] 数据也交由节点处理 [4]。
   - 关键代码/命令：未明确提及。
   - 注意事项：如果仅使用 [[系统代理]] 模式或浏览器安装代理插件，无法将 [[UDP]] 流量交给节点，会导致真实 [[IP]] 暴露 [4]。

4. **伪装系统指纹与时区**
   - 具体操作描述：通过插件修改浏览器的 User-Agent 以匹配节点服务器的操作系统；或直接在与本地物理机对应的操作系统上搭建节点（例如在 [[Windows]] 上建节点并用本地 [[Windows]] 连接）。另外，使用插件或系统设置，将本地时区改为与节点 [[IP]] 地理位置一致，或直接使用 [[指纹浏览器]] [6, 7]。
   - 关键代码/命令：未明确提及。
   - 注意事项：检测网站会通过 [[TCP]] 数据包的序列号、flag 标志位、滑动窗口大小以及 TTL 默认值来判断操作系统（[[TCP/IP指纹]]），若与 HTTP 请求中携带的系统信息不符，会被标记为代理 [6]。

5. **配置终极免检测方案**
   - 具体操作描述：给 [[VPS]] 安装 [[Windows]] 桌面操作系统，通过 [[Windows远程桌面]] 的方式连接 [[VPS]] 进行远程操控 [7]。
   - 关键代码/命令：未明确提及。
   - 注意事项：该方式相当于直接在远程电脑上操作，不会产生上述任何代理特征 [7]。

## 常见坑与解决方案
- **问题描述**：使用 [[小火箭]] 的配置模式会导致访问不存在的域名时发生超时，触发网络行为异常检测 [10]。
  - 解决方案：切换使用 [[小火箭]] 的代理模式 [10]。
- **问题描述**：[[WebRTC]] 通信导致真实 [[IP]] 泄露，[[TCP]] 通信和 [[UDP]] 通信使用的 [[IP]] 不一致 [4]。
  - 解决方案：安装浏览器插件屏蔽通信，或使用 [[软路由]] / [[TUN模式]] 全局接管 [[UDP]] 流量 [4]。
- **问题描述**：[[TCP/IP指纹]] 暴露的操作系统与 User-Agent 声明的系统不符（如节点是 [[Linux]]/[[ChromeOS]]，本地是 [[Mac]]） [6]。
  - 解决方案：修改浏览器 User-Agent，或在与本地机型相同的系统上自建节点；若是 [[iOS]] 环境则无解（因无法购买 [[iOS]] 系统的 [[VPS]]） [6, 7]。
- **问题描述**：系统本地时区与代理节点 [[IP]] 的时区不一致（即使同为 UTC+8，地区不同也会被检测） [7]。
  - 解决方案：修改系统时区使其与节点一致，或使用浏览器插件、[[指纹浏览器]] 进行规避 [7]。

## 最佳实践
- **推荐做法**：如果不想暴露任何代理特征，最彻底的做法是给 [[VPS]] 安装并使用 [[Windows远程桌面]] 进行远程操控 [7]。
- **反模式**：在运营风控等级高的网站（如 TikTok、Facebook、PayPal）时，使用多人共用的机场节点、公共代理或门槛极低的廉价 [[VPS]]，这会导致账号面临极高的被风控或封禁风险 [1, 2, 9]。

## 关键概念

### [[代理 IP 黑名单]]
- **定义**：指代理 IP 因为存在搭建公共代理、批量注册或诈骗等滥用行为，而被各大 IP 数据库公司收集并列入的屏蔽检测名单 [1, 2]。
- **视频上下文**：作者指出这是最常见的代理检测手段，网站通过调用数据对比发现访客的 IP 存在于黑名单中，从而判定其正在使用代理 [1, 2]。
- **为什么重要**：如果使用在黑名单中的不干净 IP 访问高风控要求的网站，大概率会导致账号出问题，且唯一的解决办法是更换未被拉黑的 IP [2]。
- **关联概念**：扩展 → [[伪住宅 IP]]

### [[伪住宅 IP]]
- **定义**：指被很多人共同当作住宅 IP 使用的代理节点 IP [1]。
- **视频上下文**：作者在讲解代理 IP 被屏蔽的原因时提到，使用伪住宅 IP 很有可能因为其他人的滥用行为而被连累列入黑名单 [1]。
- **为什么重要**：它揭示了即使使用了宣称为住宅用途的节点，也可能因为共享历史和滥用记录而无法绕过基础的 IP 代理检测 [1, 2]。
- **关联概念**：属于 → [[代理 IP 黑名单]]

### [[Hosting IP]]
- **定义**：由主营数据中心托管业务（hosting）的公司所分配的代理 IP 地址 [3]。
- **视频上下文**：作者解释即使 IP 不在黑名单内，只要负责分配该 IP 的商家主营业务是数据中心托管，网站的 Data center 测试项就会将其判定为代理 [3]。
- **为什么重要**：这类 IP 最显著的特点是容易获取，若要绕过此项严格检测，必须寻找公司类型为 business 或 ISP 的商家购买节点 [3]。
- **关联概念**：未明确提及

### [[延迟检测]]
- **定义**：通过对比客户端与网站建立 TCP 连接和 WebSocket 连接的耗时差值，来识别网络通讯行为是否异常的检测方式 [3, 4]。
- **视频上下文**：作者举例指出，使用美国节点访问美国网站时，节点到网站的 TCP 延迟极低，而本地电脑经过节点中转的 WebSocket 延迟极高，当两者的差值过大时即被判定为代理 [4]。
- **为什么重要**：这是一种基于物理距离和网络行为异常的高级检测手段，必须通过拉长节点与网站的延迟并缩短本地到节点的延迟来绕过 [4]。
- **关联概念**：未明确提及

### [[代理配置不当]]
- **定义**：指代理工具的网络路由规则设置存在漏洞，导致本机真实网络信息或上游 IP 泄露给目标网站的分类问题 [5, 6]。
- **视频上下文**：作者将代理检测盘点为四种主要类型，其中第三类即为代理配置不当，这种问题通常会导致网站发现访客访问网站的 IP 与本机暴露的真实 IP 不一致 [6, 7]。
- **为什么重要**：这提醒用户不仅要关注代理 IP 的自身干净程度，还要确保代理系统正确且全面地接管了各类底层协议的数据流量 [6]。
- **关联概念**：扩展 → [[WebRTC 泄露]]

### [[WebRTC 泄露]]
- **定义**：因代理配置漏洞，导致默认通过 UDP 传输的 WebRTC 通信绕过节点直连，进而暴露本地真实 IP 地址的现象 [6]。
- **视频上下文**：作者说明，当代理仅处理 TCP 数据而让 UDP 通信直连时，网站会发现进行 UDP 通信的真实 IP 与 TCP 通信的代理 IP 不一致，从而将访客标记为代理用户 [6]。
- **为什么重要**：这是系统代理模式和浏览器插件模式常见的配置盲区，必须通过安装插件屏蔽 WebRTC 或使用软路由等模式接管全局 UDP 流量来防范 [6]。
- **关联概念**：属于 → [[代理配置不当]]

### [[DNS 泄露]]
