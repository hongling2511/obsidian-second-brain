## 一句话总结
[[不良林]]详细讲解[[open clash]]的基础配置指南，并深度剖析[[redirect]]、[[tproxy]]与[[turnel]]模式实现透明代理的工作原理。

## 目标受众与前置知识
- **适合谁看**：前半部分适合不关心底层原理、只想跟着操作的普通用户；后半部分适合想要深究代理网络原理的进阶用户。
- **前置知识**：软路由基础配置（推断）、[[旁路由]]概念（推断）、[[tcp]]/[[udp]]网络传输协议基础（推断）、Linux[[防⽕墙]]基础知识（推断）。

## 核心观点
1. **针对不同隐私需求可选择不同代理规则配置**
   - 为什么重要：满足普通用户上外网需求的同时，保障高隐私要求用户的网络安全。
   - 依据：讲者原话提及，普通用户默认配置即可，但对隐私要求高或做跨境电商的用户，推荐使用白名单模式以减少平台潜在的检测因素。
2. **代理[[udp]]流量必须使用[[tproxy]]或相关混合模式，不能仅用[[redirect]]**
   - 为什么重要：解释了许多用户遇到的部分应用（如基于[[udp]]的[[quck]]协议）无法代理的问题。
   - 依据：讲者原话指出，“[[udp]]是无连接的，无法获取到数据包原的目标IP，所以[[redirect]]无法帮我们代理[[udp]]数据”，而[[tproxy]]不改变数据包本身，可轻易获取原始目标。
3. **[[turnel]]（TUN）模式不依赖防火墙规则，仅靠路由规则实现透明代理**
   - 为什么重要：提供了一种兼顾局域网和本机代理的旁路由实现方式。
   - 依据：讲者原话说明该模式会创建虚拟网卡[[utun]]，通过添加自动路由规则将数据转发至虚拟网卡，由[[System]]或[[gviser]]协议栈处理拆包解析。

## 概念关系
- [[redirect]] ↔ [[tproxy]]：[[redirect]]通过修改数据包目标IP（即[[dnet]]）代理[[tcp]]或劫持DNS；[[tproxy]]不修改数据包的目标IP和端口，通过打[[mark]]用于代理[[udp]]。
- [[iptables]] ↔ [[nf tables]]：两者均为防火墙工具，用于在链路中创建Hook拦截数据。在[[imoto wrt23.05]]固件中，[[nf tables]]代替了[[iptables]]。
- [[System]] ↔ [[gviser]]：两者均为[[turnel]]模式下的数据协议栈解析方式。[[System]]使用系统自带栈兼容性更好；[[gviser]]为[[clash]]内部协议栈，理论上无多余重复拷贝性能更好。
- [[fakeip]] ↔ [[re host]]：两者均为[[clash]]处理DNS的模式。原版[[clash]]已弃用存有问题的[[re host]]，但在[[Math内核]]中对其作了增强并保留。

## 详细笔记

### 一、 [[open clash]]基础配置指南（普通用户向）
- **环境准备**：使用[[imoto wrt23.05]]固件，安装并进入[[open clash]]。如为旁路由结构且国内网站无法访问，需勾选开启L区域的IP动态伪装。
- **内核下载**：在版本更新页下载内核。若下载失败，可更换下载地址，或手动将[[Math内核]]/[[promum]]内核上传至路由器。
- **DNS配置**：取消所有默认DNS服务器勾选，仅勾选「自定义上游DNS」与「追加上游DNS」。
- **节点订阅**：在配置订阅中添加机场订阅。若格式不兼容，需勾选在线订阅转换。讲者提醒使用他人订阅转换有隐私和远程执行漏洞风险，建议用[[CF]]的[[wer]]自建反代（后缀加`/sub`）。
- **启动与验证**：配置完成后启动插件，界面显示绿色运行中。关闭[[clash for windowows]]等电脑端代理，直连访问[[谷歌]]和[[YouTube]]进行验证。

### 二、 白名单模式与DNS泄露处理
- **白名单配置**：对隐私要求高的用户，建议在在线订阅转换中选择自定义模板。该模式下命中规则的直连，其余交由节点处理。
- **小众网站与[[udp]]直连**：部分国内小众网站或系统级[[udp]]请求（如[[Windows]]时间同步域名）可能会走代理且无下载数据导致失败。
- **操作步骤**：新建自定义规则（Type为`file`，格式`text`），录入需直连的域名，指定策略为`Direct`。将该规则文件在“规则附加”中应用并重启插件生效。

### 三、 家庭网络拓扑与DNS重定向原理解析（进阶用户向）
- **常规网络链路**：光猫拨号获取公网IP，路由器通过[[pppoe]]连接并作为网关通过[[dhcp]]分配内网IP。数据请求需先经防火墙（在[[prting]]、[[Input]]、[[output]]、[[forward]]、[[post routing]]处拦截）。
- **DNS重定向（[[redirect]]）**：所有发往53端口的UDP数据被重定向。
  - 防火墙转发：重定向至[[clash]]监听的7874端口（讲者原话）。
  - [[dnsmq]]转发：重定向至路由器当前接口53端口，再通过上游DNS设置转至[[clash]]的7874端口。
- **获取IP返回**：[[clash]]获取[[谷歌]]IP后经[[output]]和路​​由后发回电脑，连接追踪会将IP和端口修改回原样。

### 四、 [[redirect]]代理[[tcp]]流程
- **拦截[[tcp]]**：电脑发起[[http]]请求，防火墙在[[prting]]（路由前）将[[tcp]]数据重定向到7892端口（即流​​量转发端口）。
- **路由决策**：目标IP被改为路由器本身，数据进入[[Input]]（输入链）交给7892端口的[[clash]]。
- **转发至节点**：[[clash]]获取原始IP，根据分流规则加密后通过[[eth0]]发出。为防止节点返回数据引发死循环，防火墙会添加规则：目标IP是发给本机的直接return返回（讲者原话）。

### 五、 [[tproxy]]代理[[udp]]与本机流程
- **代理局域网[[udp]]**：由于[[udp]]无连接，改用[[tproxy]]模块。所有[[udp]]数据转发给7895端口，并打上[[mark]]（标​​记为354，讲者原话）。数据包本身目标IP未改。
- **策略路由**：防火墙添加路由规则，要求带打过标（如354）的数据包交由100号路由表（讲者原话）处理，强制路由到本机[[Input]]链。
- **代理本机流量**：本机发起的请求直接走[[output]]链，错过路由前拦截。解决方式是在[[output]]链给数据打[[mark]]，触发重新路由（匹配100号路由表），通过回环口转至本机再次进入路由前拦截。讲者提到，对已建立[[tproxy]]连接的[[socket]]可直接打[[mark]]处理。

### 六、 [[turnel]]模式（TUN）与混合模式
- **[[turnel]]模式原理**：不依赖防火墙，创建虚拟网卡[[utun]]。利用自动路由规则（基于优先级匹配，例如排除本地[[local]]表、排除53端口走[[man]]表等），将目标包引流至[[utun]]接口。
- **数据解包与封装**：需确保设备开启IP转发。数据经[[eth1]]转至虚拟网卡，由[[clash]]通过[[System]]或[[gviser]]协议栈拆包，获取意图后经物理网卡[[eth0]]发往节点。
- **混合模式**：利用[[redirect]]传输[[tcp]]，利用[[tproxy]]传输[[udp]]，以平衡[[turnel]]模式中[[tcp]]性能较低和部分人[[tproxy]]处理[[udp]]不稳定的问题。

### 七、 防火墙与路由规则排查
- **排查命令**：
  - 讲者演示通过命令行查看实际规则。
  - 使用 `ip a` 查看网络接口（包含[[utun]]与[[fakeip]]段）。
  - 使用 `ip rule` 查看策略路由规则。
  - 使用 `ip route show table 354` 等命令查看特定路由表（讲者原话）。

## 常见坑与解决方案
- **问题描述**：内核下载失败，日志提示没有任何反应。
  - 解决方案：通常是因为下载地址被墙，可根据提示在覆写设置中修改下载地址，或者将内核下载到电脑本地后通过配置管理上传 [9, 12]。
- **问题描述**：旁路由结构下配置好后出现国内网站无法访问。
  - 解决方案：勾选相关选项（相当于开启 LAN 区域的 IP 动态伪装） [9]。
- **问题描述**：Windows 系统时间无法同步（因相关域名基于 UDP 且节点 UDP 不通）。
  - 解决方案：在控制面板的连接页面将该 UDP 请求的域名过滤出来，并将其添加到自定义的直连（DIRECT）规则集中 [16, 19]。
- **问题描述**：使用 redirect 模式无法代理 UDP 数据（因 UDP 无连接，无法获取原始目标 IP）。
  - 解决方案：使用 `[[TProxy]]` 透明代理模块（需开启 UDP 流量转发），或使用混合模式、`[[TUN]]` 模式 [6, 7, 18, 20]。

## 最佳实践
- 推荐使用 `[[Fake-IP]]` 模式处理 DNS，因为原版 `[[Clash]]` 已弃用 Redir-Host 模式；如必须使用 Redir-Host，建议使用 `[[Clash.Meta]]` 内核以解决兼容性问题 [4, 5]。
- 推荐使用 Cloudflare Worker 自己搭建一个反代订阅转换服务，以避免使用他人在线转换带来的隐私泄露或远程执行漏洞风险（反模式：直接使用第三方未经验证的在线订阅转换） [13, 14]。
- 对于个人隐私要求较高或从事跨境电商的用户，推荐使用白名单模式（仅命中规则直连，其余全代理）以减少平台潜在的检测因素 [15]。
- 建议用户自己维护一个规则列表，将常用的小众国内网站加入直连规则，避免其误走代理 [19]。

## 关键概念

### [[Redirect]]
- **定义**：一种仅能代理 TCP 数据和处理无需访问 UDP 目标的劫持行为，通过修改数据包目标 IP 和端口实现重定向的透明代理方式 [1-3]。
- **视频上下文**：作者在讲解 DNS 劫持和 TCP 流量代理时指出，Redirect 会在防火墙的路由前阶段将数据包的目标 IP 重定向到路由器当期接口 IP 的特定端口 [1, 4]。
- **为什么重要**：它是 OpenClash 中处理 TCP 数据和 DNS 劫持的基础机制，但具有无法获取 UDP 原始目标 IP 从而无法代理 UDP 数据的局限性 [3]。
- **关联概念**：对比 → [[Tproxy]]

### [[Tproxy]]
- **定义**：一种既能代理 TCP 也能代理 UDP，且不改变数据包本身目标 IP 和目标端口的透明代理模块 [3, 5]。
- **视频上下文**：作者解释因为 Redirect 无法代理 UDP，所以必须使用 Tproxy 处理 UDP 数据，它通过在防火墙打 mark 标记配合特定路由表，将数据转交本机代理端口处理 [3, 5, 6]。
- **为什么重要**：它解决了 Redirect 模式无法有效代理 UDP 流量的痛点，是兼容模式或增强模式下 UDP 代理的核心机制 [3, 6]。
- **关联概念**：对比 → [[Redirect]]

### [[TUN]]
- **定义**：一种不依赖防火墙规则，通过创建虚拟网卡并利用路由规则实现透明代理的模式 [7-9]。
- **视频上下文**：作者介绍 TUN 模式通过自动添加路由规则，将非本机发出的数据包转入虚拟网卡 (utun)，再由 Clash 读取并利用协议栈拆包解析出访问意图 [8-10]。
- **为什么重要**：它提供了一种替代防火墙打 mark 的全局代理方案，且允许作为旁路由直接使用，但需要确保设备开启 IP 转发 [9-11]。
- **关联概念**：对比 → [[Redirect]]

### [[Fake-IP]]
- **定义**：一种代理工具的 DNS 处理模式，原版 Clash 内核目前仅保留了此模式 [12]。
- **视频上下文**：作者在介绍 DNS 处理方式时指出，由于 Redir-host 模式存在各种问题已被原版 Clash 弃用，建议用户使用 Fake-IP 模式，增强模式就是使用此模式 [6, 12]。
- **为什么重要**：它是当前更受推崇且问题较少的 DNS 处理与代理配合方式 [12]。
- **关联概念**：对比 → [[Redir-host]]

### [[Redir-host]]
- **定义**：一种将处理 UDP 请求的域名先解析成真实 IP 的 DNS 处理模式 [12, 13]。
- **视频上下文**：作者指出由于该模式存在各种问题，Clash 内核已弃用，但 OpenClash 依然保留了从 Fake-IP 回退到 Redir-host 的功能，如果非要使用建议搭配 Meta 内核 [12, 14]。
- **为什么重要**：了解此模式有助于排查部分基于 UDP 的应用（如 Windows 系统时间同步）解析真实 IP 时的不通问题 [12, 13, 15]。
- **关联概念**：对比 → [[Fake-IP]]

### [[防火墙]]
