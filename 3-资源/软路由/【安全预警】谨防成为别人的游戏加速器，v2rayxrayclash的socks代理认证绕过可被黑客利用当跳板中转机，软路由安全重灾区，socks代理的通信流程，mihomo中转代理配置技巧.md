## 一句话总结
[[v2ray]] 与 [[clash]] 因未规范分配 [[udp]] 端口，其公网 [[socks代理]] 易被黑客绕过认证当跳板。

## 目标受众与前置知识
- **适合谁看**：使用 [[软路由]]、[[vps]] 搭建科学上网节点，或将其 [[socks代理]] 暴露在公网的用户。
- **前置知识**：[[socks代理]]、[[udp]]、[[tcp]]、[[软路由]]、[[vps]]、[[端口转发]]

## 核心观点
1. **[[v2ray]] 和 [[clash]] 系的 [[socks代理]] 存在严重的认证绕过风险**
   - 为什么重要：导致小白或配置不当用户的节点被恶意利用，成为别人的白嫖加速器或非法中转跳板。
   - 依据：讲者通过扫描获取了“将近5000条结果”（讲者原话）的中国区暴露用户，并成功演示了绕过API鉴权和密码认证，利用他人的香港、日本节点代理自己的流量。
2. **风险根源在于未遵循 [[rfc规范]] 随机分配 [[udp]] 端口**
   - 为什么重要：[[udp]] 本身无认证机制，固定端口使得黑客无需通过 [[tcp]] 认证即可直接强行转发 [[udp]] 流量。
   - 依据：讲者指出，标准规范中 [[tcp]] 验证后应随机监听空闲 [[udp]] 端口，但 [[v2ray]] 和 [[clash]] 采用固定的同号端口（例如都是7890），从而被轻易猜测和利用。
3. **暴露公网下的纯 [[udp]] 漏洞可结合中转协议造成巨大流量损失**
   - 为什么重要：攻击者可利用基于 [[udp]] 的协议（如 [[Hysteria]]）通过受害者节点中转加速，消耗受害者的高昂专线流量。
   - 依据：讲者演示了通过获取 [[x-ui]] 面板“4万多条结果”（讲者原话）并用 [[nmap]] 扫描随机端口，成功白嫖配置了“240倍”（屏幕展示）费率的 [[阿里]] 专线流量。

## 概念关系
- [[rfc规范]] ↔ [[v2ray]] / [[clash]]：[[v2ray]] 与 [[clash]] 未遵循 [[rfc规范]] 随机分配 [[udp]] 端口，导致端口固定且可预测。
- [[socks代理]] → [[udp]] 认证缺陷：[[socks代理]] 的 [[udp]] 流量本无密码认证，完全依赖前置的 [[tcp]] 验证步骤，由于端口预测防线形同虚设，导致被绕过。
- [[Hysteria]] → 中转跳板：由于单纯代理 [[udp]] 作用有限，黑客利用基于 [[udp]] 的 [[Hysteria]] 协议，将受害者的节点作为中转跳板加速自己的节点。

## 详细笔记
### 现象揭示：公网代理被大范围扫描与利用
- 讲者（不良林）指出，诸多使用 [[软路由]]、电脑直接拨号在 [[dmz区域]] 或在 [[vps]] 上使用 [[x-ui]] 搭建节点的用户，无意中将 [[socks代理]] 暴露在公网。
- 即使设置了密码或API鉴权，黑客依然可以绕过。讲者展示通过特定链接能搜出近5000条中国区暴露的 [[clash]] 用户（若用 [[Fofa]] 搜索会更多）。
- 讲者利用 [[python]] 脚本扫描 [[OpenClash]] 默认的7891端口，获取了大量可用IP。
- 随后配置并运行 [[mihomo]]（原名 clash meta），将本地软件的 [[udp]] 数据（通过 [[net检测工具]] 观察）直接发送到扫描到的目标IP端口。结果显示，成功绕过对方的安全限制，返回了对方节点（如香港、日本节点）的IP，实现了直接白嫖。

### 原理解析：为何密码认证形同虚设
- 标准流程（基于 [[rfc规范]]）：[[浏览器]]与 [[clash]] 建立 [[tcp]] 连接验证密码后，应随机监听一个空闲的 [[udp]] 端口（如6666）供后续传输。
- 漏洞原因：[[v2ray]] 和 [[clash]]（包括衍生的 [[xray]] 等核心）未按规范实现，其 [[udp]] 端口与 [[tcp]] 端口固定一致（如都是7890）。
- 攻击链路：因为 [[socks代理]] 的 [[udp]] 本身没有认证机制，黑客获取到公网IP和端口后，无需知道账号密码，直接将 [[udp]] 请求发往这个固定的代理端口，受害者的 [[clash]] 即会无条件帮忙转发数据。
- 讲者提到 [[sing-box]] 是严格遵守规范每次分配随机端口的，因此不存在此问题。

### 进阶危害：组合中转协议白嫖昂贵专线
- 讲者解释单纯利用 [[udp]] 用途不大（仅限于游戏和 [[quic]] 网页），但黑客可以利用纯 [[udp]] 的代理协议（如 [[Hysteria]]）实现跳板加速。
- 演示步骤：在 [[vps]] 搭建带有 [[bbr]] 的 [[Hysteria]] 节点，配置 [[clash]] 将流量先交给本地 [[socks]] 节点，再由其把纯 [[udp]] 数据直接发给受害者的 [[clash]] 完成中转。
- 讲者指出很多用 [[x-ui]] 搭建 [[socks]] 的用户也会暴露（被扫出4万多条结果），并针对高价值目标（如 [[搬瓦工]] 的 [[cn2gia线路]]）进行攻击。
- 此外，盲目用中转机加速本机的 [[socks代理]] 是无效的，因为 [[udp]] 流量会绕过中转直连 [[vps]]。讲者通过购买“240倍”流量的 [[阿里]] 专线演示，利用 [[nmap]] 扫描出专线随机分配的端口，成功连上并白嫖了这条极贵专线的流量来加速自己的 [[Hysteria]] 节点。

### 防护与修复：如何确认并阻断风险
- **检测暴露情况**：访问特定网站获取本机的公网IP，再通过端口扫描网站查询该IP对应的 [[socks代理]] 端口（[[v2n]] 默认10810，[[Clash for Windows]] 默认7890，[[OpenClash]] 默认7891/7893）是否开放。
- **解决方案**：
  - [[软路由]] 用户：进入防火墙设置，将 WAN 区域入站改为“拒绝”；若需保留其他公网服务，可在通信规则中单独添加拒绝源区域为 WAN 对本机 [[socks]] 端口（[[udp]]/[[tcp]]）的访问。
  - 端口转发用户：建议仅开放 [[tcp]] 端口或直接关闭 [[socks]] 的端口转发。
  - 桌面端用户：除拨号或处于 [[dmz区域]] 外，被利用概率较小，需自行搜索配置 [[系统防火墙]]。
  - [[x-ui]] 用户：不建议搭建 [[socks代理]]，若必须搭建坚决不要开启 [[udp]]。
  - 终极方法：直接更换为遵守规范的 [[sing-box]]。

## 金句亮点
- “这怎么会是漏洞呢，你只要把家里的网线拔了，别人就没法利用了，会被利用都是用户自己配置的问题，所以这不是漏洞是 feature。” — 观点分享 / 警示（推断讲者在此处使用反讽语气强调人为配置失误的危害性）
- “如果你有非搭建不可的原因，建议不要开启 udp；但如果你又非要开启 udp 的话，那就自求多福吧。” — 行动建议 / 警告

## 行动建议
- **检测公网暴露风险**：关联观点 1。访问端口扫描网站，输入自家公网IP及代理软件的默认 socks 端口进行查询。验证标准：扫描结果显示端口为“关闭”状态。
- **修改软路由防火墙规则**：关联观点 2。进入软路由网络设置，添加针对本机 socks 端口拒绝外部访问的通信规则。验证标准：再次扫描该端口显示关闭，且其他正常公网服务不受影响。
- **更换合规代理软件**：关联观点 2。将系统底层的代理核心从 v2ray/clash 替换为 sing-box。验证标准：节点软件正常运行且严格按照 RFC 规范随机分配 udp 端口。

   - 具体操作描述：通过 [[python]] 脚本扫描特定的默认端口（如 [[OpenClash]] 的默认 [[socks]] 端口 7891），获取暴露在公网且符合条件的代理 IP 列表[2]。

2. **配置代理工具劫持 UDP 流量**
   - 具体操作描述：运行 [[mihomo]]（或 [[clash]]），将扫描获取到的目标 IP 替换进配置文件中。配置目的是关闭工具自身的 UDP 监听（设为 false），转而在下方规则中单独监听 [[socks]] 端口（如 7891），并将该端口收到的所有 UDP 数据通过直连方式转发给目标的 IP 和固定端口[2, 4]。
   - 关键代码/命令：未明确提及（仅提及 Target 目标 IP 和端口的替换配置）[2, 4]。
   - 注意事项：如果多次尝试连接不通，大概率是因为对方使用的节点不支持 UDP，需更换 IP 再次尝试[2]。

3. **借助纯 UDP 协议实现跳板中转**
   - 具体操作描述：在 [[clash]] 配置中创建两类节点，一类是本机 [[socks]] 节点（指向本地 7891 端口），另一类是基于纯 UDP 通信的 [[Hysteria]] 节点。将 [[Hysteria]] 的数据分流交给 [[socks]] 节点处理，由于目标代理未遵循 RFC 规范验证 UDP 且端口固定，UDP 数据会被目标代理无认证接收并转发，从而实现流量的跳板中转[3-5, 8]。
   - 关键代码/命令：未明确提及[4]。
   - 注意事项：此方法仅用于演示利用原理，实际经过中转后的网络速度可能还不如直接连接[4]。

4. **扫描并白嫖专线节点**
   - 具体操作描述：针对使用中转加速的游戏专线或高价值节点（如阿里专线、CN2 GIA），获取专线入口 IP 及随机分配的端口。使用 [[nmap]] 工具扫描该 IP 开放的端口范围（如 10000 到 50000），找出属于 [[socks]] 代理的端口后，使用 [[clash]] 连接并获取流量落地权[5, 6]。
   - 关键代码/命令：未明确提及（仅提及使用 [[nmap]] 扫描）[5, 6]。
   - 注意事项：被白嫖消耗的高昂专线流量费将会算在原节点搭建者的头上[6]。

5. **公网暴露自测与防火墙修复**
   - 具体操作描述：首先关闭本地翻墙软件并获取运营商分配的公网 IP。然后访问端口扫描网站，填入公网 IP 和使用的代理工具默认 [[socks]] 端口（如 [[v2rayN]] 的 10810、[[Clash for Windows]] 的 7890、[[OpenClash]] 的 7891/7893 等）进行查询。若显示开启，则进入软路由的防火墙设置，将 WAN 区域的入站规则改为拒绝；或在通信规则中添加单独的屏蔽规则，指定目标为 [[socks]] 端口并将操作改为拒绝[6, 7]。
   - 关键代码/命令：未明确提及[6, 7]。
   - 注意事项：直接拒绝 WAN 区入站会影响其他需要公网访问的服务，因此更建议通过通信规则单独屏蔽 [[socks]] 端口的流出或流入[7]。

## 常见坑与解决方案
- **问题描述**：使用 [[x-ui]] 搭建 [[socks]] 节点后，进一步使用中转机来加速 [[socks]] 代理，但发现 UDP 流完全绕过中转，起不到加速作用且易被利用[5]。
  - 解决方案：不要使用 [[socks]] 协议进行中转，由于其机制会直接返回本机 IP 作为 UDP 的入口，导致流向直发 VPS；建议关闭 UDP 或者直接更换为实现 RFC 标准规范的 [[sing-box]] 工具[5, 7]。
- **问题描述**：软路由防火墙将 WAN 区域入站直接改为“拒绝”后，导致家中其他正常的公网服务无法访问[6, 7]。
  - 解决方案：在网络防火墙的“通信规则”中添加新规则，将原区域设置为 WAN，目标端口填写 [[socks]] 端口（多个端口用空格隔开），仅单独屏蔽这些端口的 TCP/UDP 流量，操作选为拒绝即可[7]。

## 最佳实践
- **推荐做法**：
  - 如果必须暴露端口或需要公网访问，且注重安全性，建议更换并使用严格遵守 RFC 规范的 [[sing-box]]（每次连接 UDP 都会开启一个随机端口，从而避免被绕过认证）[3, 7]。
  - 如果通过软路由端口转发开放服务到公网，强烈建议只开放 TCP 端口，或者彻底关闭针对 [[socks]] 端口的转发规则[7]。
  - 软路由用户应利用防火墙通信规则，精准拦截特定 [[socks]] 端口的公网入站请求[7]。
- **反模式**：
  - 在完全不知情的情况下，将软路由暴露在公网，或在电脑直接拨号、放置在 DMZ 区域时开启 [[socks]] 代理[1, 3, 7]。
  - 以为给公网暴露的 [[socks]] 代理设置了密码认证就万事大吉（由于 [[v2ray]]/[[clash]] 系工具未按规范实现 UDP 随机端口，TCP 密码认证可被轻易绕过）[1, 8]。
  - 在 [[x-ui]] 面板上直接搭建且在公网暴露 [[socks]] 代理节点，同时开启 UDP 功能[5, 7]。

## 关键概念

### [[socks代理]]
- **定义**：一种网络代理服务，在本视频中因部分软件未按规范实现，导致其认证机制可被绕过 [1, 2]。
- **视频上下文**：作者指出许多软路由或 VPS 用户将其暴露在公网，并误以为设置了密码认证就万事大吉，实则正被黑客扫描利用 [1, 3]。
- **为什么重要**：它是引发本视频所探讨的安全漏洞（节点被白嫖、被当作跳板）的核心代理组件 [1, 2]。
- **关联概念**：属于 → [[中转跳板]]

### [[UDP]]
- **定义**：一种无连接的网络传输协议，在 socks 协议中本身没有直接的认证机制 [2]。
- **视频上下文**：视频详细解释了黑客由于抓住了部分代理软件的 UDP 端口固定的特点，无需账号密码即可让对方的节点乖乖代理 UDP 数据 [2, 3]。
- **为什么重要**：它是该认证绕过漏洞得以成功实施的关键流量载体 [3]。
- **关联概念**：对比 → [[TCP]]

### [[TCP]]
- **定义**：一种建立网络连接的协议，在正常的 socks 代理流程中负责处理账号密码验证 [2]。
- **视频上下文**：作者提到浏览器通常会先与代理软件建立 TCP 连接来完成密码认证，随后才分配端口处理 UDP 流量 [2]。
- **为什么重要**：说明了正常的安全防御机制，以及黑客利用漏洞绕过的正是这一层认证 [2, 3]。
- **关联概念**：对比 → [[UDP]]

### [[RFC规范]]
- **定义**：互联网标准文件，在此指代 socks 协议原本应遵循的标准代理流程和端口分配机制 [2]。
- **视频上下文**：作者指出 v2ray 和 clash 系软件未按该规范随机分配 UDP 端口，而是与 TCP 端口固定，从而造成了安全隐患 [2]。
- **为什么重要**：解释了这一安全问题产生的根本设计缺陷 [2, 3]。
- **关联概念**：对比 → [[sbox]]

### [[中转跳板]]
- **定义**：被他人利用来代理流量、隐藏真实 IP 或加速自己网络的中间代理服务器 [1, 4]。
- **视频上下文**：作者演示了如何通过配置文件，将自己的数据请求直接转发到公网上有漏洞的别家代理节点上，实现中转 [3, 4]。
- **为什么重要**：揭示了 socks 代理被破解后的实际危害和黑客的具体利用场景 [1, 4]。
- **关联概念**：依赖 → [[socks代理]]

### [[sbox]]
- **定义**：一种按照标准 RFC 规范实现 socks 代理的科学上网软件工具 [3]。
- **视频上下文**：作者指出该软件每次连接 UDP 都会开启随机端口，不存在认证被绕过的问题，并将其作为最简单彻底的解决方案推荐给用户 [3, 5]。
- **为什么重要**：提供了从根本上免疫此安全风险的替代性软件选择 [3, 5]。
- **关联概念**：扩展 → [[RFC规范]]
