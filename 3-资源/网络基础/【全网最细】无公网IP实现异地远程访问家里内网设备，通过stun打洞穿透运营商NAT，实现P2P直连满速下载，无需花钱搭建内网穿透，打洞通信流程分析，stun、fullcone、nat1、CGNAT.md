## 一句话总结
在无公网 [[IPv4]] 环境下，通过 [[STUN]] 协议打洞穿透运营商 [[NAT]]，实现异地满速访问内网设备的完整教程。

## 目标受众与前置知识
- **适合谁看**：没有运营商分配的公网IP，但需要异地远程访问家里内网设备（如 NAS、软路由配置），且希望免费跑满上行带宽的用户。
- **前置知识**：[[软路由]]基础配置、[[IPv4]]与[[IPv6]]基础、[[NAT]]原理、[[防火墙]]基础概念。

## 核心观点
1. **获取 [[Fullcone]]（[[net1]]）是成功实现异地 P2P 打洞的前提**
   - 为什么重要：只有处于通道开放后允许任何外部 IP 发送数据的网络环境，才能让异地设备使用不同 IP 成功访问内网通道。
   - 依据：讲者原话，“我们必须拿到 Focal（推断指 Fullcone）让任何 IP 都能往这个通道里面发送数据，才能让我们在 Ed地（异地）使用不同 IP 访问该通道”[1]。
2. **软路由的代理插件内核会破坏 [[STUN]] 打洞与保活机制**
   - 为什么重要：数据包只要经过代理内核，即使分流规则是直连，源端口也会被重新分配，导致运营商路由器重新添加映射，使得原外部端口失效。
   - 依据：讲者通过屏幕展示指出，未排除保活 IP 时，相当于代理内核“一直在保5万端口”，而打洞出来的端口因无数据传输被回收导致访问失败[2, 3]。
3. **利用短链或 [[Cloudflare]] 重定向可解决动态外网端口/IP变更的通知问题**
   - 为什么重要：打洞产生的运营商映射外部 IP 和端口是不固定的，需要通过脚本自动化获取并更新访问入口。
   - 依据：讲者演示了通过 [[netmap]] 触发 SSH 脚本，自动修改公开短链或调用 [[Cloudflare]] API 更新重定向域名规则的操作[4-6]。

## 概念关系
- [[STUN]] → [[NAT]]打洞：[[STUN]] 服务器用于告诉客户端其经过 [[NAT]] 转换后的公网 IP 和端口，从而建立并维持数据传输通道[7, 8]。
- [[net1]] ([[Fullcone]]) ↔ [[net3]] (端口限制锥)：[[net1]] 允许任何 IP 向打洞通道发送数据，而 [[net3]] 严格限制只接受特定目标 IP 和端口的数据，无法用于通用的异地访问[1, 8]。
- 代理内核分流 → 端口重置：[[STUN]] 流量或保活流量进入 [[Home Proxy]] 等代理内核后，即使走直连也会重新开辟空闲端口，改变原始映射端口[2]。

## 详细笔记
### 第一阶段：背景介绍与需求分析
- 痛点：因入网用户增多及 [[IPv4]] 地址耗尽，运营商不再分配公网地址，多处于大内网（[[CGNAT]]）环境[9]。[[IPv6]] 配置繁琐且与 [[IPv4]] 不兼容[9]。第三方内网穿透（如 [[FRP]]、[[花生壳]]）带宽昂贵且无法跑满家庭上行[9]。
- 解决方案：不良林（讲者）提出通过 [[STUN]]（视频字幕多处识别为 stom/stone）打洞穿透 NAT 来免费实现直连[9]。

### 第二阶段：环境检测与获取 Fullcone
- 前置准备：基于 [[OpenWrt]] v23.05 软路由系统，需暂关代理插件，建议使用软路由进行 pppoe 拨号[10]。
- 验证大内网：比对光猫/软路由获取的 100.64 等私有网段 IP 与访问 IP 查询网站的公网 IP 是否一致[10]。
- 检测 NAT 类型：使用在线测试网站检测。若显示 [[net3]]（端口限制锥），需在软路由防火墙中勾选启用 [[Fullcone]]（字幕识别为 Focal/Foc），保存应用后重测以获取 [[net1]]（全锥型）[10]。若仍无法获取，可能是运营商限制或防火墙配置问题[7]。

### 第三阶段：STUN 打洞原理与 NAT 类型解析
- 通信流程：电脑发起请求询问公网 IP，经家用路由器 [[NAT]] 转换（原IP变内网IP，分配随机源端口），再经运营商路由器 [[NAT]] 转换（原IP变公网IP，再分配随机源端口），发至 [[STUN]] 服务器[7]。服务器返回最终公网 IP 和端口，此时通道（“洞”）打通[8]。
- 核心分类（参考 [[RFC5780]]）[1]：
  - [[net1]] ([[Fullcone]])：通道打开后，接受任何 IP 发送的数据（本教程必需条件）。
  - [[net2]] (IP限制锥)：只接受开启通道的对应 IP 发送的数据。
  - [[net3]] (端口限制锥)：只接受特定 IP 和特定端口发送的数据。
  - [[net4]] (对称NAT)：同端口发送到不同目标会重新添加映射。

### 第四阶段：netmap 插件配置与防火墙放行
- 工具选择：提到 [[lucky]]，但讲者选择纯粹的 [[netmap]] 插件进行演示（直接在软件包搜 `map` 安装中文化包）[1, 11]。
- 配置步骤：
  - 协议选择 [[TCP]]，寻找支持 TCP 且未被墙的 [[STUN]] 服务器[11]。
  - 配置保活（Keep-Alive）的 [[HTTP]] 服务器：如填写 [[阿里]] DNS IP `223.5.5.5` 或 `223.6.6.6`[11]。
  - 绑定空闲端口（如 12345），目标填入内网 IP 及 80 端口[11]。
- 防火墙设置：[[OpenWrt]] 默认拦截从 wan 区域主动发起的连接。需在“通信规则”中添加规则，接受从 wan 区发往本机绑定端口（12345）的 [[TCP]] 数据包[12]。完成后即可在异地访问软路由管理后台[13]。

### 第五阶段：保活机制与端口转发应用进阶
- 维持通道：运营商通常会在几分钟无数据时回收 NAT 映射表。[[netmap]] 默认间隔约 35 秒利用绑定端口向保活服务器发送 [[HTTP]] 请求，以防映射被删[13, 14]。
- 穿透内网设备（以矿机为例）：新建 [[netmap]] 规则指向矿机 IP。由于矿机系统安全策略拒绝非同网段访问，有两套方案[14, 15]：
  1. 使用 netmap 转发模式：自动重封装数据包，源 IP 变为软路由 IP[15]。
  2. 使用防火墙端口转发（[[DNAT]]）：性能更高。需额外配置源地址转换（[[SNAT]]），将发给矿机 80 端口的数据包源 IP 改为软路由接口 IP `192.168.2.1` 绕过安全限制[15, 16]。警告：暴露无鉴权服务存在被白嫖算力风险[16]。

### 第六阶段：代理插件冲突排查与流量绕过
- 冲突现象：开启代理插件（如 [[Home Proxy]]、[[Open Clash]]、[[password]]/推断指PassWall）后打洞失败，或使用 [[lucky]] 时一段时间后失联[2, 16]。
- 原因分析：[[STUN]] 探测流量进入代理内核，导致展示的是节点 IP[16]；或者保活 IP 未绕过代理内核，导致内核新开空闲端口去发数据，原通道映射因无数据传输被运营商删除[2, 3]。
- 解决操作：在代理插件的访问控制/WAN 策略中，将 [[STUN]] 服务器 IP 和保活 IP（讲者建议均用 `223.6.6.6` 防止影响正常 DNS）设置为直连/排除，使其不进入代理内核[2, 3]。

### 第七阶段：动态端口更新的自动化通知方案
- 需求背景：运营商分配的公网 IP 和外部端口会不定期变化，需利用通知脚本（提到 TJ邮箱/推断指Telegram、网页接版、短链、[[DDNS]] 等）获取最新入口[3]。
- 方案 1：利用公开短链服务（通过 [[法]]、[[中轨]] /推断指FOFA、钟馗之眼等资产搜索工具寻找未被墙服务）。在 root 下写 SSH 脚本（利用 curl 依赖），绑定至 [[netmap]] 通知路径，端口变动时自动递增短链后缀实现跳转[4]。
- 方案 2：拥有域名的用户可利用 [[Cloudflare]]。获取 API Token（屏幕提示不可泄露给他人）和 Page ID，创建状态码为 302 的页面规则重定向。写入 SSH 脚本，当 [[netmap]] 检测到 IP/端口变化时调用 API 自动修改 [[Google]] 等占位 URL 为最新入口[5, 6]。注意该方案仅限 [[HTTP]] 流量[6]。

## 金句亮点
- “我们必须拿到Focal让任何IP都能往这个通道里面发送数据，才能让我们在Ed地使用不同IP访问该通道” — 知识型 / 明确了实现异地 P2P 穿透的底层网络原理要求。
- “只要STUN的数据经过了代理内核，就无法成功穿透……因为经过代理内核后，外部端口已经发生了变化” — 知识型 / 点破了众多用户配置内网穿透时容易忽略的核心冲突点。

## 行动建议
- **做什么**：访问在线 NAT 侧漏网站，检测自身网络环境，若为 Net 3，则在软路由防火墙中勾选启用 Fullcone 并重测。
- **关联观点**：对应核心观点 1
- **验证标准**：在线检测网页最终稳定显示为 Net 1 (Fullcone 全锥型)。

- **做什么**：在软路由的代理插件（如 Home Proxy）“WAN 策略”中，将用于 STUN 探测和保活的服务器 IP（如 223.6.6.6）设置为直连排除。
- **关联观点**：对应核心观点 2
- **验证标准**：netmap 外部 IP 正确显示国内运营商 IP 而非代理节点 IP，且长时间测试通道未断连。

- **做什么**：在 Cloudflare 获取 API Token 与 Page ID，并在软路由 root 目录下创建执行脚本，填入 netmap 通知设置。
- **关联观点**：对应核心观点 3
- **验证标准**：在 netmap 手动修改绑定端口触发强制更新后，使用设定的 Cloudflare 域名能在浏览器正确跳转访问到内网测试设备。

## 步骤拆解
1. **测试与开启 Full Cone NAT**
   - 具体操作描述：关闭软路由上运行的代理插件。使用在线 NAT 测试工具检测当前 NAT 类型。如果显示 NAT 3（端口限制型），则进入软路由防火墙设置页面，勾选“启用 FullCone NAT”，保存并应用后再重新检测，确保拿到 NAT 1（Full Cone 全锥型） [2]。
   - 关键代码/命令：未明确提及
   - 注意事项：测试期间建议关闭代理插件避免干扰。若使用光猫拨号，需登录光猫在高级 NAT 下的 DMZ 配置中填入软路由获取的 IP [2]。

2. **安装并配置 netmap 插件**
   - 具体操作描述：在软路由软件包列表中更新列表，搜索并安装 [[netmap]] 中文包及其依赖。进入服务找到插件，修改配置：协议选择 [[TCP]]，地址选 [[IPv4]]，填写支持 [[TCP]] 的 [[STUN]] 服务器 IP（需 ping 测试确保无延迟/未被墙），填写 HTTP 保活服务器 IP（如 223.5.5.5），绑定一个空闲端口（如 12345），选择“转发模式”，目标填入路由器内网 IP 和 80 端口，启用并应用 [5]。
   - 关键代码/命令：未明确提及
   - 注意事项：若无法获取外部公网 IP 和端口，尝试更换 [[STUN]] 服务器 [5]。

3. **防火墙通信规则放行**
   - 具体操作描述：进入网络-防火墙-通信规则，添加一条规则。协议选择 [[TCP]]，源区域设为 WAN，目标区域设为设备（本机），目标端口填入 [[netmap]] 绑定的端口（如 12345），动作设为接受，保存并应用 [4]。
   - 关键代码/命令：未明确提及
   - 注意事项：OpenWrt 防火墙默认拦截从 WAN 区主动发起的连接，必须放行该端口才能实现外部访问 [4]。

4. **配置保活（Keep-alive）机制**
   - 具体操作描述：在 [[netmap]] 插件中设置保活间隔数（默认约 35 秒）。使其定时通过绑定端口向设定的 HTTP 服务器发送请求，以保持运营商路由器的 NAT 映射表不被回收 [6, 7]。
   - 关键代码/命令：未明确提及
   - 注意事项：未明确提及

5. **配置防火墙直接端口转发与 SNAT 绕过限制（可选的高阶转发模式）**
   - 具体操作描述：在 [[netmap]] 中取消“转发模式”。前往防火墙的“端口转发”添加规则，将外部端口（[[netmap]] 绑定的端口）转发至内网目标设备（如矿机）的指定端口。针对限制非同网段访问的设备，需额外在防火墙自带的 NAT 规则中做 SNAT，将所有发往目标设备 80 端口的数据包的源 IP 修改为软路由的 LAN 口 IP（如 192.168.2.1） [7-9]。
   - 关键代码/命令：未明确提及
   - 注意事项：使用系统防火墙直接转发性能更高，但会暴漏公网访问者的真实 IP 导致部分内网设备拒绝访问，必须搭配 SNAT 使用 [8]。

6. **排除代理工具内部分流干扰**
   - 具体操作描述：启动代理工具（如 [[Home Proxy]]）后，将 [[STUN]] 服务器的 IP 以及保活用的 HTTP 服务器 IP（如 223.5.5.5 或 223.6.6.6）设置为直连（绕过代理内核处理） [9-11]。
   - 关键代码/命令：未明确提及
   - 注意事项：只要打洞或保活的数据流量进入了代理内核，哪怕走的是直连分流，内核也会重新开一个空闲端口转发出去，导致运营商映射的外部端口发生变化而断连 [10, 11]。

7. **配置动态入口通知脚本**
   - 具体操作描述：通过 [[SSH]] 登录进入软路由 `root` 目录，创建 `.sh` 脚本文件，赋予可执行权限。在脚本中填入通过资产搜索工具找的未被墙的短链服务代码，或利用 [[Cloudflare]] API 结合页面规则（Page Rule）做 302 重定向的代码。最后将该脚本路径填入 [[netmap]] 的通知脚本处 [3, 12, 13]。
   - 关键代码/命令：未明确提及（仅提及 `chmod +x` 的文字描述“赋予可执行权限”）
   - 注意事项：如果使用 [[Cloudflare]]，页面规则的转发状态码需使用 302（临时重定向），不要用 301 以免导致浏览器缓存 [12]。

## 常见坑与解决方案
- **问题描述**：检测 NAT 类型时显示 NAT 3（端口限制型）。
  - 解决方案：在软路由防火墙中勾选启用 FullCone NAT 并保存应用 [2]。
- **问题描述**：开启 FullCone 后依然拿不到 NAT 1。
  - 解决方案：大概率被运营商限制，可尝试联系运营商要求开放 NAT 1；或自行排查是否改动过防火墙相关配置 [14]。
- **问题描述**：通过公网 IP 和端口访问路由器后台时提示访问被拒绝。
  - 解决方案：在防火墙的通信规则中，添加一条接受从 WAN 区发往本机对应绑定端口的规则 [4]。
- **问题描述**：使用系统防火墙端口转发访问内网设备时，网页显示 Invalid Host 或无法完全加载。
  - 解决方案：因为目标设备系统内置安全策略只允许私有同网段访问。需在防火墙 NAT 规则中做 SNAT，将源 IP 改为路由器接口 IP [8]。
- **问题描述**：使用代理插件后，一开始可以穿透访问，但过一段时间后失效无法访问。
  - 解决方案：保活 IP 的流量进入了代理内核导致外部端口发生改变。必须将保活流量 IP 和 [[STUN]] 服务器 IP 排出在代理内核之外（在插件中设为直连绕过） [10, 11]。
- **问题描述**：排除 223.5.5.5 这个保活 IP 后，导致本地 DNS 行为异常。
  - 解决方案：将保活 IP 和排除 IP 都改为 223.6.6.6 [11]。

## 最佳实践
- 进行 NAT 测试时，建议暂时关闭软路由上的代理插件，以避免其他因素干扰 [2]。
- 建议使用软路由直接进行 PPPoE 拨号，而非通过光猫拨号 [2]。
- 使用 netmap 穿透时，选择 STUN 服务器前建议先在本地 ping 一下，确保有延迟说明没有被墙 [5]。

## 关键概念

### [[NAT打洞]]
- **定义**：在没有公网IP的情况下，利用STUN协议在运营商的大内网NAT映射表中打通数据通道，实现异地直连内网设备的技术 [1, 2]。
- **视频上下文**：作者将其作为解决无公网IP家庭网络远程访问需求的核心方案，用于替代昂贵且限速的第三方内网穿透工具 [1]。
- **为什么重要**：它能实现无需花钱中转且跑满家庭上行带宽的远程访问体验 [1]。
- **关联概念**：依赖 → [[STUN]]

### [[STUN]]
- **定义**：一种用于探测网络类型并向设备返回其在运营商路由器上映射的公网IP和端口的服务或协议 [3-5]。
- **视频上下文**：视频演示了软路由通过发送STUN请求（UDP或TCP格式），从STUN服务器获取转换后的公网入口信息，从而打通数据通道 [3, 6]。
- **为什么重要**：没有它就无法知道打通的“洞”在公网的具体位置，是实现NAT穿透的第一步 [4, 5]。
- **关联概念**：属于 → [[NAT打洞]]

### [[Full Cone NAT]]
- **定义**：也称作NAT1（全锥型NAT），指只要数据通道打开之后，任何外部IP都能往这条通道里发送数据并被接收的NAT网络类型 [4, 7]。
- **视频上下文**：作者在操作前测试了网络环境，强调必须在软路由防火墙中开启此功能并从运营商处拿到NAT1，才能允许异地不同IP发起的直连访问 [7, 8]。
- **为什么重要**：它是当前NAT穿透方案能成功让异地任意网络设备连入通道的严格先决条件 [7]。
- **关联概念**：对比 → [[对称NAT]]

### [[对称NAT]]
- **定义**：也称作NAT4，指使用相同源端口向不同目标发送数据时，会重新添加映射规则的NAT网络类型 [7]。
- **视频上下文**：在解释几种NAT工作原理差异时简要提及，以此作为Full Cone NAT的对照参照物 [7]。
- **为什么重要**：它代表了一种对映射规则限制极严的网络环境，在此环境下常规的打洞穿透会失败 [7]。
- **关联概念**：对比 → [[Full Cone NAT]]

### [[保活机制]]

- **为什么重要**：在进行异地访问时，它是用来绕过部分内网设备“仅允许局域网私有网段访问”限制的关键技巧 [12]。
