## 一句话总结
不良林详解在OpenWrt软路由上配置IPv6及代理插件（HomeProxy、OpenClash等）的完整流程、原理与分流避坑指南。

## 目标受众与前置知识
- **适合谁看**：需要通过公网访问家中的设备、要求观看IPTV，或解决在IPv6网络环境下遇到科学上网代理冲突、DNS污染等问题的用户。
- **前置知识**：[[OpenWrt]]、[[IPv6]]、[[软路由]]、[[OpenClash]]、[[HomeProxy]]、[[sing-box]]。

## 核心观点
1. **代理环境中需彻底禁用IPv6 DNS以防污染**
   - 为什么重要：代理环境下同时开启公网IPv6 DNS处理不当，会直接破坏分流规则，导致网站访问慢或无法访问。
   - 依据：讲者原话“如果你配置了公网的ipv6dns服务器那么查询DNS请求的流量将会直接发送到互联网于是DNS污染”。
2. **获取域名的IPv6地址无需依赖IPv6 DNS**
   - 为什么重要：打破了“IPv6和IPv4不兼容，解析IPv6必须用IPv6 DNS”的常见认知误区，为在代理环境中仅保留IPv4 DNS提供了理论支撑。
   - 依据：讲者原话“用ipv4的DNS服查询a记录就是返回ipv4的地址，查询4a记录就是返回ipv6地址”。
3. **网络拓扑与光猫模式决定了下级路由的IPv6获取策略**
   - 为什么重要：不同的家庭网络架构需要不同的接口设置方法，生搬硬套会导致局域网设备拿不到IPv6地址。
   - 依据：桥接模式下软路由可获取长度为64（讲者原话）的[[PD]]前缀；路由模式若无PD可尝试中继（Relay）模式；而校园网等获取-128（讲者原话）地址的极端情况必须使用[[NAT66]]。

## 概念关系
- [[RA]] → [[SLAAC]]：路由器向局域网广播包含前缀的RA（路由器通告）包，设备接收后通过SLAAC（无状态自动配置）自动拼合生成自己的IPv6地址。
- [[EUI-64]] ↔ 随机后缀：EUI-64依赖网卡固定的[[Mac]]地址拼接生成IPv6后缀，存在被全网追踪的隐私风险；现代系统（如[[Windows]]、[[Mac]]）默认使用随机且定期更换的后缀生成临时地址。
- [[HomeProxy]] ↔ [[OpenClash]]：在处理IPv6代理时，基于[[sing-box]]的HomeProxy支持real-ip端到端访问控制；而[[OpenClash]]的DNS模块默认基于fake-ip模式，终端以IPv4虚假IP访问，由路由器代为发起真实的IPv6访问。

## 详细笔记

### 准备工作与清理基础环境
讲者[[不良林]]指出，为满足公网访问家中设备及看[[iptv]]的极速需求，可启用IPv6。但在开始配置前，必须关闭正在运行的所有代理插件和第三方DNS插件以防干扰。基于[[emoto wrt23.05]]固件，在“全局网络选项”中删除Ula前缀，并在DNS高级设置中取消勾选“过滤ipv6解析”。

### 多种光猫环境的IPv6接口配置实操
- **情景一：光猫桥接，软路由拨号**：会生成wan6虚拟动态接口并获取IPv6地址及[[PD]]前缀。在LAN口设置中分配长度选择64（屏幕展示/讲者原话），开启DHCPv6的RA服务为服务器模式。
- **情景二：光猫路由，软路由作二级**：需在[[OpenWrt]]中手动创建名为wan6的dhcpv6客户端协议接口。若能拿到PD前缀则操作同情景一；若光猫不发送PD，讲者建议改桥接、换光猫，或将LAN与WAN6口的RA、DHCPv6、NDP均设置为中继（Relay）模式。
- **情景三：仅获取到单个地址（如校园网）**：仅获得一个-128（讲者原话）前缀长度的地址，必须配置[[NAT66]]，为LAN口生成一个Ula唯一本地地址前缀，并在防火墙开启IP动态伪装。

### 地址生成机制与生命周期（Lifecycle）
- **后缀生成原理**：IPv6总长128比特，前缀64比特由运营商提供。设备后缀可通过[[EUI-64]]生成（由于网卡[[Mac]]地址仅48个比特，中间会插入ff1填充到64个比特，讲者原话如此描述，推断实为fffe），或使用随机值以防泄露隐私。
- **地址状态与生命周期**：生成的IP先发送组播包进行DAD（重复地址检测），无人占用即变为首选（Preferred）。寿命由plt（首选生命周期）控制，讲者演示环境中为2700秒也就是45分钟（讲者原话）。倒计时完毕后IP进入“受到抨击”（Deprecated，即弃用状态），vlt（有效生命周期）倒计时结束后IP将从网卡被彻底删除。前缀变化时，路由器会通过发送新旧两个前缀的[[RA]]包实现平滑切换。

### 代理避坑：DNS与DHCPv6的取舍
因[[Android]]设备不支持，且前缀更新机制不佳，讲者建议直接禁用[[DHCPv6]]，全靠无状态自动配置。
在防代理泄露方面，取消将路由器当作IPv6 DNS服务器，不让设备获取IPv6 DNS。老系统如[[Windows 7]]不支持从RA获取DNS，[[Windows 11]]在双栈环境下存在不分配IPv6 DNS的Bug（讲者原话推断），但利用IPv4 DNS即可顺利解析出[[百度]]（Baidu）等网站的IPv6地址。

### 代理插件对IPv6流的劫持与处理逻辑
- **使用 [[HomeProxy]]（基于[[sing-box]] / sbox内核心）**：
  - 如果DNS策略为“仅IPv4”，设备仅能通过IPv4访问。改为“默认”后即可解析IPv6。
  - 若不开启IPv6支持，流量不进sbox直接直连；开启支持后流量被劫持，如果启用“绕过中国大陆IP”，国内请求如百度会由路由器直接转发；若节点不支持IPv6（如尝试用不支持的节点访问[[谷歌]]/Google），将导致科学上网失败。
- **使用 [[OpenClash]]**：
  - 基于fake-ip模式，需在IPv6设置中勾选“允许IPv6解析”。本机以获取的IPv4 fake-ip发起访问被Clash劫持，随后Clash通过上游DNS获取目标IPv6真实地址，并用路由器自身的IPv6向目标发起请求（呈现一层NAT效果），无法实现端到端真实IPv6访问。
- **关于 [[mihomo]]**：视频标题提及了mihomo，但在讲者口播与演示的上下文内容中并未明确提及（未明确提及）。

## 金句亮点
- “ipv6加代理环境会产生奇妙的化学反应，配置不当的话会出现分流异常、DNS被污染、网站访问慢或者无法访问等症状” — 内容指出了科学上网与IPv6并存的核心痛点 / 业务价值。
- “ipv4的DNS服务器可以获取域名的ipv6地址，ipv6的DNS服务器也同样可以获取域名的ipv4地址” — 破除了DNS协议必须与IP版本对应的常识误区 / 知识普及价值。

## 行动建议
- **做什么**：在软路由LAN口的DHCPv6设置中，取消勾选“本地DNS服务器”（即不下发路由器的IPv6地址作为DNS），且留空RDNSS，强制客户端仅使用IPv4 DNS解析。
- **关联观点**：对应核心观点 1 与 2。
- **验证标准**：保存并重启网络后，在电脑端输入 `ipconfig /all`（或对应系统命令），确认网卡配置中仅存在IPv4的DNS服务器，且能在浏览器中正常打开开启了IPv6支持的网站，且科学上网分流不受影响。

## 步骤拆解

1. **前期准备与清理**
   - 具体操作描述：关闭所有正在运行的代理插件和第三方 [[DNS]] 插件；在全局网络选项中，删除 ULA 前缀并保存；在 [[DNS]] 高级设置中，取消勾选“过滤 ipv6 解析”，点击保存并应用 [1][4]。
   - 关键代码/命令：未明确提及
   - 注意事项：操作前必须关闭代理，防止出现预期之外的化学反应和分流异常 [1]。

2. **场景一配置：光猫桥接（路由于拨号，获取 PD）**
   - 具体操作描述：编辑 lan 口，进入高级设置，分配长度选择 64，输入长后缀或填入 eui64；进入 [[DHCPv6]] 设置页面，将 RA 设置为“服务器模式”，取消勾选“本地 [[DNS]] 服务器”；进入 IPv6 RA 设置，取消勾选“o 标记”；最后保存并应用 [4]。
   - 关键代码/命令：未明确提及
   - 注意事项：理想情况下 wan6 的虚拟动态接口会自动拿到一个 [[IPv6]] 地址以及一个 PD（前缀委托） [4]。

3. **客户端状态验证**
   - 具体操作描述：在电脑上启用 [[IPv6]] 协议栈，等待片刻后通过命令行查看物理网卡接口的 IP 地址分配情况 [4][5]。
   - 关键代码/命令：`ipconfig /all` [5]
   - 注意事项：未明确提及

4. **场景二/三配置：光猫路由模式（二级路由环境）**
   - 具体操作描述：手动添加新接口 `wan6`，协议选择 [[DHCPv6]] 客户端，设备选择物理 wan 口，防火墙设为 wan 区域。若成功获取到 PD，lan 口配置同步骤2；若无法获取 PD，则使用中继模式：编辑 `wan6` 口的 [[DHCPv6]] 服务器勾选为主接口，RA 选择中继，禁用 [[DHCPv6]]，NDP 代理选择中继；lan 口的 RA 和 NDP 同步设置为中继并保存 [5][6]。
   - 关键代码/命令：未明确提及
   - 注意事项：拿不到 PD 时建议优先尝试将光猫改桥接或更换新光猫，中继模式是不太推荐的备用方案 [6]。

5. **场景四配置：极端情况（仅获取到 /128 地址）**
   - 具体操作描述：配置 [[NAT66]]；在全局网络选项中随机设置一个 ULA 唯一本地地址前缀并保存；进入防火墙编辑 wan 区域，高级设置开启“[[IPv6]] 伪装”（即 NAT）；lan 口的配置步骤同步骤2 [6][7]。
   - 关键代码/命令：未明确提及
   - 注意事项：常出现于校园网中，此方案相当于回退到了 IPv4 的 NAT 时代 [6]。

6. **代理插件 [[homeproxy]] 配置**
   - 具体操作描述：进入 [[homeproxy]] 的 [[DNS]] 设置页面，将 DNS 策略由“仅 IPv4”改为“默认”并应用；如需代理 [[IPv6]] 流量，需开启 [[homeproxy]] 的 [[IPv6]] 支持，并在代理网站使用的解析策略中将其修改为“默认” [8][9][10]。
   - 关键代码/命令：未明确提及
   - 注意事项：开启截持后，检测到的公网 IP 会变成软路由设备的 IP（类似做了一层 NAT）；若仅需直连不建议开启截持支持 [9][10]。

7. **代理插件 [[OpenClash]] 配置**
   - 具体操作描述：进入插件设置下的 [[IPv6]] 设置，勾选“允许 ipv6 解析”，点击应用配置 [11]。
   - 关键代码/命令：未明确提及
   - 注意事项：[[OpenClash]] 默认的 [[DNS]] 模块是 fake-ip 模式，电脑只能拿到 fake-ip，无法拿到域名的真实 [[IPv6]] 地址，因此截持场景非常有限，无特殊需求不建议开启 [[IPv6]] 流量代理 [11][12]。

## 常见坑与解决方案

- **问题描述**：按视频配置后 lan 口没有出现 [[IPv6]] 地址 [5]。
  - 解决方案：尝试重启软路由 [5]。
- **问题描述**：[[IPv6]] 加代理环境产生异常，如分流异常、[[DNS]] 被污染或网站无法访问 [1][13]。
  - 解决方案：不要配置 [[IPv6]] 的 [[DNS]] 服务器（在 RA 设置中取消本地 DNS 服务器和 o 标记），直接禁用 [[IPv6]] 的 [[DNS]] 解析，依靠 IPv4 的 [[DNS]] 服务器返回 AAAA 记录 [14][13]。
- **问题描述**：使用代理访问 [[IPv6]] 节点时显示无法访问 [11]。
  - 解决方案：确保使用的机场节点本身支持 [[IPv6]] 访问；节点不支持会导致访问失败 [11]。
- **问题描述**：开启 [[homeproxy]] 后，检测网站提示未接入 [[IPv6]] 且仅返回 IPv4 地址 [13][8]。
  - 解决方案：将 [[homeproxy]] 的 [[DNS]] 策略由“仅 IPv4”改为“默认” [8]。

## 最佳实践

- 最好的网络拓扑配置是想办法让路由器获取到 PD（前缀委托），实在没办法再使用中继和 [[NAT66]] [7]。
- 建议手动为 lan 口输入一个较长的后缀，不要留空（默认是 `::1`），以防遭到网络端口扫描 [4]。
- 建议直接禁用 [[DHCPv6]] 服务以减少出问题的概率，因为前缀更换后它不能主动推送更新到局域网设备，且 Android 系统不支持 [[DHCPv6]] [3]。
- 不配置 [[IPv6]] 的 [[DNS]] 服务器，直接使用 IPv4 的 [[DNS]] 服务器获取域名的 IPv6 地址，以规避早期的 RDNSS 兼容性问题与 Windows 11 的双栈 Bug [3][14]。
- 若访问 [[IPv6]] 的请求都是直连的，建议直接不启用代理插件的 [[IPv6]] 支持功能，让流量直接由路由器转发而不进代理内核 [10]。
- 除非有非常明确的业务需求，否则不建议使用代理访问 [[IPv6]]，因为目前代理访问 [[IPv6]] 并没有明显优势且机场支持度不一 [11]。

## 关键概念

### [[前缀委托 (Prefix Delegation / PD)]]
- **定义**：运营商下发给路由器的IPv6地址前缀，由路由器负责将其委托分配给局域网内的其他网络设备 [1, 2]。
- **视频上下文**：作者指出最理想的IPv6网络环境是光猫桥接、路由器拨号并获取到一个公网IPv6地址以及一个长度通常为64或60的PD [1, 2]。如果光猫路由模式且不下发PD，下级路由器的LAN口下的电脑就无法拿到前缀并生成IPv6地址 [3, 4]。
- **为什么重要**：它是大部分家庭局域网设备能否正常获取公网IPv6地址的核心前提和最佳实践 [2, 4]。
- **关联概念**：依赖 → [[RA (Router Advertisement)]]

### [[RA (Router Advertisement)]]
- **定义**：路由器向局域网内推送的通告数据包，包含运营商分配的IPv6前缀、前缀的生存周期以及一些标志位（flag）信息 [5]。
- **视频上下文**：电脑开启IPv6后会发送RS（路由器请求）寻找路由器，路由器接收后会回复RA数据包，此外路由器也会定期自动往局域网里推送RA通告用于更新IP生存周期或更换前缀 [2, 5, 6]。
- **为什么重要**：它是局域网设备获取网络前缀及自动更新、更换IPv6地址机制的关键传递载体 [5, 6]。
- **关联概念**：属于 → [[无状态自动配置 (SLAAC)]]

### [[无状态自动配置 (SLAAC)]]
- **定义**：设备根据路由器发送的RA数据包中的前缀信息，自动为自己生成完整IPv6地址的配置方式 [5, 7]。
- **视频上下文**：视频提到在OpenWrt中勾选“启用SLAAC”后，RA数据包里的A标志位等于1，电脑就会利用拿到的64位前缀，自行生成后64位从而拼凑成完整的IPv6地址 [5]。
- **为什么重要**：它是IPv6网络中最常用的自动配置地址模式，替代了依赖DHCPv6的分配方式，且作者建议直接使用该模式以减少网络环境出问题的概率 [5, 7]。
- **关联概念**：依赖 → [[RA (Router Advertisement)]]

### [[eui64]]
- **定义**：一种利用当前网卡MAC地址并在中间插入fffe填充至64位，从而自动生成IPv6地址后缀的规则 [5]。
- **视频上下文**：作者在配置LAN口IPv6地址后缀时提到了eui64规则，但也指出由于MAC地址固定且全球唯一，使用该规则生成的后缀容易被追踪从而产生隐私问题 [1, 5, 8]。
- **为什么重要**：解释了IPv6地址后缀的一种生成逻辑，以及现代操作系统（如Windows、macOS）为何默认采用随机后缀而非eui64的重要背景 [5, 8]。
- **关联概念**：未明确提及

### [[链路本地地址 (Link-local address)]]
- **定义**：一般以fe80开头的IPv6地址，仅能在同一链路（即同一局域网）中用于通信 [2, 8]。
- **视频上下文**：作者提到当在电脑开启IPv6协议栈后，网卡会自动生成该地址，且负责发送RA数据包的OpenWrt路由器的IPv6默认网关也是一个链路本地地址 [2, 8]。
- **为什么重要**：它是局域网内部设备进行底层通告和寻找网关（如发送RS/RA数据包）的基础通信地址 [2, 5, 8]。
- **关联概念**：未明确提及
