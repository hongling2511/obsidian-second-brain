## 一句话总结
本视频讲解了全场景通用代理传输协议 [[xhttp]] 的原理，并演示了基于 [[3x-ui]] 的多种节点搭建及上下行分离高阶配置。

## 目标受众与前置知识
- **适合谁看**：需要绕过 [[GFW]] 审查、追求科学上网高网速与抗封锁能力的进阶用户（推断）。
- **前置知识**：[[CDN]]、[[VPS]]、[[VLESS]]、[[REALITY]]、[[WS]] 协议、[[DNS]] 解析、[[Cloudflare]] 基本操作。

## 核心观点
1. **[[xhttp]] 协议大幅提升了代理传输效率**
   - 为什么重要：解决了传统将代理数据封装在正常 [[http]] 流量中过 [[CDN]] 时，因传输效率过低导致网速极慢、无法当主力使用的问题。
   - 依据：讲者指出 [[xhttp]]（原 [[splithttp]]）采用了“分包上行加流式下行”的方式，相比 [[mik]] 的全分包模式，下行传输效率有“质的提升”（讲者原话）[1, 2]。
2. **[[xhttp]] 使得高阶“上下行分离”成为可能**
   - 为什么重要：能进一步给 [[GFW]] 上强度，并优化特定网络环境（如下行直连速度快的 [[VPS]]）的资源利用率。
   - 依据：讲者演示了将上行配置为通过 [[CDN]]（如 `a.121343.xyz`），下行配置为直连或另一个域名（如 `b.888005.xyz`），从而实现流量路径分离 [3-5]。

## 概念关系
- [[mik]] ↔ [[xhttp]]：对比传输效率，前者采用“分包上行+分包下行”（速度极慢），后者采用“分包上行+流式下行”（效率高得多）[1, 2]。
- 普通 [[http]] → [[CDN]]：因为代理数据被封装为正常的 HTTP 流量，所以能够自然地通过 CDN 进行传输防封锁 [1]。
- [[VLESS]] + [[REALITY]] ↔ [[VLESS]] + [[xhttp]] + [[REALITY]]：对比常规节点，引入 [[xhttp]] 后增加了字节填充、多路复用、上下行分离等新传输特性 [6]。

## 详细笔记
### 1. 代理传输协议的演进背景
- **2023年5月**（讲者原话）：[[v2ray]] 添加了 [[mik]] 协议，可通过 [[http]] 伪装过 [[CDN]]，但由于效率太低仅能作为备用方案 [1]。
- [[Xray]] 作者 [[rbrx]] 提出“分包上行加流式下行”的优化构想 [1]。
- **2024年6月**（讲者原话）：开发者基于此构想推出了 [[splithttp]] 协议 [1]。
- **2024下半年**（讲者原话）：协议经过不断迭代，更名为 [[xhttp]]，实现了多路复用、字节填充、上下行分离等特性 [7]。

### 2. 传输原理对比分析
- **普通 [[tcp]] 传输**：代理工具与节点服务器直接通过 TCP 连接，实现流式上行与流式下行 [1, 2]。
- **[[mik]] 传输**：浏览器访问 [[Google]] 时，请求被分为小数据包发送（分包上行），下行时代理工具需要不断找节点“要数据”（分包下行），重组后交给 [[Google Chrome]] [2]。
- **[[xhttp]] 传输**：上行同样是分包（Pocket APP），但下行时节点会将数据封装成支持流式传输的不定长 chunk 响应，直接返回给代理工具，无需不断请求 [2]。

### 3. xhttp + REALITY 基础直连搭建
- **工具准备**：使用 [[3x-ui]] 面板进行一键安装配置（官方更推荐 [[x-ui]] 手写配置，讲者为方便新手选用面板）（推断）[7]。
- **面板设置**：添加入站，端口建议设为 **443**（讲者原话），安全设置选 [[REALITY]]，传输协议改为 [[xhttp]]，并设置 path 路径 [6, 7]。
- **客户端适配**：
  - [[Windows]] 端：使用 [[v2rayN]] [6]。
  - [[Mac]] 端：使用支持该内核的 [[v2rayN]]（需在线或手动下载更新 [[Xray]] 内核及 [[geo数据库]]）[6]。
  - [[Android]] 端：使用 [[v2rayNG]] [6]。

### 4. xhttp 套用 CDN 搭建步骤
- **面板设置**：搭建裸 [[VLESS]] + [[xhttp]]（不加 REALITY），端口设为 **80** 或 **6666**（讲者原话）[8]。
- **[[Cloudflare]] 设置**：添加 [[DNS]] 解析并开启小黄云（代理状态），TLS 加密模式改为“灵活”（浏览器到 CF 加密，CF 到 VPS 不加密），在网络页面启用 [[grpc]] 以支持流式上传 [8]。
- **规则配置**：若使用非 80/443 端口，需在 Origin Rules 中设置端口重写，将目标端口指向 **6666**（讲者原话）[8]。

### 5. 高阶：上下行分离配置与 fallback 机制
- **双域名分离**：上行配置域名 A（如 `a.121343.xyz`），下行配置域名 B（如 `b.888005.xyz`），两者指向同一 [[VPS]] 的 [[IP]]。通过在客户端修改下行配置代码实现分离，可用 [[vsock]] 工具抓包验证 [3, 4]。
- **上行 CDN + 下行 REALITY 直连**：
  - 适用场景：下行直连速度快（如 GIA 线路）但怕 IP 被墙的 VPS [5]。
  - 搭建步骤：面板创建一个普通 [[VLESS]] + [[Vision]] + [[REALITY]] 节点，临时关闭安全设置以调出 [[fallback]]（回落）选项，设置目标端口为裸 [[xhttp]] 节点的 **6666** 端口，再将安全设置改回 REALITY [5]。
  - 运行逻辑：客户端看似与 REALITY 建立下行连接，[[GFW]] 看到的是访问伪装的 SNI（如 [[Yahoo]]）；但 REALITY 节点收到自己无法处理的 [[xhttp]] 数据时，会通过 fallback 机制转交给内部 **6666** 端口的裸 [[xhttp]] 节点，最终成功打通上行 CDN、下行直连的数据通路 [9]。

   - 注意事项：如果裸节点直接使用了 80 端口，可省略 Origin Rules 的端口重写操作；若需优选 IP，将域名填入 sni 并在地址栏填优选后的 IP 即可 [4, 6]。

3. **配置上下行分离（CDN 双域名）**
   - 具体操作描述：准备两个域名（或同域名的不同优选 IP），例如 A 域名作上行，B 域名作下行 [5, 6]。为 B 域名添加 DNS 解析并开启代理状态 [5]。复制上行节点，在其基础上“添加额外参数”，指定下行使用的 server name（B 域名）和与节点一致的 path 路径 [5]。
   - 关键代码/命令：下行配置代码（未明确提及，视频指引在说明栏） [5]。
   - 注意事项：节点修改配置后，在客户端的列表里会跑到最下面；可以通过 [[vsock]] 抓包来查看上下行分离的实际效果 [5]。

4. **配置上行 CDN + 下行 REALITY 直连**
   - 具体操作描述：新建一个普通的 [[vless]] 加 tcp 加 [[REALITY]] 节点，并在面板中为其添加回落（fallback），目标端口指向前文搭建的裸 [[xhttp]] 端口（6666） [7]。利用 [[v2rayN]] 导出该节点的客户端 JSON 配置。将其配置内容替换到 [[CDN]] 节点的下行额外配置中，从而实现通过 [[CDN]] 上传、通过 [[REALITY]] 直连下载 [7]。
   - 关键代码/命令：REALITY 下行配置代码（未明确提及，视频指引在说明栏） [7]。
   - 注意事项：[[3x-ui]] 面板存在隐藏回落选项的 UI 问题，需通过特定方式绕过（详见常见坑） [7]。

## 常见坑与解决方案
- **问题描述**：Mac 版 [[v2rayN]] 第一次打开可能是英文，且不包含内核导致无法直接使用 [3]。
  - 解决方案：在界面中手动切换为中文；点击“更新”，勾选 [[Xray]] 和 geo 数据库进行更新。如果无法在线更新，可以在设置中找到存储位置，手动下载内核放入对应目录 [3]。
- **问题描述**：在 [[3x-ui]] 面板中编辑节点时，开启了 [[REALITY]] 就看不到 fallback（回落）的设置项 [7]。
  - 解决方案：先将安全设置为“无”，此时会出现回落项。添加回落目标（如 6666）后，再将安全改回 [[REALITY]]。此时回落配置项虽然在界面上不见了，但实际设置已经生效并保存 [7]。

## 最佳实践
- 官方并不推荐使用 [[3x-ui]] 面板来配置 [[xhttp]]，但为了方便普通用户快速上手，仍作为折中方案演示 [2]。
- 搭建 [[xhttp]] 时尽量使用默认参数，如需深度定制请务必自行查阅作者发布的官方文档 [2]。
- 在使用 [[CF]] 配置 [[CDN]] 时，推荐将 TLS 加密模式设置为“灵活”（Flexible），这样能让 [[CF]] 到 VPS 之间不加密，免去用户自行配置证书的麻烦 [4]。
- 如果 VPS 的下行直连速度比较快（比如 GIA 线路），全套 [[CDN]] 太浪费速度，又怕直连被墙，推荐采用“上行 [[CDN]] + 下行 [[REALITY]] 直连”的配置架构 [7]。
- 目前普通的 [[REALITY]] 节点已经非常稳定，如果没有使用 [[xhttp]] 特定特性（如过 [[CDN]]、字节填充、多路复用等）的刚需，直接使用普通 [[REALITY]] 即可，不需要盲目追新 [3, 4]。

## 关键概念

### [[xhttp]]
- **定义**：在2024年下半年由 splithttp 迭代更名而来的新型传输协议，支持流式上行、多路复用和上下行分离等特性 [1, 2]。
- **视频上下文**：作者作为核心主题进行演示，展示了如何使用该协议配合 CDN 或 reality 搭建节点，并强调其大幅提升了传输效率 [1, 2]。
- **为什么重要**：它解决了以往协议通过 CDN 传输时网速慢的痛点，开启了代理的崭新时代，可作为主力使用 [1, 2]。
- **关联概念**：对比 → [[mik]]

### [[mik]]
- **定义**：2023年5月添加到 v2ray 中的一种代理传输协议，能将代理数据封装在正常 HTTP 流量里通过 CDN 传输 [1]。
- **视频上下文**：作者将其作为背景知识提及，指出其采用“分包上行加分包下行”的方式，导致网速极慢，只能做备用方案 [1, 3]。
- **为什么重要**：它验证了将代理数据伪装成普通网站 HTTP 流量过 CDN 的可行性，但暴露了早期协议在传输效率上的致命缺陷 [1, 3]。
- **关联概念**：对比 → [[xhttp]]

### [[分包传输]]
- **定义**：将代理请求或响应数据拆分成一个个小的 HTTP 数据包进行发送和接收的低效传输机制 [3]。
- **视频上下文**：作者在解释 mik 协议原理时指出，它需要代理工具不停找节点要数据包以重新拼装，造成了极大的开销 [3]。
- **为什么重要**：解释了导致传统 HTTP 代理方式网速缓慢的根本技术原因 [1, 3]。
- **关联概念**：对比 → [[流式传输]]

### [[流式传输]]
- **定义**：直接通过 TCP 连接将连续的字节流发送到节点或返回给代理工具的高效传输机制 [1, 3]。
- **视频上下文**：作者说明 xhttp 协议利用此机制，将数据封装成支持流式传输的不定长 chunk 响应，免去了频繁请求数据包的过程 [3]。
- **为什么重要**：它是 xhttp 能够实现质的提升、超越分包传输效率的核心底层逻辑 [1, 3]。
- **关联概念**：对比 → [[分包传输]]

### [[上下行分离]]
- **定义**：代理工具在向服务端发送请求（上行）和接收服务端返回数据（下行）时，分别建立并使用不同的网络连接通道 [4]。
- **视频上下文**：作者重点演示了通过配置不同的域名或 IP，甚至上行使用 CDN、下行使用直连 reality 的方式来实现这一功能 [4-6]。
- **为什么重要**：这是一种高级抗封锁策略，能给 GFW（防火墙）“上强度”增加隐蔽性，同时还能利用特定线路（如 gia 线路）提升下行速度 [4, 6]。
- **关联概念**：依赖 → [[xhttp]]

### [[CDN]]
- **定义**：内容分发网络，在视频语境中被用来作为传输中转，掩盖真实的代理服务器 IP [1]。
- **视频上下文**：作者演示了将 xhttp 的传输建立在 CDN 之上，并配合域名解析将请求伪装成对正常网站的访问 [1, 7]。
- **为什么重要**：配合支持特定传输协议的节点，可以实现几乎无法被墙的隐蔽代理 [1]。
- **关联概念**：未明确提及

### [[reality]]
- **定义**：一种代理安全设置，具有极高稳定性，常用于直接连接或与特定传输协议组合搭配 [2, 6, 8]。
- **视频上下文**：作者不仅演示了基础的 xhttp+reality 搭建，还利用它作为上下行分离架构中的直连下行通道 [2, 6]。
- **为什么重要**：本身作为防封锁手段十分成熟，在高级架构中能兼顾伪装安全性与直连的高速度 [6, 8]。
- **关联概念**：扩展 → [[回落]]

### [[回落]]
- **定义**：当节点端口收到其无法解析识别的数据请求时，将该数据自动转交给预先设定的备用端口处理的路由机制 [9]。
- **视频上下文**：作者在演示复杂配置时，通过设置 reality 的回落功能，把无法处理的裸 xhttp 数据成功转发给内层搭建的 6666 端口的裸 xhttp 节点 [6, 9]。
- **为什么重要**：它是实现不同协议嵌套运作、欺骗防火墙探测（表面访问 reality，实际交互 xhttp）的核心设置 [9]。
- **关联概念**：属于 → [[reality]]
