## 一句话总结
讲者 [[不良林]] 详尽讲解了代理工具 [[sbox]] 的客户端操作指南及在软路由插件 [[home proxy]] 上的高级透明代理与分流配置。

## 目标受众与前置知识
- **适合谁看**：软路由用户、网络代理初学者，及希望解决 DNS 泄露并追求精细化流媒体解锁的高阶网络玩家（推断）。
- **前置知识**：软路由与 [[openwrt]] 基础知识、[[clash]] 与 [[open CL]] 使用经验、[[fakeip]] 概念、基础的 [[DNS]] 原理。

## 核心观点
1. **[[sbox]] 是目前最强大的网络代理工具**
   - 为什么重要：相较于其他工具，它既具备全面的服务端协议，又提供极其丰富的客户端配置，且保持极快的更新频率。
   - 依据：讲者明确指出其拥有比 [[v2a]] 系更全的服务端协议支持，又有比 [[clash]] 系更丰富的客户端配置（讲者原话）。
2. **结合 [[ecs]] 与 [[realip]] 可以优雅解决精确分流与 [[DNS]] 泄露问题**
   - 为什么重要：传统的 [[fakeip]] 模式有局限性（可能导致游戏断连或未收录的国内小众网站访问极慢），依靠配置良好的 [[realip]] 能兼顾隐私与访问精度。
   - 依据：讲者演示配置 [[ecs]]（填写国内 IP 1.0.1.0），通过权威 DNS 成功解析未被规则集收录的国内网站并获取正确的 CDN 节点 IP 从而实现直连（讲者演示）。
3. **流量嗅探（[[sniif]]）与覆写目标地址是解锁流媒体服务的关键**
   - 为什么重要：机场节点本身无法直接解锁特定流媒体或 AI 网站，必须将真实的域名提取出来并提供给节点服务器。
   - 依据：讲者演示访问 [[奈飞]] 报错失败后，在设置中开启“覆写目标地址”，通过嗅探提取出域名发给节点，节点再借此访问 [[sni]] 反代服务器从而完成解锁（讲者演示）。

## 概念关系
- [[fakeip]] ↔ [[realip]]：讲者建议放弃存在局限性（影响特定网络及链式代理环境）的 [[fakeip]]，转而拥抱符合真实网络环境请求逻辑的 [[realip]]。
- [[sbox]] → [[home proxy]]：[[home proxy]] 是运行在软路由上，直接调用 [[sbox]] 内核处理流量并实现透明代理的图形化控制插件。
- [[sniif]] → 覆写目标地址：必须先使用 [[sniif]] 嗅探出数据的真实域名，才能进行目标地址的复写并交由后端节点服务器去绕过封锁。
- [[clash API]] → [[wbui]] / [[dashboard]]：[[sbox]] 支持接入 [[clash API]]，使得没有原生界面的内核可以借用 [[clash]] 经典且便利的网页控制台（[[wbui]]/[[dashboard]]）来切换路由和节点策略。

## 详细笔记
### 基础与客户端上手演示
讲者 [[不良林]] 介绍 [[sbox]]（即 sing-box）是当今最强代理工具，客户端逻辑对齐 [[clash]]。在移动平台与 PC 端，官方已支持 [[苹果]] 生态（包含 [[iOS]]、[[micos]]、[[tvos]]），直接在 [[APP STORE]] 下载即可；暂无官方支持的 [[Windows]] 系统则可用第三方工具 [[v2n]] 替代。在 [[安卓]] 等手机端，只需在界面内导入机场订阅链接，即可自动完成分流实现科学上网。

### 软路由环境与 [[home proxy]] 基础设置
在基于 [[emoto wrt23.05]] 固件的 [[openwrt]] 软路由上，讲者演示安装 [[home proxy]] 插件。该插件界面比 [[open CL]] 更加简洁。导入机场节点后，演示了基于局域网的访问控制设定：通过底层配合 [[nf tables]] 防火墙，可设置仅允许白名单特定设备通过代理内核。

### 进阶修改与启用 [[clash API]] 
由于原版 [[home proxy]] 缺失网页管理端（[[wbui]]/[[dashboard]]），讲者提供并演示了如何安装他修改后的 ipk 插件，需搭配 1.9 版本以上的 [[sbox]] 内核使用（讲者原话）。演示添加规则标签以适配面板控制：建立自动选择的 [[uil text]]、手动选择的 [[select]] 以及强制大写名称的 [[Global]] 组。关闭 [[ipv6]] 支持。为节约软路由闪存空间，讲者推荐按需使用 `.srs` 的二进制格式规则集，例如将 200k 的文本规则压缩至仅 32k，并演示添加仅 22K 的中国网站规则（屏幕展示/讲者原话）。

### [[DNS]] 防泄漏模块与核心 [[ecs]] 设定
讲者演示配置 [[Google]] 的 [[doh]]（即 [[dnsoverhttps]]）服务器，将解析策略设为仅 [[ipv4]]。为避开 [[fakeip]] 弊端并防范 [[DNS]] 泄露，实施 [[realip]] 方案的核心在于配置 [[ecs]]：填写就近的中国 IP（如 1.0.1.0，讲者原话），部分注重隐私的服务（如 [[cloudfare]]）不支持此特性。讲者说明，部分 [[VPN]] 测漏网站的提示可忽略，重点在于未泄露国内 DNS 即可；另外也支持配置加密等级更高的 [[dot]]（推断自全局理解）。

### [[sbox]] 代理工作原理解剖（以访问 [[Google]] 为例）
讲者利用 [[sftp]] 将软路由配置导出并在本地解析，将其拆解为四大模块：[[DNS]]（含 [[servers]] 与 [[ROOT]] 规则）、[[inb]]（入站）、[[outb]]（出站）以及 [[ROOT]] 路由。访问 Google 时：请求先被 [[dnsmc]] （推断自 dnsmasq 语境）劫持至内核的 533 端口（讲者原话），因未命中路由规则，交由 114 默认 DNS 并通过 [[Direct]] 出站。获取节点后，请求通过 [[SS]] / [[sarsocks]] 协议和 [[as]] 加密，发往节点的 10001 端口进行解析。

### 流量重定向嗅探与精确分流测试（访问 [[百 度]] 与 [[哔哩哔哩]]）
以访问 [[百 度]] 为例，流量重定向（[[redirect]]）至 5331 端口后，执行 [[sniif]] 嗅探出 [[HTTPS]] 协议和具体域名，最终命中中国规则从而走直连。面对未被收入规则的小众网站（如 [[哔哩哔哩]]），凭借上方配置的 [[ecs]]，Google DNS 会返回准确的国内 CDN 的 IP（例如 5.5.5.5，推断自屏幕），从而成功命中 IP 规则实现完美直连（[[Direct]]）。此时测试网站漏出真实 IP，实际上是由于上游规则将其误判入中国网站集导致。

### 流媒体（[[奈飞]]）与 [[chatgpt]] 解锁原理
讲解机场节点本身无力解锁流媒体。必须开启“覆盖目标地址”，利用 [[sniif]] 嗅探获取如 netflix.com 等真实域名，覆盖掉原有的 IP 并在封装后发往节点。节点借此重新解析，获取专门用来解锁的 [[sni]] 反代服务器 IP 进行访问从而成功解锁 [[奈飞]]。另外提到，[[v2n]] 和 [[x 越安]] 默认开启此项，但在暗网假域名访问时必须关闭以免网络断连。针对 [[chatgpt]] 解锁失败，讲者解释它采用基于 [[udp]] 的 [[quck]]（即 [[http3]]）协议，由于无法嗅探到域名导致复写失败，需在浏览器屏蔽 QUIC 以降级至 [[http2]] 才能成功嗅探复写。

### 客户端配置复用与自定义策略
最后演示了如何对软路由的 json 文件稍作修剪即可移植给本地客户端（创建 [[loco]] 配置）：移除本地环境无需的配置、删掉旧的 [[inb]] 并替入嗅探覆写、配置 [[阿 里]] 公网 DNS（223.5.5.5，推断自屏幕），并绑定 127.0.0.1 启动。讲者更进一步演示了创建自定义的 [[Block]]（屏蔽）模式，展示了极高的自由度。局域网共享还可以创建 [[Mix]] 入站（讲者原话）。

## 常见坑与解决方案
- **问题描述**：安装 [[homeproxy]] 后服务页面未显示插件
  - 解决方案：刷新网页，或退出 [[OpenWrt]] 重登。
- **问题描述**：安装修改版 [[homeproxy]] 后提示错误或 Clash 模式无法开启
  - 解决方案：因旧配置文件未删除，需使用新配置文件替换旧配置文件。
- **问题描述**：使用 [[fakeip]] 导致部分游戏无法连接或链式代理出错
  - 解决方案：放弃使用 [[fakeip]]，改为配置并使用 [[realip]] 搭配 ECS。
- **问题描述**：奈飞等流媒体解锁失败（提示页面不存在）
  - 解决方案：开启“覆盖目标地址”（嗅探覆写），将嗅探到的域名而非真实 IP 发送给节点，以命中服务端的 DNS 解锁规则。
- **问题描述**：开启嗅探覆写后，智能家居无法连接或无法进入 Tor 暗网
  - 解决方案：这些服务使用随机假域名，需关闭嗅探覆写功能才能正常访问。
- **问题描述**：ChatGPT 解锁失败（即使开启了嗅探覆写）
  - 解决方案：由于 ChatGPT 访问使用了基于 UDP 的 QUIC 协议（HTTP/3），无法嗅探到域名。需关闭浏览器的 QUIC 功能，或添加规则屏蔽 QUIC 流量/HTTPS DNS 查询，强制其回退到 HTTP/2。
- **问题描述**：将软路由配置复制到本地客户端后启动报错
  - 解决方案：将 Clash API 绑定的外部控制 IP 从路由器 IP 更改为 `127.0.0.1`。

## 最佳实践
- **推荐做法**：使用二进制格式（`.srs`）的规则集代替文本格式（`.json`）或巨大的 Geo 数据库，文件体积极小（如 32K），可大幅节省软路由存储空间并提升灵活性。
- **推荐做法**：配置 ECS 时，为了个人隐私，建议填入同地区或国内的其他公共 IP（如 `1.0.1.0`），不要填入自己的真实公网 IP。
- **推荐做法**：推荐使用 [[doh]] 进行节点 DNS 解析，不仅因为其具有 HTTP 的长链接特性（延迟更低），也因为它比单纯的 UDP 解析对机场节点的兼容性更好。
- **反模式**：盲目全局开启“覆盖目标地址”（嗅探覆写）。该功能得按需启用，在特定的去中心化网络环境或智能家居场景中会导致网络连通性彻底中断。

## 关键概念

### [[sing-box]]
- **定义**：目前强大的网络代理工具，支持比 v2ray 系更全面的服务端协议和比 clash 系更丰富的客户端配置[1]。
- **视频上下文**：作者将其作为贯穿全文的核心工具，演示了如何在手机、电脑和软路由客户端上进行基础和进阶的网络代理配置[1, 2]。
- **为什么重要**：它是本期视频讲解所有代理过程、分流策略及 DNS 处理的底层内核核心[1-3]。
- **关联概念**：扩展 → [[clash API]]
