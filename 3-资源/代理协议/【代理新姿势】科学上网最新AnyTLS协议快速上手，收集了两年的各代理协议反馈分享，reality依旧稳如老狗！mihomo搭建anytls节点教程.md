## 一句话总结
不良林回顾了各类代理协议的存活率，并详细演示了如何使用 [[AnyTLS]] 协议及 [[mihomo]] 搭建高防封锁节点。

## 目标受众与前置知识
- **适合谁看**：对翻墙稳定性要求高、希望自行搭建科学上网节点以避免机场被攻击断网的用户。
- **前置知识**：[[VPS]] 基础操作、[[TLS]] 证书基础、[[代理协议]] 概念（推断）、[[命令行终端]] 与 [[CMD]] 操作（推断）。

## 核心观点
1. **自行搭建节点比使用机场在特定情况下更稳定**
   - 为什么重要：机场容易受网络攻击导致断网，自建节点由于无利益相关，通常不会被攻击。
   - 依据：「前两天本频道推荐的机场因遭受网络攻击更换前置高防期间断线持续近一天时间，并做出了五天补偿」（讲者原话）、「自己搭建自己用的话不涉及到什么利益相关一般都不会有人攻击」（讲者原话）。

2. **[[reality]] 协议目前具备极高的防封锁稳定性**
   - 为什么重要：对比旧协议的极高封锁率，该协议能大幅降低用户的维护成本。
   - 依据：讲者收集两年反馈发现 [[W]]+[[vision]]+[[reality]] 节点「一粒被封的反馈都没有」（讲者原话）；相比之下，[[SS]] 节点搭建第二天大概率被封IP，[[WMS]]+[[WS]] 节点自2024年3月4号（讲者原话）后大概率且循环被封端口。

3. **[[AnyTLS]] 通过灵活自定义填充参数有效突破防火墙识别**
   - 为什么重要：解决了传统流控被特征识别或灵活性不足的问题。
   - 依据：过去的 [[vision]]（又称 [[微卷]]）流控只能对开头几个短包做固定长度填充；而 [[AnyTLS]] 可以通过参数指定填充的数据包序号以及具体填充长度，具备更高的灵活性和个性化设定空间，增加了 [[防火墙]] 的识别难度（讲者原话）。

## 概念关系
- [[TLSTTLS]] → 防火墙阻断：说明「在 [[TLS]] 里传输 TLS 数据」的双层特征已经被 [[防火墙]] 明确识别并达成共识。
- [[nip.io]] 域名证书 → 节点无法使用：说明近期（自今年5月4号，讲者原话）部分 [[WinL]]+[[vision]] 节点失效是因为该特定域名被阻断，而非 [[vision]] 流控本身被识别。
- [[vision]] ↔ [[AnyTLS]]：两者同为消除 TLS 特征的方案，但 [[vision]] 为固定填充，[[AnyTLS]] 允许高度自定义的灵活填充。
- [[TLS GO]] ↔ [[mihomo]] / [[SBOX]]：对比搭建工具，[[TLS GO]] 仅为官方验证协议的 demo（默认用自签证书），而 [[mihomo]] 和 [[SBOX]]（[[Sinbox]]）适合实际配置正规域名证书搭建。

## 详细笔记
### 节点搭建背景与两年协议封锁反馈
- 机场因利益相关常受网络攻击，讲者频道推荐的机场近期断线近一天并给了五天补偿（讲者原话），建议高要求用户使用 [[VPS]] 自建。
- **历史协议存活率总结**：
  - [[SS]]：搭建使用后第二天大概率被封 IP（讲者原话）。
  - [[WMS]]+[[WS]]：搭建第二天大概率被封端口，更换端口后会陷入循环封锁，自 2024年3月4号（讲者原话）起陆续收到大量反馈，现已不推荐。
  - [[WinL]]+[[微卷]]（[[vision]]）：自今年5月4号（讲者原话/即视频录制10天前）起部分反馈不可用。排查发现是由于教程中使用的 [[nip.io]] 域名证书被 [[防火墙]] 阻断，与协议和 [[vision]] 流控本身无关。
  - [[W]]+[[vision]]+[[reality]]：两年来 0 被封反馈（讲者原话），「稳如老狗」。白名单地区用户若不可用，改用大厂域名即可恢复正常。

### [[AnyTLS]] 协议原理与客户端支持生态
- 核心原理：解决 [[TLSTTLS]]（在 TLS 里传输 TLS）带来的明显流量特征。相比于只能固定填充的 [[vision]] 流控，[[AnyTLS]] 支持用户自定义要填充的数据包范围及其字节大小，极大增加了被识别的难度。
- 支持生态：
  - **服务端/内核**：官方 [[TLS GO]]，[[SBOX]]（或称 [[Sinbox]]），[[mihomo]]（前身是 [[Clashm]]）。
  - **移动端**：[[苹果]] / [[Apple Store]] 的 [[小火箭]]，[[安卓]] 的 [[N Box]]（或称 [[Nobox]]）。
  - **暂不支持**：[[XR]] 目前不支持且大概率不会支持，因其自身拥有未发布的 [[VG set]] 方案支持自定义（讲者原话）。

### 使用官方 [[TLS GO]] 快速上手搭建（测试级）
- **服务端部署**：
  - 在 [[VPS]] 上通过 wget 下载最新版本 008（讲者原话）并解压出三个文件。
  - 仅需执行一行代码 `aning TLS server` 即可运行在 8443 端口（讲者原话）。
- **客户端部署 ([[Windows]]/[[Mac]])**：
  - 下载对应系统版本解压后，在 [[CMD]] 或命令行终端执行带服务器 IP、端口（需改为 8443）、密码的 `AnyTLS client` 指令。
  - 客户端将在本地监听 1080 端口（讲者原话）开启 [[socks]] 代理。
  - 借助 [[WN]]（或称 [[n]]）等代理工具添加本地 [[socks]] 节点实现分流。
- **移动端配置**：
  - 升级 [[小火箭]] 或 [[N Box]] 到最新版，填入 IP、端口及密码。
  - **关键点**：由于 [[TLS GO]] 默认使用自签证书，必须在客户端启用「允许不安全」才能成功连接。

### 使用 [[mihomo]] 进行进阶自定义搭建（生产级）
- **为什么使用 mihomo**：[[TLS GO]] 仅为 demo 不便配置自有证书，若需使用自有证书建议用 [[mihomo]]（或 [[SBOX]]）。
- **服务端部署与配置**：
  - 在 [[VPS]] 生成自签证书或向 [[CA]] 机构申请正规证书。
  - 配置 [[mihomo]] 的监听端口（如 443，讲者原话）与证书路径。
  - **配置核心 padding 规则**：可以灵活设置前八个数据包（讲者原话）的填充字节范围。例如给第0个包填充30字节，第1个包填充100到400之间的随机字节大小，第2个包拆分为先发送400到500字节，若还有数据继续发送500到1000字节等（定量数据均来自讲者原话演示配置）。若不配置，协议自带上述默认八个包填充方案。
  - 下载重命名并赋予可执行权限后，运行 [[mihomo]] 即可在控制台看到访问日志。
- **使用正规域名证书的特别提醒**：
  - 若使用的是正规 [[CA]] 机构签发的证书，客户端需「取消启用允许不安全」，并在 SNI 处填入对应的域名证书名称。

## 常见坑与解决方案
- **问题描述**：使用视屏同款 VPS 搭建并使用 `[[SS]]` 节点，第二天大概率会被封 IP [1]。
  - 解决方案：未明确提及。
- **问题描述**：搭建 `[[VMess]]` + `[[WS]]` 节点，第二天大概率被封端口，更换端口后继续被封，陷入死循环 [1]。
  - 解决方案：在同一台 VPS 上改为搭建并使用 `[[Reality]]` 节点 [2]。
- **问题描述**：搭建的 `[[VLESS]]` + `[[Vision]]` 节点无法使用，但端口未被封锁 [2]。
  - 解决方案：防火墙阻断了教程中使用的 nip.io 域名，换个大厂域名或者改为自签证书即可解决 [2, 3]。
- **问题描述**：在手机客户端中没有找到 `[[AnyTLS]]` 协议选项 [4, 5]。
  - 解决方案：进入应用商店将 `[[Shadowrocket]]` 或 `[[NekoBox]]` 更新到最新版本 [4, 5]。

## 最佳实践
- 对节点稳定性要求较高的用户，建议自己搭建 `[[Reality]]` 节点，目前反馈极其稳定，很少有被封的情况 [1, 2]。
- `[[AnyTLS-GO]]` 只是为了验证 `[[AnyTLS]]` 协议的一个 demo，默认使用自签证书且无法自己配置证书；如果要使用自定义域名证书，建议使用基于 `[[Sing-box]]` 或 `[[mihomo]]` 内核的代理工具来搭建节点 [5]。
- 如果使用的是正规 CA 机构颁发的域名证书，建议在客户端中取消启用“允许不安全”选项，并在 SNI 设置中填入你的域名 [7]。

## 关键概念

### [[AnyTLS]]
- **定义**：一种可通过参数指定填充数据包及填充长度的全新代理协议，用于消除流量特征 [1, 2]。
- **视频上下文**：作者作为本期教程核心进行介绍，演示了使用官方项目及 Mihomo 搭建该协议节点的方法，以解决节点被墙的问题 [1, 2]。
- **为什么重要**：相比以往协议，它能进行更加灵活和个性化的设定，从而增加防火墙识别的难度 [2]。
- **关联概念**：对比 → [[Vision流控]] [2]

### [[Reality]]
- **定义**：一种代理节点协议，在作者两年的反馈收集中表现出极高的稳定性 [1, 3]。
- **视频上下文**：作者推荐对稳定要求高的用户自行搭建使用，指出该协议目前“稳如老狗”，且没有任何被封的反馈 [1, 3]。
- **为什么重要**：作为目前极其稳定的代理节点方案，能有效避免因节点端口被封锁导致的断网问题 [1, 3]。
- **关联概念**：未明确提及。

### [[Vision流控]]
- **定义**：一种通过对开头的几个短数据包填充指定长度的随机数据，来消除流量特征的机制 [2, 3]。
- **视频上下文**：作者在介绍背景时提到，它的出现是为了解决 TLS in TLS 被识别的问题，但其填充是固定的，无法进行自定义 [2, 3]。
- **为什么重要**：它是解决流量特征被识别的早期重要方案，也是理解 AnyTLS 协议灵活特性的重要背景参照 [2, 3]。
- **关联概念**：对比 → [[AnyTLS]] [2]

### [[TLS in TLS]]
- **定义**：在 TLS 隧道中传输 TLS 数据的现象，其流量特征能够被防火墙成功识别 [3]。
- **视频上下文**：作者在解释为什么需要引入 Vision 流控和 AnyTLS 协议时，将其作为核心痛点和协议开发的背景原因提出 [2, 3]。
- **为什么重要**：它是导致许多基于 TLS 的代理节点被防火墙阻断的核心特征 [3]。
- **关联概念**：未明确提及。

### [[Mihomo]]
- **定义**：一款前身为 Clash.Meta 的代理工具，可作为内核用于搭建和高级配置代理节点 [2]。
- **视频上下文**：作者演示了使用它作为服务端搭建 AnyTLS 节点，指出它比官方演示工具更适合用来配置自定义域名证书和自定义数据填充规则 [2, 4, 5]。
- **为什么重要**：提供了比官方基础工具更完善的配置能力，是实际部署高级 AnyTLS 节点的重要工具 [4, 5]。
- **关联概念**：未明确提及。

### [[数据填充]]
- **定义**：向数据包中加入指定或随机大小的字节（如 100 到 400 之间），以消除流量特征的操作 [3, 5]。
- **视频上下文**：作者在讲解 Vision 流控的原理以及 Mihomo 的 AnyTLS 关键配置时，详细解释了如何对前几个数据包进行自定义字节大小的填充或策略拆分 [3, 5]。
- **为什么重要**：它是 AnyTLS 和 Vision 流控抵抗防火墙识别的核心底层机制 [2, 3, 5]。
- **关联概念**：属于 → [[AnyTLS]] [2, 5]
