# 全程不碰终端，用龙虾（OpenClaw）对话部署 sub2api

> 今天做了一件以前觉得很麻烦的事——部署一个开源项目。
> 区别是：我全程没有打开过任何终端，没有 SSH，没有登录服务器。
> 我只是在 Telegram 里跟龙虾聊了几句。

---

## 背景：龙虾是什么

[[OpenClaw Docker部署保姆级教程]] 里写过部署过程，简单说：

OpenClaw（中文叫龙虾）是一个开源 AI Gateway，可以把大模型（Claude、DeepSeek 等）接入 Telegram。部署完成后，你有一个 7×24 在线的 Telegram Bot，它不只是聊天，**它本质上是一个可以帮你操作服务器的 AI 助手**。

---

## 今天做了什么

我想部署 [sub2api](https://github.com/sub2api)——一个把订阅链接转换为 API 格式的开源服务。

以前的流程大概是：

1. SSH 登录服务器
2. 找到合适的部署方式（Docker / 手动编译）
3. 写 docker-compose.yml 或者配置环境变量
4. 处理报错
5. 配置反代、防火墙、开机自启……

**今天的流程：**

打开 Telegram，找到龙虾 Bot，告诉它我想部署什么，然后等它搞定。

---

## 实际对话长什么样

核心体验是：你描述意图，龙虾理解并执行。

类似这样的交互（示意）：

```
我：帮我部署 sub2api，用 Docker，映射到 8080 端口

龙虾：好的，我来处理。正在拉取镜像…
      ✅ 镜像拉取完成
      正在创建容器…
      ✅ 容器已启动，端口 8080 已映射
      需要配置反代或开机自启吗？

我：加上开机自启

龙虾：✅ 已设置 restart: unless-stopped
```

整个过程没有一行命令是我手打的。

---

## 为什么这件事让我觉得值得记录

不是因为 sub2api 有多复杂，而是因为这个**交互模式的转变**：

过去，服务器是我要"登录进去操作"的对象。
现在，服务器是龙虾帮我管理的基础设施，我只需要表达意图。

这跟用 Vercel 点点鼠标部署前端是两回事——龙虾是运行在**我自己服务器上**的，我保有完整控制权，只是操作方式变了。

这个模式的影响：
- **降低了运维摩擦**：不需要记命令、不需要查文档，用自然语言描述就行
- **适合间歇性维护**：平时不用 SSH 的场景，偶尔需要操作时对话即可
- **有审计意识**：龙虾会告诉你它做了什么，比自己盲操更透明

---

## 适合用这个方式部署什么

理论上任何能 Docker 化的开源项目都行。典型场景：

| 场景 | 示例项目 |
|------|---------|
| API 代理 / 转换 | sub2api、one-api、new-api |
| 订阅管理 | subconverter |
| 自建服务 | Bitwarden、Vaultwarden |
| 数据工具 | Metabase、Grafana |

前提是龙虾已经部署好并接入了有足够能力的模型。

---

## 前置条件

1. 已经部署好 OpenClaw 并连接 Telegram（见 [[OpenClaw Docker部署保姆级教程]]）
2. OpenClaw 配置了具备代码执行能力的模型
3. `tools.exec` 配置为 `ask`（每次执行会征询你的确认）

---

> **L3 总结：** OpenClaw 的 Telegram Bot 不只是聊天工具，它是一个对话式服务器助手。今天通过它完成了 sub2api 的部署，全程零终端操作。这个模式的本质是：把"命令式运维"变成"意图表达"，降低了操作摩擦，同时保留了服务器的完整控制权。
