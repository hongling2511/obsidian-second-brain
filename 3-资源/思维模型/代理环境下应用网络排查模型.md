# 代理环境下应用网络排查模型

## 背景

将 OpenClaw 从 Docker 迁移到宿主机直装后，Telegram bot 能收消息但模型 API 调用挂死。经过系统性排查，定位到代理对流式响应的处理问题。

## 排查分层模型

### 第一层：隔离变量法

> 逐层验证网络链路中的每个环节，确定问题在哪一层。

| 测试组合 | 工具 | 目标 | 目的 |
|----------|------|------|------|
| curl 直连 | curl | API | 验证 API 本身可用 |
| curl + 代理 | curl --proxy | API | 验证代理转发正常 |
| Node.js 直连 | fetch() | API | 验证语言运行时可用 |
| Node.js + 代理库 | undici.ProxyAgent | API | 验证代理库可用 |
| 应用 + 代理 | 应用内部 | API | 验证应用集成可用 |

**关键原则：** 每次只改变一个变量。如果前四层都通过但第五层失败，问题在应用的代理集成方式。

### 第二层：连接状态观测法

> 用系统工具观测实际网络行为，而非依赖应用日志。

```bash
# 查看进程的 TCP 连接状态
ss -tnp | grep <pid>

# 关注 recv-q（接收缓冲区未读数据）
# recv-q > 0 且长时间不变 → 应用没有读取数据（卡住）
# recv-q = 0 → 正常等待或数据已处理

# 用 strace 追踪系统调用
strace -p <pid> -e trace=read,recvfrom -f -s 600
```

**关键指标：** `recv-q` 持续不变说明应用层没有消费数据，问题在应用内部而非网络。

### 第三层：代理作用域分析法

> 区分「谁需要代理」和「谁不需要代理」。

```
应用
├── 模块A（Telegram）→ 需要代理 → 配置 proxy
├── 模块B（aicoding API）→ 不需要代理 → 应走直连
└── 问题：应用把模块A的 proxy 当全局代理 → 模块B也走了代理
```

**排查步骤：**
1. 用 `ss -tnp` 查看所有出站连接的目标地址
2. 如果本该直连的请求也走了代理 → 应用的代理作用域有问题
3. 解决方案：环境变量 `NO_PROXY`、TUN 模式、或应用配置修复

### 第四层：流式响应兼容性检查

> 代理对 HTTP 流式响应（SSE/chunked transfer）可能有特殊处理。

**常见问题：**
- 代理缓冲整个响应后才转发（破坏流式传输）
- HTTP CONNECT 隧道中的 TLS 握手超时
- 代理对长连接的超时设置过短

**验证方法：**
```bash
# 关闭应用的流式传输，测试非流式是否正常
# 如果非流式正常 → 确认是代理的流式处理问题

# 或直接绕过代理测试
# 如果绕过代理正常 → 确认是代理层问题
```

## 解决方案优先级

| 方案 | 侵入性 | 适用场景 |
|------|--------|---------|
| **TUN 模式**（透明代理） | 低 | 代理软件支持 TUN，最干净 |
| **代理分流规则** | 低 | 在代理端设置特定域名直连 |
| **应用配置调整** | 中 | 关闭 streaming、修改代理作用域 |
| **NO_PROXY 环境变量** | 中 | 应用支持读取 NO_PROXY |
| **iptables 透明代理** | 高 | 没有 TUN 模式时的替代方案 |

## 经验教训

1. **Docker → 宿主机迁移时，网络环境完全不同**
   - Docker 容器内的 DNS、路由、代理方式与宿主机不同
   - 不能假设 Docker 内能工作的配置在宿主机也能工作

2. **Node.js 的 fetch/undici 不等于 curl**
   - `curl` 支持系统代理环境变量，Node.js 的 `fetch` 不自动支持
   - Node.js 的 Happy Eyeballs (IPv6/IPv4 双栈) 可能导致连接超时
   - `--dns-result-order=ipv4first` 可以缓解 IPv6 问题

3. **应用级代理 vs 系统级代理**
   - 应用配置的 proxy 可能影响范围超预期（全局化）
   - 系统级透明代理（TUN）对应用无侵入，是更可靠的方案

4. **recv-q 是排查网络卡死的利器**
   - 持续不变的 recv-q 直接说明问题在应用层
   - 结合 strace 可以确认应用是否在尝试读取数据
