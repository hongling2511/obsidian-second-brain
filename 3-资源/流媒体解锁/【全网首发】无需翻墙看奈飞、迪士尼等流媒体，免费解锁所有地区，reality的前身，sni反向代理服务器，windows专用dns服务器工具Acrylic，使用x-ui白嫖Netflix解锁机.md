## 一句话总结
讲者 [[不良林]] 介绍在不翻墙情况下使用 [[Acrylic]] 和 [[sni反代服务器]] 解锁 [[Netflix]] 的方法，并解析了 [[reality协议]] 的优缺点 [1-4]。

## 目标受众与前置知识
- **适合谁看**：需要免翻墙观看流媒体的重度电影爱好者，以及节点本身无法解锁流媒体的用户 [1, 5]。
- **前置知识**：[[DNS]]、[[sni反代服务器]]、[[host文件]]、[[x-ui面板]]、[[reality协议]]、[[GFW]]（推断：防火墙概念）。

## 核心观点
1. **本地 DNS 配合 sni 反代可免翻墙解锁特定流媒体**
   - 为什么重要：这种方式不会消耗用户的节点流量，适合重度观影者 [1]。
   - 依据：「这种方式不会消耗你的节点流量」（讲者原话）[1]。由于 [[Netflix]] 和 [[Disney]] 只是不对中国大陆提供服务，请求并未被拦截 [1, 2]。
2. **sni 反代具有特定局限性**
   - 为什么重要：防止误用导致连接阻断或 IP 被封禁 [3, 6]。
   - 依据：讲者指出，如果用来反代 [[YouTube]]、[[Google]] 等被封锁的网站，会被 sni 阻断，并且 IP 很快就会被墙 [6]。
3. **reality 协议的实际应用存在隐患**
   - 为什么重要：打破对新协议的盲目推崇，提示其实际搭建门槛和稳定性问题 [4, 7]。
   - 依据：由于协议每次建立连接都要访问目标网站拿证书，如果找大型网站（如 [[Microsoft]]），容易被误认为 CC攻击 而被屏蔽；如果找同 IP 段邻居或自建网站，则丧失了无域名证书的优势，甚至不如原生 [[udp]] 或 [[tls]] [4, 7]。

## 概念关系
- [[Acrylic]] → [[DNS]]：利用在 [[Windows]] 上安装的 [[Acrylic]] 工具提供本地的 [[DNS]] 服务 [2]。
- [[sni反代服务器]] → [[Netflix]]：通过本地 [[DNS]] 劫持，将访问流媒体的请求转发给 [[sni反代服务器]]，从而获取真实数据 [3]。
- [[APP客户端]] ↔ 本地 [[DNS]]：部分 APP 客户端硬编码了 [[Google]] 的 [[DNS]] 服务器，会绕过本地设置，导致该免翻墙方法失效 [6]。

## 详细笔记
### 视频引言与免翻墙原理
讲者 [[不良林]] 演示在未开启代理的情况下访问 [[Netflix]]，指出网页返回“不可用”是因为奈飞本身不对中国大陆提供服务，而不是被防火墙拦截 [1]。同理 [[Disney]] 也可访问但无服务 [2]。由此引入通过修改本地 [[DNS]] 解锁流媒体的方法 [1]。

### 工具安装与 DNS 服务配置
在 [[Windows]] 系统上安装 [[Acrylic]] 工具以提供本地 [[DNS]] 服务 [2]。修改其配置文件，将 DNS 替换为通过命令行查看到的路由器默认网关 IP，并添加指令以允许局域网内其他设备使用 [2]。

### 获取 sni 反代服务器与修改 host
通过特定链接获取 [[sni反代服务器]] IP [8]。尝试在前面加入 [[HTTPS]] 访问，如果网页出现不支持的协议，“有95%的概率可以使用”（讲者原话）[8]。将选定的 IP 替换到讲者提供的 [[host文件]] 模板中，并追加至 [[Acrylic]] 的 [[host文件]] 底部 [8, 9]。

### 浏览器端测试与合租平台推荐
将本机的本地 [[DNS]] 修改为 127.0.0.1 [9]。若由于浏览器缓存导致无法访问，需清除缓存、开无痕窗口，或更换其他地区（如香港、台湾）的 IP 并重启服务 [9, 10]。成功解锁后，讲者推荐了一个可购买 [[Netflix]]、[[Disney]]、[[ChatGPT]]、[[YouTube]] 等会员账号的合租网站，使用优惠码可获“93折的优惠”（讲者原话）[10]。

### sni 反代局限与解决不支持解锁的节点
强调该方法只能用于未被墙的网站，无法免翻墙看 [[YouTube]] 或 [[Google]] [6]。此外，[[APP客户端]] 可能因内置 [[Google]] DNS 绕过本地配置 [6]。随后演示利用不支持解锁的自建 [[vps]] 节点，在 [[x-ui面板]] 的配置模板中替换入可用的台湾 sni IP，实现流媒体解锁 [5]。也提示用户可自建 sni 反代服务器 [5]。

### 解析 reality 协议的优缺点
讲者指出当前未出正式版的 [[reality协议]] 与找不到代点的情况类似 [3, 4]。该协议优势在于无需申请域名和证书（像 [[SS]] 节点一样简单），且能维持真实 [[tls]] 连接 [4]。但它不适合新手，因为寻找能承受频繁访问的大型目标网站（如 [[Microsoft]]）极度困难 [4]。若退而求其次使用伪装网站，又会失去其自身优势，且容易暴露 sni 反代服务器本质，综合体验未必优于原生 [[tls]] 或原生 [[udp]] [7]。

2. **配置本地 [[DNS]] 服务器**
   - 编辑配置文件，将默认的 `8.8.8.8` 替换为路由器的 IP，并在最下方复制指令以允许其他设备使用本机提供的 [[DNS]] 服务 [2]。

   - 注意事项：修改完成后保存时，如果提示是否需要重启服务，可以先选择“否”，因为还需要进一步修改 host 文件 [3]。

3. **获取并配置 [[SNI]] 反代服务器**
   - 通过指定链接获取 [[SNI]] 反代服务器 IP，打开提供的 host 模板，将模板内的 IP 替换为反代服务器 IP，然后将其全部复制并追加到 [[Acrylic]] 的 host 文件最下方 [3]。
   - 关键代码/命令：未明确提及。
   - 注意事项：测试反代服务器时，如果网页出现“不受支持的协议”，有 95% 的概率可以使用；建议直接使用 IP，不要使用带有域名的反代 [3]。

4. **重启服务与修改网络设置**
   - 保存 host 文件后点击“是”重启服务，接着将本机网络设置中的 [[DNS]] 服务器修改为本地 IP [7]。
   - 关键代码/命令：未明确提及。
   - 注意事项：本机可以直接填入 `127.0.0.1`，如果是局域网的其他设备，需要填入本机的局域网 IP [7]。

5. **（可选）通过 [[x-ui]] 面板配置解锁**
   - 进入 [[x-ui]] 面板设置的 [[Xray]] 模板，将提供的模板中的两个 IP 替换为找到的可用地区 IP（如台湾 IP），替换掉原来的配置模板后保存并重启面板 [4]。
   - 关键代码/命令：未明确提及。
   - 注意事项：此步骤适用于已有自建节点但节点不支持解锁流媒体的情况 [4]。

## 常见坑与解决方案
- **问题描述**：配置完成后，仍然进不去奈飞官网 [7]。
  - 解决方案：可能是浏览器缓存导致，建议开启无痕窗口，或清除 [[DNS]] 缓存并重启浏览器；如果仍无法访问，说明该 IP 不可用，需要更换其他地区的 IP [7, 8]。
- **问题描述**：在 APP 客户端上无法使用该方式解锁观看 [9]。
  - 解决方案：部分客户端硬编码了谷歌的 [[DNS]] 服务器，会绕过本地设置，需要进行 [[DNS]] 劫持（视频中未演示具体操作流程，判定为过于费劲） [9]。
- **问题描述**：使用 [[SNI]] 反代访问 YouTube、Google 等被 GFW 封锁的网站 [9]。
  - 解决方案：不要这么做，因为 [[SNI]] 是明文的，会导致 [[SNI]] 阻断，严重时会导致 IP 被封 [5, 9]。
- **问题描述**：自建 [[SNI]] 反代服务器变成公共服务器被滥用 [5]。
  - 解决方案：必须做好 IP 限制 [5]。
- **问题描述**：搭建 [[reality]] 协议时连接不稳定或被目标网站屏蔽 [6]。
  - 解决方案：需要找一个符合条件的大型网站（如微软）作为目标，并且 VPS 和目标网站之间的速度要好以维持稳定连接；一般网站会误以为遭受 CC 攻击而屏蔽连接 [6]。

## 最佳实践
- 每次修改 host 文件之后，建议清除 [[DNS]] 缓存并且重启浏览器，或者使用无痕窗口尝试访问 [8]。
- 如果已经开启了代理工具，记得设置让奈飞走直连，或者直接关闭代理工具，否则仍然会消耗节点流量 [7]。
- 寻找 [[SNI]] 反代服务器时，更进一步可以在前面加入 [[HTTPS]] 访问来测试其连通性与可用概率 [3]。
- 反模式：不要使用 [[SNI]] 反代去反弹被墙的网站，这会导致 IP 被墙 [5, 9]。
- 反模式：搭建 [[reality]] 时不建议寻找同 IP 段的“邻居”伪装网站，也不推荐在自己的 VPS 上搭建网站将目标设置为自己，这样会丧失 [[reality]] 不用域名和证书的优势，甚至不如直接使用原生 [[udp]] 或 [[tls]] [10]。
- 反模式：[[reality]] 搭建看似不需要申请证书很简单，但其实由于找目标网站等难点，并不适合新手 [6]。

## 关键概念

### [[SNI 反向代理]]
- **定义**：负责端口转发，能够根据 SNI 判断是否帮忙转交网络请求的服务器 [1]。
- **视频上下文**：作者指导用户寻找公开的 SNI 反代服务器 IP，将发往奈飞的请求转发给它，由其再转发给真正的奈飞，从而免翻墙解锁流媒体 [1, 2]。
- **为什么重要**：它是本视频实现免翻墙观看奈飞的核心机制，且服务器仅负责转发而不会对数据做任何处理 [1]。
- **关联概念**：依赖 → [[DNS 劫持]]

### [[DNS 解锁]]
- **定义**：通过修改 DNS 设置或使用特定代理机制，实现不消耗代理节点流量也能观看受地区限制的流媒体（如奈飞）的方法 [3]。
- **视频上下文**：视频开篇介绍这是不翻墙看奈飞的基础原理，也是整个教程的核心目标 [3]。
- **为什么重要**：对于重度电影爱好者而言，这种方式不会消耗宝贵的节点流量 [3]。
- **关联概念**：依赖 → [[Acrylic]]

### [[Acrylic]]
- **定义**：一款可以让 Windows 系统提供本地 DNS 服务的专用工具 [4]。
- **视频上下文**：作者演示了下载安装该工具、修改其默认 DNS 服务器为路由器 IP，并编辑其附加指令以允许局域网其他设备使用 [4]。
- **为什么重要**：是在 Windows 环境下搭建本地自定义 DNS 服务并使本地 host 规则生效的基础软件 [2, 4]。
- **关联概念**：扩展 → [[hosts 文件]]

### [[hosts 文件]]
- **定义**：用于将目标域名强制映射到指定 IP 地址的配置文件 [2]。
- **视频上下文**：作者演示了将奈飞等流媒体域名与获取到的 SNI 反代服务器 IP 填入 Acrylic 的 hosts 文件中 [2, 5]。
- **为什么重要**：通过修改该文件，配合本地 DNS，才能将直连流媒体的请求重定向到反代服务器上 [2, 6]。
- **关联概念**：依赖 → [[SNI 反向代理]]

### [[DNS 劫持]]
- **定义**：在本地强行将访问目标网站的请求截获，并重定向到另一台服务器（如 SNI 反代服务器）的操作 [1]。
- **视频上下文**：作者解释本地通过 DNS 劫持将访问奈飞的请求截持到 SNI 反代服务器；同时也提到 APP 客户端若硬编码了谷歌 DNS 会绕过本地设置，必须在路由器层面进行更费劲的 DNS 劫持才能解决 [1, 7]。
- **为什么重要**：揭示了修改 DNS 和 hosts 从而改变网络请求流向的底层网络原理 [1]。
- **关联概念**：依赖 → [[SNI 反向代理]]

### [[reality]]
- **定义**：一种不需要域名和证书、能维持真实 TLS 连接的新型代理协议 [8]。
- **视频上下文**：作者在视频后半段探讨了该协议，指出其搭建像 SS 节点一样简单，但需要寻找符合条件的目标网站拿证书，对网络连通性和服务器性能要求极高 [8]。
- **为什么重要**：它是目前备受关注的新型科学上网协议，但由于存在选定目标网站困难、易被识别为 SNI 反代服务器或遭到探测等缺陷，作者认为其暂不适合新手 [8, 9]。
- **关联概念**：依赖 → [[TLS]]

### [[TLS]]
- **定义**：未明确提及 [1, 8]。
- **视频上下文**：作者提到使用过 TLS 节点的用户对 SNI 不会陌生，且 reality 协议的区别在于能维持一条真实的 TLS 连接 [1, 8]。
- **为什么重要**：它是 reality 协议赖以实现其伪装特性的核心连接形式 [8]。
- **关联概念**：属于 → [[reality]]

### [[防火墙]]
- **定义**：负责拦截或阻断中国大陆地区用户访问特定境外网站的网络审查系统 [3, 7]。
- **视频上下文**：作者说明奈飞并没有被防火墙拦截，而是奈飞主动拒绝大陆用户；但如果试图用 SNI 反代已被防火墙封锁的 YouTube 或谷歌，会导致 SNI 阻断且 IP 被墙 [3, 7]。
- **为什么重要**：明确了 SNI 反代技术的局限性——它只能反代未被防火墙主动封锁的网站 [7]。
- **关联概念**：对比 → [[SNI 反向代理]]
