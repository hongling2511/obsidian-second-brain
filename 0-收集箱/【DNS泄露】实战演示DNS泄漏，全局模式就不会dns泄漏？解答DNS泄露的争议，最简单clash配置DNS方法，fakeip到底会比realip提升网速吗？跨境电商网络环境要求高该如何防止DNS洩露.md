## 一句话总结
[[不良林]]实战演示[[DNS]]泄露触发原因，破解[[全局模式]]防泄露误区，对比[[fakeip]]与[[realip]]速度。

## 目标受众与前置知识
- **适合谁看**：对网络环境要求高的跨境电商从业者、希望解决[[DNS]]泄露争议的科学上网用户（推断）。
- **前置知识**：[[DNS]]、[[IP]]、[[代理]]、[[Clash]]、[[fakeip]]、[[realip]]、[[分流]]。

## 核心观点
1. **[[全局模式]]无法绝对防止[[DNS]]泄露**
   - 为什么重要：打破了用户的普遍误解，证明多端因素都会导致泄露。
   - 依据：在[[Clash for Window]]启用[[TUN模式]]和[[全局模式]]下，使用[[wsock]]抓包发现，[[redir-host]]（视作rehost）模式、[[Windows]]的[[智能多宿主名称解析]]以及浏览器的[[quic]]协议（基于[[udp]]）都会引发发往国内服务器的真实[[DNS]]请求（屏幕展示/讲者原话）。
2. **[[DNS]]泄露的争议源于定界不同**
   - 为什么重要：解释了社区中关于“漏不漏”的分歧原因。
   - 依据：跨境电商要求严苛，任何非节点发起的业务[[DNS]]请求（如访问[[PayPal]]）都算泄露；而多数普通用户只在乎[[Google]]、[[YouTube]]等黑名单网站是否明文泄露（讲者原话）。
3. **普通用户应极简配置上游[[DNS]]**
   - 为什么重要：避免过度复杂的配置导致不稳定或新的安全/连通性隐患。
   - 依据：直接使用运营商分配的路由器[[IP]]或[[dhcp]]即可；国内的[[doh]]依然会被服务商监控，而国外的[[doh]]直连会被墙（讲者原话）。
4. **[[fakeip]]比[[realip]]解析速度更快，但实际感知差异小**
   - 为什么重要：解答了测速面板与实际浏览体验之间的落差。
   - 依据：通过[[sbox]]（即[[sinbox]]）演示，[[fakeip]]的[[DNS]]查询时间不到1毫秒（屏幕展示），而使用美国[[c12gia]]线路的[[realip]]消耗约160毫秒（屏幕展示/讲者原话）；但由于浏览器缓存和预解析机制（例如访问[[Twitter]]、[[百度]]等），除了首次打开新网站，日常使用无明显差异（讲者原话）。

## 概念关系
- [[全局模式]] ↔ [[DNS]]泄露：开启全局不代表无泄露，系统特性和应用层协议会绕过代理发起查询。
- [[Windows]]的[[智能多宿主名称解析]] → [[DNS]]泄露：导致系统通过物理网卡向国内服务器（如114）发起查询。
- [[quic]]协议 → [[DNS]]泄露：因其基于[[udp]]，导致[[Clash]]处理请求时强制发起真实[[DNS]]解析。
- [[fakeip]] ↔ [[realip]]：前者本地响应极快（<1毫秒）无需配置[[cdn]]优化，后者需节点转发延迟高（≥160毫秒）且需复杂的同节点策略配置。

## 详细笔记
### 1. [[DNS]]泄露的背景与两种定义标准
- **触发背景**：有[[安卓]]及[[iPhone]]用户收到访问特定网站（如[[哔哩哔哩]]）的运营商警告短信，大概率是通过网络层面的[[DNS]]泄露收集的。
- **争议焦点**：
  - **严格标准（跨境电商）**：只要未通过节点进行[[DNS]]请求就算泄露。使用[[IPS]]或[[wlg]]类传统[[VPN]]若不配路由表极易泄露。
  - **宽松标准（普通用户）**：只关心黑名单网站是否泄露，其余依赖[[DNS]]获取[[IP]]来进行精准[[分流]]，国内小众网站漏了无所谓。

### 2. 实战演示：[[全局模式]]下的泄露原因与解决步骤
- **误区验证**：使用[[Clash for Window]]启用[[TUN模式]]和[[全局模式]]，访问检测网站[[iplake]]，依然出现中国服务器。
- **泄露原因及解决**：
  - **原因1**：配置为rehost（[[redir-host]]）模式，必须返回真实[[IP]]。解决：切换至[[fakeip]]（fip）。
  - **原因2**：[[Windows]]系统的[[智能多宿主名称解析]]，会使用所有网卡发请求。解决：在组策略中禁用该功能。
  - **原因3**：浏览器的[[quic]]协议基于[[udp]]，[[Clash]]处理时必定发起[[DNS]]请求获取真[[IP]]。解决：在浏览器中关闭[[quic]]功能。

### 3. [[分流]]模式下的泄露与业务安全
- **掩耳盗铃的做法**：仅将检测网站[[iplake]]加入代理规则。此时访问未在规则中的业务网站（如[[PayPal]]），依然会发起国内[[DNS]]请求进行IP匹配，对[[PayPal]]而言已构成泄露。
- **高要求解决思路**：将业务相关域名（即使非黑名单）全加入规则；或采用白名单模式，国内网站直连，其余一律交由节点处理。

### 4. 普通用户的极简[[DNS]]配置建议
- **配置原则**：普通用户只需填入一个能正常提供服务的[[DNS]]（如路由器[[IP]]），或开启[[dhcp]]获取。
- **避坑指南**：
  - 不必迷信复杂的114或223，运营商默认的通常最快。
  - 弃用fback（fallback）：在仅用[[fakeip]]且分流规则完整时毫无必要。
  - 慎用[[doh]]：国内[[doh]]防不住提供商窥探；国外[[doh]]原版直连易被封锁。

### 5. [[fakeip]] 与 [[realip]] 的速度对比测试
- **测试环境**：使用[[sbox]]（支持这两者及[[透明代理]]，类似[[v2r]]等），同一条美国[[c12gia]]线路。
- **测速数据对比**：
  - **[[fakeip]]**：请求在本地流转，解析时间不到1毫秒（屏幕展示）。
  - **[[realip]]**：获取真实[[IP]]需向节点发起请求，消耗约160毫秒（屏幕展示）。若线路较差则会达到200-300毫秒（讲者原话）。
- **实际体验评估**：相差的100多毫秒在人类感知中差异不大。因存在[[DNS]]缓存和预解析，仅在初次打开新网页时有区别。且[[fakeip]]无需像[[realip]]那样去专门匹配节点以获得[[cdn]]优化。

2. **切换 fake-ip 模式并排查 Windows 网卡泄漏**
   - 将 DNS 配置切换为 `fake-ip` 模式，开启全局模式后再次抓包，发现 DNS 请求未发给上游，而是发给了本地物理网卡（如 192.168.2.1） [1, 4]。

3. **关闭浏览器 QUIC 协议**
   - 禁用多播名称解析后，再次访问 Google 并抓包，发现依然向 [[Clash]] 的上游发起了 DNS 请求 [4]。在浏览器中关闭 QUIC 功能，清空记录后重新测试，不再发起 DNS 请求 [4]。

4. **配置分流规则以防泄漏**
   - 将 [[Clash]] 切换到规则模式，将特定网站域名加入代理规则中 [2]。
   - 关键代码/命令：未明确提及

5. **精简普通用户的 DNS 配置**
   - 将 [[Clash]] 自带的复杂 DNS 全部移除，填入运营商分配的 DNS（通常填入路由器 IP）或直接使用 DHCP 获取 [6]。

6. **实测 fake-ip 与 real-ip 解析延迟对比**

## 常见坑与解决方案
- **问题描述**：在 Windows 中即便开启了全局模式和 `fake-ip`，仍会向本地网卡发起 DNS 请求导致泄漏。
  - 解决方案：在 Windows 的组策略中找到并启用“禁用智能多播名称解析” [4]。
- **问题描述**：浏览器使用全局模式 + `fake-ip` 访问 Google 时发生 UDP 泄漏。
  - 解决方案：在浏览器设置中关闭 [[QUIC]] 功能 [4]。
- **问题描述**：只把 DNS 泄漏检测网站（如 ipleak）加入分流规则，访问业务网站（如 Paypal）时由于缺乏域名规则，仍触发 IP 规则匹配导致 DNS 泄漏。
  - 解决方案：将 Paypal 等业务相关的域名一并加入规则列表；或采用白名单模式通用配置，将所有国内网站直连，其余全部丢给节点处理，不做任何 DNS 分流判断 [5]。
- **问题描述**：使用原版 [[Clash]] 配合国外的 [[DoH]] 发起 DNS 请求时，出现间歇性被墙和连接不畅。
  - 解决方案：普通用户直接填入运营商下发的 DNS 服务器（路由器 IP）或通过 DHCP 获取即可，无须强制使用国外 DoH [6]。
- **问题描述**：使用 `real-ip` 模式时，如果代理 DNS 请求的节点与访问网站的节点不一致，可能会导致 CDN 解析出非最优 IP，产生绕路并导致网速变慢。
  - 解决方案：用哪个节点代理网站，就用哪个节点代理 DNS 请求，以获取最佳 CDN 优化 [9]。

## 最佳实践
- **测试顺序**：只有当解决了全局模式下的 DNS 泄漏问题后，才有必要去研究分流模式下的泄漏。如果在全局模式下都没防住，分流模式怎么调都会漏 [2]。
- **Fallback 配置**：如果你正在使用 `fake-ip` 且分流规则比较完整，完全没必要配置 fallback（fallback 主要是为解决 DNS 污染，通常搭配旧版 `redir-host` 使用） [6]。
- **极简 DNS 策略**：黑名单与白名单模式均适用简单的 DNS 策略，绝大多数普通用户只需使用本地运营商提供的 DNS，不需要把 DNS 配置写得花里胡哨 [5, 6]。
- **反模式**：掩耳盗铃式的防泄漏配置——仅将 DNS 检测网站加入代理规则，而忽略了真正的业务网址，会被业务网站后台的检测机制判定为网络环境存在风险 [5]。
- **反模式**：国内环境盲目使用国内的加密 DNS（[[DoH]]），这只能防运营商，DoH 服务商依然能看到你的请求记录，反而不如不加密 [6]。

## 关键概念

### [[DNS泄露]]
- **定义**：指在科学上网时，本应通过代理节点进行的DNS查询却使用了节点外的DNS服务器，导致运营商得知用户访问目标的现象 [1, 2]。
- **视频上下文**：作者指出目前对此有分歧：一种认为只要非节点DNS查询就算泄露（如跨境电商所关注的），另一种认为仅黑名单网站发起明文请求才算泄露 [2, 3]。
- **为什么重要**：它是导致隐私暴露或网络环境被判定异常的核心安全问题 [1, 2, 4]。
- **关联概念**：未明确提及

### [[全局模式]]
- **定义**：一种代理软件的运行模式，常被误以为开启后所有流量强制走代理从而绝对不会发生DNS泄露 [5]。
- **视频上下文**：作者通过抓包演示证明，即便在全局模式下，由于redir-host配置、Windows智能多宿主解析或浏览器QUIC协议的影响，依然可能向国内DNS发起请求导致泄露 [5, 6]。
- **为什么重要**：打破了用户的常见安全误区，强调排查泄露问题必须以实际抓包为准，而非单纯依赖模式开关 [6, 7]。
- **关联概念**：对比 → [[分流规则]]

### [[fake-ip]]
- **定义**：一种透明代理DNS处理模式，在设备发起请求时直接返回伪装IP段（如198.18.x.x），无需获取真实IP [5, 8]。
- **视频上下文**：作者通过测速对比指出，该模式下DNS解析基本在本地瞬间完成（1毫秒内），不会像真实解析那样产生显著延迟 [8, 9]。
- **为什么重要**：这是目前主流且能有效提升首次网页加载速度的DNS代理模式，且通常不需要配置复杂的fallback参数 [8-10]。
- **关联概念**：对比 → [[real-ip]]

### [[real-ip]]
- **定义**：一种透明代理DNS处理模式，必须向远端发起真实DNS查询以获取目标网站的真实IP地址 [9]。
- **视频上下文**：作者演示指出，在此模式下DNS请求需要由远端节点代为解析并返回，会导致160毫秒以上的额外网络延迟 [8, 9]。
- **为什么重要**：虽然能拿到真实IP，但配置较复杂（建议由哪个节点代理网站就由该节点代理DNS以获CDN优化），且对线路质量要求极高 [8, 11]。
- **关联概念**：对比 → [[fake-ip]]

### [[分流规则]]
- **定义**：基于域名或IP地址判断流量是直连还是走代理节点的配置文件规则 [2, 7]。
- **视频上下文**：作者指出如果规则配置不当（例如IP规则插入顺序错误或遗漏特定域名），会触发系统使用内置或国内DNS发起请求，从而造成泄露 [4, 7]。
- **为什么重要**：它是决定非全局状态下DNS是否发生泄露的核心机制 [2, 7]。
- **关联概念**：属于 / 扩展 → [[黑名单模式]]

### [[黑名单模式]]
- **定义**：仅将特定的被墙网站加入规则走代理，其余未知网址通过DNS解析获取IP后再按归属地进行分流的代理策略 [7]。
- **视频上下文**：被作者描述为绝大多数普通用户的常规习惯，在此模式下，只要被代理的特定网站不发生泄露即可，其余网站泄露无妨 [2, 7]。
- **为什么重要**：代表了兼顾日常便利性与基础访问需求的常见代理理念 [2, 7]。
- **关联概念**：对比 → [[白名单模式]]

### [[白名单模式]]
- **定义**：将国内网站设为直连，其余所有未标明的网络请求一律交由代理节点处理，不进行任何本地DNS分流判断 [4]。
