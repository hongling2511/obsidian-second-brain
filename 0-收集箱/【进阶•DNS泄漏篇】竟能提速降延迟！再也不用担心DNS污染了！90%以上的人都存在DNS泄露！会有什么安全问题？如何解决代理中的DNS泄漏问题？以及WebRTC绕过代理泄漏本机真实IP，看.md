## 一句话总结
解析 [[DNS泄漏]] 与 [[WebRTC]] 泄漏的原理危害，并演示在各类代理客户端中避免泄漏的配置步骤。

## 目标受众与前置知识
- **适合谁看**：使用代理工具进行科学上网，关注隐私安全或因泄漏导致无法访问特定区域服务（如 [[奈非]]）的用户（推断）。
- **前置知识**：[[DNS]]、[[IP]]、[[代理]]、[[路由器]]、[[网关]]。

## 核心观点
1. **[[DNS泄漏]] 会暴露用户的真实网络访问意图**
   - 为什么重要：会导致运营商或防火墙知道你在访问 [[google]]、[[yubefacebook]]（推断为 youtube 与 facebook 的口误连读）等网站，或被 [[奈非]] 判定使用了 [[VPN]] 工具而禁止观看。
   - 依据：即使配置了国外的 DNS 提供商，只要本地发起了明文的 [[DNS请求]]，运营商或中间路由器就能看到访问目标（讲者原话）。
2. **[[DNS泄漏]] 的根本原因是代理状态下本地仍发起了解析请求**
   - 为什么重要：本地获取 IP 只是为了匹配分流规则，在合理配置下应直接将包含域名的数据加密发给节点服务器，由节点负责 DNS 解析。
   - 依据：使用代理访问目标网站时，电脑给互联网发送了请求目标网站 IP 地址的明文 [[DNS请求]]，测试网站正是通过记录该查询得出泄漏结果（讲者原话）。
3. **合理配置代理客户端规则可彻底解决 [[DNS泄漏]] 并提速**
   - 为什么重要：阻止本地发起请求不仅能解决泄漏，还能降低延迟，并解决 [[DNS污染]] 导致部分网站无法访问的情况。
   - 依据：通过配置 [[v2n]]、[[clash]]、[[v2ng]]、[[小火箭]] 的远端解析或 [[fakeip]]（即 [[费IP]]）模式，能够成功避免向物理网卡发送 DNS 请求（屏幕展示配置过程）。

## 概念关系
- [[泛解析]] → [[权威DNS服务器]]：测试网站利用随机域名的泛解析穿透缓存，迫使请求必须到达权威DNS服务器，以此来获取并记录用户使用的上游 DNS IP。
- [[系统代理]] ↔ [[ton模式]]：系统代理由客户端直接处理请求接管解析；而 [[ton模式]] 操作网络层，受 [[Windows]] 系统（被讲者戏称为 [[微然]]）的 [[智能多数组名称解析]] 影响，会向所有 [[物理网卡]] 发送请求，因此更容易造成泄漏，需额外修改 [[主策略]]（推断为组策略）。
- [[明文DNS请求]] → [[DNS泄漏]]：由于默认 [[DNS请求]] 未使用 [[dot]] 或 [[doh]] 等加密技术，任何中间节点均可截获分析。

## 详细笔记
### 现象与危害识别
讲者 [[布良]]（推断为频道 [[不良林]] 的自称）建议访问特定检测网站。如果在 [[全局模式]] 和 [[分流模式]] 下检测到的 DNS 提供商不同，或列表中出现国内 DNS，则说明发生 [[DNS泄漏]]。泄漏会导致运营商知晓用户在科学上网，并且部分对地区敏感的服务（如 [[奈非]]）会通过 DNS 归属地与代理 IP 不一致来判定使用了代理工具。

### 常规家庭网络中的 DNS 工作流程
在无代理环境中，[[路由器]] 通过 [[pppo]] 获取公网 [[IP]]（例：2.2.2.2，讲者原话）和上游 [[DNS]] 服务器（例：3.3.3.3，讲者原话），并通过 [[dhcp]] 给家庭设备分配 [[内网IP]]（例：192.168.0.1，讲者原话）。访问 [[百度]]（即 [[百度.com]]）时，首先检查 [[浏览器DNS缓存]] 和 [[操作系统DNS缓存]]（可通过命令行 [[ipcfigure display DNS]] 检查）。若无缓存，则通过 [[udp]] 协议将目标端口设为 53（讲者原话），封包 [[mark地址]] 发送给路由器。路由器再通过运营商的 [[负载均衡]] 发送给上游服务器，获取并缓存对应记录（如 [[ttl值]] 30秒，讲者原话）。

### 网站检测 DNS 泄漏的原理
DNS 泄漏专指使用代理时的情况。检测网站会不断向浏览器发送完全随机的域名以避开各级缓存。此时请求必定通过运营商的 [[DNS]] 转发至该域名的 [[权威DNS服务器]]。因为设置了 [[泛解析]]，权威服务器在返回 IP 的同时，会暗中记录发起查询的 [[DNS]] 服务器 IP 并返回给检测网站展示。国内用户即使使用 [[任播DNS]]（如 1.1.1.1、8.8.8.8），因反向优化实则连接的是美国服务器，查出美国 IP 仍意味着已发生请求泄漏。

### 代理环境产生泄漏的成因
以 [[系统代理]] 为例，浏览器不再直接查 DNS，而是将请求交给代理客户端。如果客户端（如 [[clash]]）需要根据 IP 规则分流，就会去发起明文的 [[DNS请求]]。除非使用 [[dot]]、[[doh]] 加密，或强制通过节点进行 [[远程DNS]]（两者都会增加延迟），否则请求意图将被彻底暴露。解决之道在于：通过调整分流规则，让本地完全不发送 DNS 请求，直接将带域名的数据加密交给节点服务器处理。

### 客户端防泄漏实操演示
- **[[v2n]]**：使用“绕过大陆”路由，进入路由设置将 [[域名策略]] 改为 S。
- **[[clash]] 及其衍生版（如 [[clash for Window]]）**：由于分流写在订阅中，需借助 [[订阅转换]] 注入专用的防泄漏远程配置文件。模式上需使用 [[fakeip]]（讲者口述为 [[费IP]]），不要用 [[re host模式]]。
- **针对 [[ton模式]] 的额外设置**：因 [[微然]]（推断为微软）系统的 [[智能多数组名称解析]] 会绕过 [[虚拟网卡]] 向 [[物理网卡]] 发送请求，需在操作系统的 [[主策略]] 中将其禁用。
- **[[v2ng]]（安卓端）**：启用本地DNS，启用虚拟DNS（即 [[fakeip]]），[[域名策略]] 改为 S，设置绕过局域网及大陆。
- **[[小火箭]]（iOS端）**：默认配置已能避免泄漏但分流不完善，导入讲者提供的配置并选为“配置模式”即可。
全部配置后重新检测，若全局与分流模式测出同一国家DNS（节点的上游DNS），即为无泄漏。

### 解决 WebRTC 泄漏
除了 DNS，浏览器本身的 [[wbatc]] / [[wbrtc]]（即 [[WebRTC]]）功能会直接绕过代理泄露本机真实 IP。解决方式是安装专属浏览器插件以禁用该功能。

## 最佳实践
- **推荐做法**：代理系统所有流量时，推荐使用 `[[Clash]]` 的 `[[TUN模式]]` [1]。
- **推荐做法**：使用 `[[Clash]]` 时，内置 DNS 应保持使用默认的 `[[fakeip]]` 模式 [4]。
- **推荐做法**：通过合理的分流配置避免本地在代理情况下直接发送 DNS 请求获取 IP，直接将数据加密发送给节点服务器由其代为解析，既能防泄漏还能降低延迟 [3, 12]。
- **推荐做法**：验证是否彻底解决泄漏的标准是，全局模式与分流模式下检测出相同的 DNS 提供商（且为节点配置的上游 DNS），则大概率未泄漏 [7]。
- **反模式**：直接把 DNS 泄漏检测的网站添加到分流规则中走代理，这是一种“掩耳盗铃”的做法，实际上并未修复底层泄漏问题 [11]。
- **反模式**：为了防止明文 DNS 被劫持而盲目使用 `[[dot]]`、`[[doh]]` 加密，或者默认将所有 DNS 请求强制通过节点加密进行远程查询，这些方法都会增加延迟，不建议大部分用户作为首选方案 [2]。
- **反模式**：使用 `[[Clash]]` 时修改或使用已经被弃用的 `[[redir-host]]` 模式，会引发泄露风险 [4]。

## 关键概念

### [[DNS 泄漏]]
- **定义**：在使用代理访问目标网站时，本地电脑向互联网发送了请求目标网站 IP 地址的明文 DNS 请求[1]。
- **视频上下文**：作者指出即使使用了代理，如果不正确配置导致该问题发生，用户的访问意图就会暴露给运营商或防火墙，也可能被奈飞等应用判定使用了 VPN 工具而遭封禁[2], [3]。
- **为什么重要**：它是代理网络中常被忽视的安全隐患，会直接暴露用户的真实物理位置和浏览行为[2], [1]。
- **关联概念**：属于 / 依赖 / 对比 / 扩展 → [[Fake-IP 模式]]

### [[TUN 模式]]
- **定义**：一种在网络层运作的系统代理模式，能够接管并代理系统的所有流量[4], [5]。
- **视频上下文**：作者推荐使用 Clash 的该模式来代理系统流量，但也指出由于它操作网络层，处理 DNS 会更复杂，Windows 系统默认会将请求发给所有网卡，如果不禁用智能多宿主名称解析依然会导致泄露[4], [5]。
- **为什么重要**：它是目前比较完善的全局代理解决方案，但必须配合底层系统策略设置才能彻底杜绝 DNS 泄露[4], [5]。
- **关联概念**：未明确提及

### [[DNS 缓存]]
- **定义**：浏览器、操作系统以及各级路由器和 DNS 服务器临时存储“域名与对应 IP 地址”映射结果的列表[6], [7]。
- **视频上下文**：作者在讲解 DNS 工作流程时说明，系统在发送查询数据包前会逐级检查该列表；泄漏检测网站正是通过随机域名绕过它来强制触发全新查询的[6], [1]。
- **为什么重要**：它决定了设备是否需要向外部发送新的 DNS 查寻请求，是理解 DNS 工作机制及检测原理的前置基础[6], [1]。
- **关联概念**：依赖 → [[TTL]]

### [[TTL]]
- **定义**：全称 Time to Life（或 Time to Live），在 DNS 中表示记录的缓存有效时间[8]。
