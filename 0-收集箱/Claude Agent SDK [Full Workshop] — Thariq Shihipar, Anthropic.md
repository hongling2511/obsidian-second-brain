## 一句话总结
Thariq 讲解了基于 [[Claude Code]] 构建的 [[Claude Agent SDK]]，并展示如何通过 [[Bash]] 和 [[codegen]] 构建强大且灵活的智能体工作流。

## 目标受众与前置知识
- **适合谁看**：AI 工程师、产品经理及致力于构建复杂自动化系统的开发者（推断）。
- **前置知识**：[[LLM]]、[[Agent]]、[[Bash]]、[[TypeScript]]、[[Claude Code]]

## 核心观点
1. **[[Bash]] 是最强大的智能体工具。**
   - 为什么重要：它提供了极高的动作组合性，能大幅降低上下文占用，并允许智能体通过命令行完成几乎所有底层交互。
   - 依据：“Bash is what makes code so good”（讲者原话），它让智能体可以直接调用现成软件（如 [[ffmpeg]]、[[jq]]、[[LibreOffice]]）处理复杂任务，而不必人为构建成百上千个单一工具。
2. **优秀的智能体设计必须具备强大的验证机制（Verifying work）。**
   - 为什么重要：智能体操作越容易被规则化、确定性地验证（如通过编译器或静态检查），其越能向更通用的方向演进。
   - 依据：对于代码生成任务可以进行 lint 或编译检查，而如果在 [[DoorDash]]（原文 door-ash.com）下单买错了饮料，由于状态机的复杂性导致不可轻易回滚，Agent 表现会很差（讲者原话举例）。
3. **比起纯粹的预定义工具，更推荐“渐进式上下文披露”模式。**
   - 为什么重要：过多的预定义 API 会使模型困惑，让智能体在需要时按需编写脚本、查阅本地文件会更高效。
   - 依据：提供“50到100个工具”会大量消耗上下文导致模型困惑（讲者原话），应利用基于文件系统的 [[skills]] 机制让代理动态 CD 到目录下读取执行指南。

## 概念关系
- [[Claude Code]] → [[Claude Agent SDK]]：前者是后者的底层应用，后者将构建前者过程中提炼出的工具、文件系统框架、[[skills]] 抽象成供外部调用的库（推断与明确合并）。
- 传统 [[tool use]] ↔ [[Bash]] / [[codegen]]：对比关系。前者是原子化操作，高度结构化但消耗大且不可组合；后者动态且高度组合，虽然调用耗时更长，但泛化能力强。
- [[sub agents]] → 上下文管理：子智能体通过在独立沙盒或并行进程中执行特定任务（如同时读取多个表单），防止复杂的中间思考过程污染主智能体的上下文。

## 详细笔记
### 1. 简介与技术演进背景
- 讲者 Thariq Shihipar 来自 Anthropic，指出 AI 已经从单纯的单一 [[LLM]] 特征（如 [[GPT-3]] 的文本分类），发展到了带有流程的交互（如结合 [[RAG]] 提供代码补全），最终走向高自治度的真正 [[Agent]]（如基于不受限文本交互的 [[Claude Code]]）。
- 内部为了不再重复造轮子，将其封装为了 [[Claude Agent SDK]]，包含：模型层、Harness、工具提示、文件系统操作，以及近期推出的 [[skills]] 和 [[sub agents]]。

### 2. SDK 架构理念与安全策略
- 用户正在基于 SDK 开发诸如软件可靠性诊断、网站与面板构建（推断包含了后文提到的 [[Lovable]] 工具）、以及利用 [[GitHub]] 与 [[Slack]] 自动化工作流等产品。当运行涉及测试环境时，通常需要在 [[Docker]] 中跑代码。
- 强调使用 [[Unix primitives]] 构建。
- **安全体系（瑞士奶酪防御）**：防御分为三层。底层是模型级对齐（讲者提到了一篇关于 Reward Hacking 的内部论文）；中间层是 Harness 权限管控（如研发了复杂的 [[Bash]] 语法解析器以阻断违规指令）；顶层是网络与运行沙盒隔离。如果不做沙盒，讲者提到可能会遭到“致命三因素”攻击：执行代码、修改文件系统、向外层渗出数据（讲者原话意译）。讲者推荐使用 [[Cloudflare]]、[[Modal]]、[[E2B]]、[[Daytona]] 等提供商的安全容器。

### 3. Bash 是一切需要的核心
- 讲者认为 [[Bash]] 是最初的“代码模式”并且主张全盘采用（被观众称为 Yolo mode）。不采用为每种操作单独写代码的套路，而是让智能体通过 [[grep]] 搜索、或者在无包管理器时利用 [[npm]] 或安装 [[eslint]] 解决代码问题。如果遇到 [[Gmail]]，不发给智能体 100 封邮件，而是写个搜索 [[Uber]] 或 [[Lyft]] 的脚本在后台过滤后汇集金额。
- 观众问及如何处理类似 [[playwright]] 的复杂 CLI，讲者指出虽然会增加前期潜伏期（如代理需要输入 `--help` 指令去了解功能），但可明显省去大批上下文占用。

### 4. 智能体主循环的三大步骤
- **Gather context（收集上下文）**：是极具创造性的过程。
- **Taking action（采取行动）**：通过执行变更与调用外部服务完成。
- **Verifying work（验证工作）**：应在任意阶段尽早介入确定性逻辑检查。在对抗性测试时，讲者比喻可建立一个被赋予低 GPA、毕业于非名校的初级 [[McKinsey]]（讲者发音为 McKenzie）分析师设定的 [[sub agents]] 去故意挑刺代理输出的结果（讲者原话）。

### 5. 挑战：处理大型数据集与搜索设计
- 现场通过 [[Figma]] 进行头脑风暴探讨如何让智能体在极大电子表格中查找“2026年收入”。
- 不应将 100,000 列或几百万行（讲者原话）推入上下文。提议利用 [[CSV]]/[[TSV]] 结合 [[AWK]] 解析，或直接调用 [[Google]] APIs 处理计算。
- 高级解法：将数据预处理为分布内（in-distribution）格式，例如通过智能体转换为 [[SQLite]] 并利用 SQL 查询，或者将其视作 [[XML]]（甚至底层如 XLX 文件也是基于 XML的）进行定位。
- 对于过大的代码库也是同样道理。不依赖死板的语义搜索，而是结合提示与文件挂载进行导航。在上下文缩减方面，推荐直接清除并让模型查看 [[Git]] diff 以知晓当前变更（讲者原话操作偏好）。

### 6. 上下文与工具接口策略探讨
- 通过一个“被锁在房间 6 个月拿 500,000 美元奖励”的 Mr. Beast 节目挑战做类比（讲者原话），强调应该给智能体提供探索工具（像电脑），而不是给它一堆印出来的纸质文档（被动堆砌上下文）。
- 有观众提问为何不依赖外部现成 API/[[MCP]]，讲者指出如果数据库需严格防越权，可以用传统工具绑定 RBAC；但更优雅的方式是赋权结合临时 API 凭证，加上实时提示与拦截反馈供模型自动试错。
- 在讨论工具框架更迭时，讲者用前端生态打比方：[[Facebook]] 推出包含 [[JSX]] 并需要打包工具的 [[React]] 曾显得十分烦人且取代了轻量的 [[jQuery]] 和 [[Backbone]]，但它提升了上限；SDK 也是属于这类为了解决硬核工程问题而生、伴有阵痛的基础设施。
- 在关于 [[skills]] 机制探讨上，讲者提及了此前 Barry 和 M 两位同事的演讲，强调短期内使用包含 `skill.md` 与 `claude.md`（或 `claud.md`） 等技能文件，对于传递前沿前端专家的 Prompt 必不可少。独立产品 [[Claude.ai]] 中现在也嵌入了文件系统、[[DOCX]] 生成和 [[memory tool]] 功能。

### 7. 原型编码演示（基于 Pokemon API）
- **初期设置**：通过编写少于 50 行（讲者原话）的薄壳脚本进行配置。为了避开编译流程，在后端采用了 [[bun]] 而不是原始的 [[Node.js]]，结合 [[TypeScript]] 方便智能体理解强类型，同时指出如果缺少指引，模型有时会退化输出 [[JavaScript]]。
- **动态生成任务**：让模型调用现成互联网接口 [[Poke API]] 找寻第二世代（Gen 2）的水系宝可梦。屏幕展示它自己写代码跑完了 200 多个实例并找到了约 20 个水系宝可梦（如 Toadile、Crocenoff/Croconaw）。
- **游戏态的模拟拓展**：讲者提到开发者可以用一个通过内存抓取与控制操作状态的 GBA 模拟器，去让智能体通关 [[Pokemon Red]]。并指出 Adam Wolf 曾在 QCon 上展示过对于处理复杂沙盒中多进程竞态条件的精湛解法。
- **实战重构**：讲者加载了 [[Smogon]] 的游戏战术分析纯文本库，要求 Agent 围绕“Venusaur（妙蛙花）”构建队伍，Agent 自行发散思维寻找关联队友、挖掘计数器关系并在沙盒中结合 HTTP 服务器生成报告表格。

## 步骤拆解
1. **生成基础 API 封装库**
   - 编写 Prompt 让模型查阅目标 API 文档（如 Poke API），并利用代码生成（Codegen）构建出 [[TypeScript]] 接口库。同时通过 `claud.md` 提供模块化定义与示例文本，引导 Agent 后续调用。
   - 关键代码/命令：`import Poke API from the Poke API SDK`
   - 注意事项：建议为 Agent 设计专用的 CLI 脚本或接口，并包含 `--help` 指令，以便 Agent 进行渐进式功能发现。

2. **构建纯工具调用的 Agent 循环（对比基准）**
   - 使用传统的 Messages/Completion API 注入多个特定的原子化工具（如获取种类、获取能力等），构建对话脚本来获取数据，作为与 Agent SDK 对比的基准线。
   - 关键代码/命令：`bun chat.ts`
   - 注意事项：使用 [[Bun]] 执行以跳过编译步骤降低复杂度，但生成代码建议使用带有静态类型的 [[TypeScript]]；必须在终端或日志中显式打印所有工具调用过程以便调试。

3. **使用 Claude Code 进行快速原型验证**
   - 在本地打开 [[Claude Code]]，注入 API 密钥与基础脚手架文件，通过自然语言对话让 Agent 自由生成检索脚本、处理非结构化数据（如纯文本分析）。
   - 关键代码/命令：未明确提及
   - 注意事项：在此阶段重点观察 Agent 是如何搜索上下文、如何编写 `grep` 或 API 脚本处理任务的，通过错误日志迭代其逻辑。

4. **Agent SDK 生产化部署**
   - 将验证成功的逻辑封装至约 50 行的 `agent.ts` 中，指定工作目录和可用的 Bash/文件系统权限，最后将代码放入沙盒环境中启动。
   - 关键代码/命令：`sandbox.start` 与 `bun agent.ts`
   - 注意事项：若构建包含前端 UI 的 Agent，可在沙盒中启动 Dev Server 暴露端口，让 Agent 通过修改代码来实现 UI 的热更新交互。

## 常见坑与解决方案
- **问题描述**：使用 [[bash]] 工具时产生长文本输出，导致上下文污染与长度爆炸。
  - 解决方案：将工具的执行结果写入文件系统的特定文件（如内存目录），并使工具仅返回该文件的路径。
- **问题描述**：向 Agent 开放 [[bash]] 导致严重的安全风险（可执行代码、修改文件、网络数据外发）。
  - 解决方案：采用“瑞士奶酪防御”（Swiss cheese defense），结合底层模型对齐限制、Harness 层的 [[bash]] 解析器拦截，并必须部署在具有网络与系统隔离的第三方沙盒中。
- **问题描述**：Agent 尝试直接往尚未读取的文件中写入内容，破坏文件状态。
  - 解决方案：在验证步骤中增加确定性校验拦截，如果检测到未在读取缓存中的文件被写入，则向模型抛出错误反馈：“你尚未读取此文件，请先读取”。
- **问题描述**：对于其预训练数据中已有的知识（In-distribution），Agent 会产生幻觉直接给出结果，而不去执行脚本读取最新数据。
  - 解决方案：使用 Hooks 机制监听事件，当 Agent 未执行规定脚本即返回结果时触发反馈，强制要求其编写并执行查阅数据的脚本。
- **问题描述**：处理极大规模的外部数据（如百万级电子表格）时发生上下文过载。
  - 解决方案：不要将完整文件读入，需将底层数据转换为可查询的接口（如用 [[SQLite]] 查询 [[CSV]]），并使用多个 Sub-agents 并行处理不同分块，主 Agent 仅需接收摘要并维护“草稿本”（Scratchpad）。

## 最佳实践
- **推荐做法**：
  - 将 [[bash]] 视为最强大的 Agent 核心工具，结合文件系统作为 Agent 专属的动态内存与上下文工程载体。
  - 针对不可逆的、具备强破坏性的操作（如发邮件、修改生产数据库）使用严格定义的原子化 Tools；针对高动态性、需组合 API 的分析任务使用代码生成（Codegen）。
  - 高频清空上下文（Compact/Clear Context），让模型基于代码库现有的 Git Diff 或文件系统的实时状态进行增量推理，而非依赖长对话历史。
  - 大量使用 Sub-agents 隔离上下文，非常适合用于处理检索任务或作为对抗性角色独立检查主 Agent 的代码逻辑。
  - 构建具有极强确定性的“工作验证”（Verify Work）规则（如 Linting 校验、空指针扫描）。
- **反模式**：
  - 将 [[bash]] 工具托管在隔离的虚拟化容器中，导致其无法与 Agent 循环的其他部分（如文件写入缓存）交互通讯。
  - 一次性向大模型注入大量（如 50-100 个）传统 Tools，这会导致严重的上下文消耗与意图识别混乱。

## 关键概念

### [[Claude Agent SDK]]
- **定义**：Anthropic 构建的一个用于开发智能体（Agent）的工具包，它封装了模型、工具、提示词系统和文件系统等核心组件 [1-4]。
- **视频上下文**：讲者提到它是基于 Claude Code 构建的，目的是将 Anthropic 内部构建非编码智能体时反复使用的核心部件和最佳实践标准化并打包提供给开发者 [1, 4-6]。
- **为什么重要**：它为开发者提供了一个包含了丰富的上下文工程能力（如 Bash 工具、文件系统）的标准化且带有强烈最佳实践导向的框架 [4, 7-9]。
- **关联概念**：扩展 → [[Agent loop]]

### [[Agent]]
- **定义**：能够自主构建上下文、决定执行轨迹并自主执行广泛非结构化操作的 AI 实体 [10, 11]。
- **视频上下文**：讲者将其与单次 LLM 调用（GPT-3时代）和高度结构化的工作流对比，指出 Agent 适用于不限制其执行边界，并通过自然语言进行灵活交互的场景 [10-14]。
- **为什么重要**：代表了当前 AI 能力演进的前沿模式，其高度的自主性使得它能够解决复杂且长线的任务 [1, 10, 13]。
- **关联概念**：对比 ↔ [[Workflow]]

### [[Workflow]]
- **定义**：具有严格定义的输入和输出、针对重复性流程的确定性执行模式 [13, 14]。
- **视频上下文**：讲者在回应提问时举例，如 GitHub PR 的代码审查，指出在使用 Agent SDK 时也可以构建利用结构化输出（Structured outputs）的稳定工作流 [14, 15]。
- **为什么重要**：它代表了解决业务中高度重复和确定性任务的最佳方式，与需要自主探索的 Agent 形成互补 [14, 16, 17]。
- **关联概念**：对比 ↔ [[Agent]]

### [[Bash tool]]
- **定义**：一种高度可组合、低上下文占用量的编程工具，允许 Agent 动态生成脚本、调用软件并与文件系统交互 [18-21]。
- **视频上下文**：讲者反复强调“Bash is all you need”，指出这是赋予 Claude Code 强大能力的核心机制，避免了为每个单一操作硬编码大量独立工具 [18-20, 22, 23]。
