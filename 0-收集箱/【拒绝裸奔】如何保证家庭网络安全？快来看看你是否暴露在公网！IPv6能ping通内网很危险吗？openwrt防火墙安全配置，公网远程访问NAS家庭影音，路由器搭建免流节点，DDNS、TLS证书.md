## 一句话总结
讲解如何配置 [[openwrt]] [[防火墙]]保障 [[IPv6]] 网络安全，并实现安全的公网远程访问与 [[节点]] 搭建。

## 目标受众与前置知识
- **适合谁看**：拥有公网 [[IPv6]] 且希望通过路由器进行远程网络管理、内网穿透或服务搭建的家庭网络用户。（推断）
- **前置知识**：[[IPv6]] 基础、[[openwrt]]（视频中称 openwr/opwit）基础、[[防火墙]]区域划分（[[L区域]]/[[万区域]]）、[[端口转发]]。

## 核心观点
1. **开启 [[IPv6]] 并不会让内网设备“裸奔”**
   - 为什么重要：消除用户对公网 [[IP]] 带来安全隐患的恐惧心理。
   - 依据：讲者原话指出，[[openwrt]] 默认 [[防火墙]] 禁止互联网络流量进入家庭网络，公网能 ping 通仅仅是因为默认优先放行了 [[icmpv]]/[[smpv6]] 的例外规则，并不构成威胁。
2. **异地远程访问需配合 [[端口转发]] 或指定放行规则**
   - 为什么重要：解决运营商封禁常用端口（讲者原话提到“大部地区80和443端口被封禁”）的问题，并避免直接全开防火墙导致被黑客攻击。
   - 依据：讲者原话演示将外部“2万”（20000）端口通过 [[dnet]]（即 DNAT）转交到路由器内部 80 端口，从而实现外网访问后台。
3. **暴露在公网的后台及代理服务必须配置加密与强密码鉴权**
   - 为什么重要：[[http]] 是明文传输，极易泄露密码；无密码暴露代理端口容易被盗用网络。
   - 依据：讲者原话演示使用 [[acm]]（即 acme.sh 插件）申请 CA 机构颁发的证书开启 [[HTTPS]] 防止 [[中间人攻击]]，并强调暴露 [[open clash]] 等工具必须设置复杂密码。

## 概念关系
- [[万区域]] ↔ [[L区域]]：[[万区域]]（即 WAN 区域）连接光猫面对整个互联网，[[L区域]]（即 LAN 区域）连接局域网设备。默认情况下 [[L区域]] 允许转发到 [[万区域]]，但 [[万区域]] 拒绝转发到任何区域。
- [[DDNS]] → 动态 [[IPv6]] 前缀：由于运营商定期更换 [[IPv6]] 前缀（讲者原话），必须依赖 [[DDNS]] 工具（如 [[ddnsg]]）定期检测并将新 IP 绑定到固定域名，以保证异地访问不中断。
- [[反掩码]] → 动态 [[IP]] 匹配：利用 [[IPv6]] 后 64 位（接口 ID）固定不变的特性，配合 [[反掩码]] 规则（讲者原话演示配置），可实现在前缀变化时依然精准匹配并放行目标内网设备流量。
- [[dnet]] → [[端口转发]] 放行：端口转发实际上就是做 [[dnet]]（目标地址转换），防⽕墙内部通信规则包含“做过 dnet 就接受数据包”的例外，因此转发后能够被外网访问。

## 详细笔记
### 防火墙基础概念与 IPv6 裸奔疑虑解答
- 核心原理解释：[[openwrt]] 默认创建 [[L区域]] 和 [[万区域]]。默认配置下，[[万区域]] 也就是互联网禁止转发到 [[L区域]]。入站/出站是相对路由器本身而言。
- 为什么能 ping 通：虽然主界面禁止了转发，但 [[openwrt]] 自带例外规则（优先级更高），如果收到的是 [[icmpv]]（包含 ping 规则）数据包，不管目标哪个区域都会接受转发。
- 结论：如果不擅自删改默认配置，即使设备有公网 [[IPv6]] 且能被 ping 通，也是安全的，不用担心裸奔。
- 工具扩展：深入研究防火墙可查阅 [[iptables]]、[[nft tables]] 和 [[netfeldter]]。（讲者原话提及）

### 异地远程登录路由器与端口转发
- 前置条件：必须使用主路由配置 [[IPv6]]。不建议用 [[旁路由]]，因为依赖主路由型号且部分品牌（如小米）防火墙屏蔽了入站流量。（讲者原话）
- 步骤：在防火墙“端口转发”页面，将外部自定义端口（如 6 万以内，演示用 20000）转发到目标区域为“未指定”的内部端口 80，协议勾选 [[tcp]]。
- 注意事项：内网设备（[[L区域]]）无法直接用上述外网端口转发规则访问，必须额外添加一条 [[L区域]] 的端口转发规则，或通过 [[NAT反射]] 解决 [[非对称路由]] 导致的问题。（讲者原话）

### DDNS 配置与子域名隐私保护
- 痛点：[[IPv6]] 前缀非固定。
- 解决步骤：安装 [[ddnsg]] 插件，使用 [[cloudfare]] 的 API 令牌（Token）绑定域名，获取 IP 方式选择万口。
- 隐私策略：主动做 [[DDNS]] 相当于主动暴露 IP，建议设置多级子域名（如 openwrt.1.2.3.4.xxx）缓解被扫描工具发现的概率。

### 部署 TLS 证书实现 HTTPS 访问
- 痛点：[[http]] 协议明文传输，密码经公网传播极不安全。
- 方案A（自签证书）：安装 [[uhttp低]]（即 uhttpd）的 tls 插件和 [[op ssl]]（即 openssl），生成密钥长度为 1024（讲者原话演示数据）的密钥。缺点是浏览器报错且无法防范 [[中间人攻击]]。
- 方案B（CA机构证书）：
  - 安装 [[acm]] 插件和 [[dnsapi]] 插件。
  - 使用 [[cloudfare]] 的 Zone ID（获取自域名概述页面）和 Token 添加 API 凭证。
  - 等待后台申请成功后，在 uhttpd 插件中手动修改证书和私钥（key）的路径（位于 /etc/acme 目录下）。
  - 若报错可通过 ssh 登录使用相关指令排错。最后将外网访问端口转发至 443 端⼝即可。

### 远程访问局域网 NAS 服务
- 防火墙设置：无需做端口转发，只需在“通信规则”中添加放行规则。
- 规则配置步骤：原区域选 [[万区域]]，目标区域选 [[Windows]] 电脑所在 [[L区域]]，使用 [[反掩码]] 技术填写设备固定不变的后 64 位接口 ID，目标端口填入 [[NAS]] 监听的 9000（讲者原话演示数据）端口，动作设为“接受”。
- 前置调整：需要统一交给路由器防火墙管理，直接关闭 [[Windows]] 电脑本身的防火墙。
- 动态域名：给 NAS 单独添加一个 [[DDNS]] 配置，需通过指令脚本正确获取 NAS 所在区域的 [[IPv6]] 前缀。也可以用类似 Nginx 进行反代（推断：视频中说ang反待）。

### 搭建免流节点与代理服务端配置
- 服务端搭建：使用 [[home proxy]] 搭建服务端，类型选择 [[wm]]（vmess），传输层选择可制定 host 的协议如 [[wscket]]（websocket），监听未被封禁的端口（某些地区免流需 80 端口），勾选“自动配置防火墙”。
- 混淆与免流：在客户端填入对应 host，可开全局实现 [[免流]]。
- Clash服务端：[[open clash]] 也可作为服务端提供 [[http]] 和 [[socks]] 代理（演示使用 7893 混合端口），必须在复写设置中添加认证信息（强密码）。
- 安全警告：强烈建议不放行 [[udp]] 端口，因为 clash 的 socks UDP 存在绕过密码认证的漏洞（[[x瑞]] 即 xray 已修复此 bug）。针对部分自带 socks 代理支持的软件（如 [[tj]]，推断为特定软件客户端），可直接填入代理使用。

### 防火墙 NAT 分载功能
- 概念：[[NAT分载]] 让已建立连接的转发数据包不再经过防火墙的各个 Hook 节点。
- 优势与冲突：开启硬件流量分载（若设备支持）可大幅降低 CPU 使用率提升转发性能。但会跳过控制节点导致流量控制工具（如 [[openfw]]）失效，需按需开启。

## 步骤拆解

1. **配置远程管理路由器的端口转发**
   - 具体操作描述：由于运营商通常封禁 80/443 端口，需要在防火墙“端口转发”中添加规则。将外网（WAN）的自定义高端口（如 20000）转发至未指定目标区域的内部 80 端口，并限定为 TCP 协议 [2, 12]。
   - 关键代码/命令：未明确提及。
   - 注意事项：配置完成后，通过公网访问需在地址后加冒号和外部端口号（如 `[IPv6地址]:20000`）。若要在内网通过此外部端口访问，需额外添加一条将源区域设为 LAN 的同类转发规则 [12, 13]。

2. **配置 DDNS（动态域名解析）**
   - 具体操作描述：安装并启用 DDNS 插件，选择服务商（如 [[Cloudflare]]），填入从服务商后台获取的 API Token。关闭 IPv4 选项并启用 [[IPv6]]，将 IP 获取来源设置为 WAN 口，并填入目标域名 [3, 4]。
   - 关键代码/命令：未明确提及。
   - 注意事项：API Token 等同于密码，切勿泄露 [4]。

3. **配置 TLS 证书实现 HTTPS 访问**
   - 具体操作描述：安装 [[uHTTPd]] 和 [[openssl]] 插件开启基础加密；进一步安装 [[ACME]] 和 [[dnsapi]] 插件以申请正规 CA 证书。在 ACME 插件中通过 DNS 验证模式（填入 Zone ID 和 Token）申请证书。待后台申请完成后，进入 [[uHTTPd]] 修改证书（crt/cer）与私钥（key）的路径，最后将之前的端口转发规则的目标内部端口从 80 改为 443 [5, 6]。
   - 关键代码/命令：`/etc/init.d/acme restart` （用于在 SSH 中重新运行证书申请以排错） [14]。
   - 注意事项：在 [[uHTTPd]] 修改证书和私钥路径时切勿选错文件，否则会导致服务无法启动，无法进入 Web 管理页面 [6]。

4. **利用反掩码放行内网设备（如 NAS）端口**
   - 具体操作描述：在防火墙“通信规则”中添加放行规则。源区域选 WAN，目标区域选 LAN。为应对 [[IPv6]] 前缀动态变化，利用设备的后 64 位接口 ID 结合掩码进行匹配，并设置目标端口（如 9000），将动作设为“接受” [7, 14]。
   - 关键代码/命令：在目标 IP 处输入 `::设备接口ID/掩码位`（反掩码格式） [7]。
   - 注意事项：内网目标设备（如 Windows 电脑）需要关闭自身防火墙，或在其系统防火墙中开放对应端口 [14, 15]。

5. **搭建内网免流与代理节点**
   - 具体操作描述：使用 [[HomeProxy]] 或 [[OpenClash]] 搭建服务端。在 [[HomeProxy]] 中设置监听端口、传输层协议（如 WebSocket 搭配 Host 制空以实现免流），并勾选自动配置防火墙。在 [[OpenClash]] 中利用混合端口提供 HTTP 和 Socks 代理，并在防火墙单独放行该端口 [8, 9]。
   - 关键代码/命令：未明确提及。
   - 注意事项：若要在公网使用此节点，必须设置复杂的鉴权密码 [9]。

## 常见坑与解决方案

- **问题描述**：开启 [[IPv6]] 后，公网可以直接 ping 通内网设备，引发安全担忧 [1, 10]。
  - 解决方案：无需干预。[[OpenWrt]] 默认防火墙策略拦截了 WAN 到 LAN 的转发，能 ping 通是因为防火墙内置了优先级更高的 ICMPv6 例外规则，设备实际上依然受防火墙保护 [11, 16]。
- **问题描述**：旁路由模式下配置 [[IPv6]] 遇到困难，部分主路由防火墙屏蔽且无法关闭 [2]。
  - 解决方案：放弃旁路由，建议直接使用主路由方式配置 [[IPv6]] [2]。
- **问题描述**：IPv4 端口转发导致内网设备无法通过公网 IP 访问路由器（非对称路由问题） [13]。
  - 解决方案：通过 NAT 反射解决（具体配置未明确提及） [13]。
- **问题描述**：[[ACME]] 申请正规证书长时间无文件生成或出错 [6, 14]。
  - 解决方案：登录 [[SSH]]，输入重置指令查看错误日志并根据提示排错 [14]。
- **问题描述**：开启 NAT 硬件/软件流量分载（Offload）后，部分流量控制类工具（如 QoS）失效 [17]。
  - 解决方案：因为分载会跳过防火墙的 netfilter hook 节点，若需使用流控工具则不应开启此功能，需按需启用 [17]。

## 最佳实践

- **保留默认防火墙配置**：不建议不熟悉防火墙规则的用户删改默认配置项，极易导致路由器异常或让内网直接裸奔 [16]。
- **防范域名扫描**：做 [[DDNS]] 时，建议设置多级子域名（如 `1.2.3.4.xxx.xyz`）来缓解直接被扫描工具扫到的风险 [4]。
- **拒绝“大门全开”**：切勿为了省事直接在防火墙中允许 WAN 区域的所有数据无条件转发至 LAN 区域，应遵循“按需针对性放行”原则 [7, 14]。
- **加密与鉴权**：公网访问路由器后台必须使用 [[TLS]]/HTTPS 加密传输防密码窃听，推荐正规 CA 证书以防中间人攻击；所有暴露在公网的服务（包含节点、[[OpenClash]] 的 WebUI）必须设置长而复杂的鉴权密码 [5, 9]。
- **屏蔽 UDP 代理端口**：使用 [[OpenClash]] 等工具提供 Socks 代理时，强力建议不要在防火墙放行该端口的 UDP 流量，因为存在可绕过密码认证的已知安全漏洞 [9]。

## 关键概念

### [[防火墙]]
- **定义**：运行在路由器中，决定是接受还是拒绝进出网络数据包的系统组件 [1]。
- **视频上下文**：作者解释即使设备分配了公网 IPv6，若不改动默认设置外部用户依然无法随意访问内网，因为防火墙默认禁止互联网流量进入内部网络 [1, 2]。
- **为什么重要**：它是保障拥有公网 IP 后家庭网络不被外部恶意访问的核心防线 [1, 3]。
- **关联概念**：未明确提及

### [[区域]]
- **定义**：OpenWrt 防火墙默认创建的网络逻辑划分，主要分为代表局域网的 LAN 区域和代表互联网的 WAN 区域 [2]。
- **视频上下文**：视频说明局域网设备接入 LAN 区域，而连接光猫的整个互联网络属于 WAN 区域，且默认禁止 WAN 区域数据转发到 LAN 区域 [2]。
- **为什么重要**：这是理解 OpenWrt 防火墙数据转发规则以及配置放行策略的基础 [2, 3]。

### [[入站与出站]]
