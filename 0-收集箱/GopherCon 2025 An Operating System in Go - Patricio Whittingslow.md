## 一句话总结
探讨在没有操作系统的裸机环境（[[GOOS=none]]）下运行 [[Go]] 程序的现状、[[TinyGo]] 的性能优势及嵌入式开发经验。

## 目标受众与前置知识
- **适合谁看**：[[Go]] 开发者、嵌入式系统工程师、对底层硬件与操作系统交互感兴趣的程序员。
- **前置知识**：[[Go]]、[[操作系统]]、[[TCP]]/[[IP]] 协议、[[微控制器]]（[[MCU]]）、[[C]]。

## 核心观点
1. **并非所有 Go 程序都需要操作系统环境**
   - 为什么重要：去除对操作系统的依赖可让开发者完全掌握硬件限制，具有巨大的安全潜力，并能消除所有的 [[CGO]] 依赖。
   - 依据：讲者提到了 [[Andrea Barisani]] 在问题追踪器上提出的在 Go 运行时中加入裸机（Bare metal）支持的提案（讲者原话）。
2. **TinyGo 在嵌入式设备上展现出了极强的性能与稳定性，优于部分传统 C/Rust 方案**
   - 为什么重要：打破了微控制器开发必须完全依赖 C 或 Rust 的固有印象，证明 Go 在底层系统开发中兼具小体积与实时性。
   - 依据：学术论文测试 [[NIST]] 决赛密码学算法 [[tiny jamboo]] 发现，[[TinyGo]] 生成的可执行文件比 [[Rust]] 和 [[C]] 小 3.5 倍（讲者原话）；在另一篇针对 [[ESP32]] 的评测论文中，与其他运行在 [[FreeRTOS]] 上的语言相比，TinyGo 展现出 0 抖动（Jitter-free）的恒定时间运行特性（屏幕展示及讲者原话）。
3. **嵌入式 Go 开发需要采取特殊的内存与指针管理策略**
   - 为什么重要：微控制器（如 [[Raspberry Pi Pico]]）内存资源极其有限，必须避免运行时的动态堆分配以保证稳定运行。
   - 依据：讲者分享了三种实战技巧，如利用容量检查进行切片复用的模式（reuse slice idiom）以及使用 `print-alloc` 标志检测编译分配（讲者原话）。

## 概念关系
- `[[Syscall6]]` → `[[Linux]]` 等操作系统：正常的 Go 程序进行如 `[[DialTCP]]` 的网络请求时，底层依赖汇编级别的系统调用（如 `[[RawSyscall]]`、`[[Syscall6]]`、`[[ABIInternal]]`）来驱动操作系统的网络内核（如发送 [[ARP]]、[[IP]] 包）。
- `[[GOOS=none]]`（或 goose is noose） → `[[Bare Metal]]`：将环境标志设为无操作系统时，代码直接作为独立环境运行在 [[微控制器]]（[[MCU]]）上，此时无操作系统的干预。
- `[[TinyGo]]` ↔ `[[FreeRTOS]]`：虽然 TinyGo 自身没有被称为操作系统，但它直接实现了多核处理、协程管理等操作系统核心任务，并在某些层面比拥有 20 年历史（讲者原话）的嵌入式实时操作系统 FreeRTOS 更加轻量且无抖动。

## 详细笔记
### 引入：程序的操作系统依赖真相
- 讲者 [[Patricio Whittingslow]] 在 [[Gophercon]] 2025 会议上分享他作为社区中的“反主流”开发者在操作系统层面的探索，他在去年曾写过一个 3D 设计库。
- 通过追溯 `[[DialTCP]]` 处理 [[HTTP]] 请求的底层调用栈（`[[internetSocket]]` -> `[[socket]]` -> `[[RawSyscall]]` -> `[[Syscall6]]`），他展示了底层其实是一段没有任何 Go 代码的汇编程序。该汇编指令最终调用了由操作系统（如 [[Linux]]）提供的功能。
- 讲者以自己的 [[ThinkPad E590]] 笔记本及 Wi-Fi 加密狗为例（讲者原话），指出即使是简单的网络连接也深度依赖操作系统的网络内核处理，就像《黑客帝国》里的 [[Alice]] 一样，开发者日常被操作系统“蒙蔽了双眼”。

### 裸机 Go 提案与生态演进
- 讲者介绍了 Go 团队及社区中对“去除操作系统”的努力：[[Andrea Barisani]] 提出了支持 Bare metal（裸机）的提案，该提案的目的是移除 Go 运行时的系统依赖。
- 讲者引出了“嵌入式 Gopher”的概念，指代为像 [[Airbus]] 飞机飞行计算机或智能手表等设备开发裸机 Go 程序的人。
- 展示了多个无操作系统依赖的 Go 生态项目：
  - [[Tamago]]：由 [[Andrea Barisani]] 创建，专注于移除 Go 程序中的 [[C]] 和操作系统依赖。
  - [[TinyGo]]：让 Go 在小型设备、或者由 [[CDN]] 分发的 [[WebAssembly]] 二进制文件中运行。
  - [[Embedded Go project]]：去除了操作系统和 C 的依赖，实现了一个基于 [[ARMv7-M]] [[Thumb]] / [[Thumb-2]] 架构的编译器。

### 性能评测：TinyGo 对比 C 与 Rust
- 提到了针对嵌入式语言表现的两篇学术论文对比了 [[TinyGo]]、[[Rust]]、[[C]] 及 [[MicroPython]]。
- 在运行密码学算法 [[tiny jamboo]] 于 [[Raspberry Pi Pico]] 上的测试中，TinyGo 的体积比 Rust 和 C 小 3.5 倍（讲者原话），虽然性能慢 1.5 到 2 倍，但讲者对该论文作者未开源实现代码表示质疑（讲者原话）。
- 在另一篇基于 [[ESP32]] 的论文中，数据显示 TinyGo 在性能上与 C 互有胜负，最惊人的是 TinyGo 表现出零性能抖动（Jitter-free），而基于 [[FreeRTOS]] 的 Rust、C 和 MicroPython 都有不同程度的波动（屏幕展示）。

### TinyGo 是操作系统吗？
- 讲者将 [[TinyGo]] 与由 [[AWS]] 资助开发的传统微控制器操作系统 [[FreeRTOS]] 进行对比。
- 虽然 FreeRTOS 具备多年的打磨并支持多达 40 种架构（讲者原话），但仅有 7 年历史（讲者原话）且由社区驱动的 TinyGo 已经支持跨 150 种开发板的 50 种架构（讲者原话）。
- 目前 TinyGo 已基本实现了操作系统核心任务，如微控制器上的多核处理和协程处理（感谢了贡献者 [[Aika]]），并且有了全新的用户空间网络栈 [[Elnetto]]（感谢了讲者自己、贡献者以及 [[Ron]]）。
- 讲者认为相比于处理繁琐的 [[CMake]] 文件和链接器冲突，TinyGo 开箱即用。讲者引用了前一天 [[Cameron]] 在主题演讲中提及的一句 [[Rob Pike]] 名言表示高度赞同：“Go 的语言设计是为软件工程服务的。”

### 面向下一代的嵌入式 Go 实践智慧
- 作为 [[Raspberry Pi Pico SDK]] 和 TinyGo 核心代码库的贡献者，讲者分享了三种避坑模式：
  - **复用切片模式（Reuse slice idiom）**：针对微控制器开发，利用切片容量预分配，若首个 Reset 采用最大 size 参数即可保证在生命周期内永不产生内存重分配。
  - **分代索引（Generational indices）**：讲者在开发网络栈 [[El Netto]] 时发现该模式极适合网络或文件系统。在关闭一个 TCP 协议连接重用内存时，可借此一次性使所有相关的旧指针句柄失效。
  - **分配检测策略**：使用 TinyGo 的 `print-alloc` 标志（可结合 `[[regexp]]` 过滤）以检测潜在的堆分配。展示了一个极简程序，解释了初始的三个合理分配：底层的 `[[make]]` 调用分配、[[Raspberry Pi Pico]] 默认的 8KB（讲者原话）协程栈分配、以及与协程关联的任务数据结构分配。

## 金句亮点
- "The operating system is everywhere... It is the world that's been pulled over our eyes to blind you from the truth. The truth being you don't need an operating system for some tasks." — 意译 / 深刻揭示了软件开发中被忽视的操作系统依赖盲区，并引出裸机编程的价值。
- "Go is about language design in the service of software engineering." — 价值 / [[Rob Pike]] 的原话，高度总结了 Go 语言无需配置、开箱即用的工程实用主义理念。

## 因果关系链
- 发起 `DialTCP` 调用 → 逐层向下触发 `socket` 调用和 `raw syscall` → 最终通过 Go 汇编中的 `syscall` 指令调用底层操作系统的网络内核（如 Linux）[5, 6, 10, 11]。
- 设置 `GOOS=none` 并引入 [[Bare metal]] 支持 → 消除 Go 运行时对底层操作系统的依赖 → 用户完全控制硬件提供的功能，产生特定的安全优势并可能消除程序对 CGO 的依赖 [2, 3, 12]。
- 使用 [[TinyGo]] 在 [[MCU]]（如 ESP32）上运行程序而没有传统操作系统 → 避免了操作系统的多任务调度干扰 → 实现了零 [[Jitter]]（恒定时间运行性能）[7, 8]。
- 在代码中使用 Reuse slice idiom（复用切片惯用法）并在首次调用时分配最大值 → 利用底层内存并仅在需要时分配 → 确保程序在运行期间不会发生意外的内存分配 [9, 13]。

## 历史背景与发展脉络
- 时间线上的关键节点：传统嵌入式操作系统 [[FreeRTOS]] 已开发了 20 多年并由 AWS 托管；相比之下，[[TinyGo]] 是一个由社区驱动的项目，仅有 7 年历史 [14]。演讲者本人参与 [[Bare metal]] Go 开发也已有近 5 年时间 [15]。
- 重要人物和事件的作用：Andrea Barisani 提出了在 Go 问题追踪器中增加 [[Bare metal]] 支持的关键提案，这也是为了消除 Go 程序对 C 语言和操作系统的依赖 [2-4]。此外，演讲者的贡献现已正式成为 [[TinyGo]] 和 Raspberry Pi Pico SDK 的一部分 [9, 15]。

## 常见误解澄清
- **关于 TinyGo 内存分配的误解**：开发者可能认为即使是极简的 [[TinyGo]] "Hello World"（如 blinky 程序）在 [[MCU]] 上也不会有任何堆分配。实际上它在启动时会有 3 次基础分配（基本的 allocator 调用、8KB 的 goroutine 栈大小分配、以及与 goroutine 关联的 task 数据结构），但这仅发生在程序启动时 [16, 17]。
- **关于 TinyGo 缺乏操作系统功能的误解**：因为 [[TinyGo]] 没有搭载操作系统，常被认为功能缺失。实际上，[[TinyGo]] 已经处理了多核处理、[[MCU]] 上的 goroutine 调度以及用户态网络栈等传统 OS 任务，演讲者指出它在某种程度上已经非常接近一个操作系统 [8, 14, 18, 19]。

## 类比与通俗解释
- **《爱丽丝梦游仙境》与《黑客帝国》类比**：演讲者将开发者探索 Go 底层代码（如发现没有主体的 Go 汇编指令）比作“掉进兔子洞（tumbling down the rabbit hole）”，并将无处不在的操作系统比作《黑客帝国》中“蒙蔽双眼的世界（the world that's been pulled over our eyes to blind you from the truth）”，以此通俗地解释操作系统是如何在背后默默控制一切的 [2, 20]。
- **智能手表类比**：为了通俗解释什么是“嵌入式系统（包含在更大机械系统中的计算机）”，演讲者使用智能手表作为例子——当你戴着智能手表时，“你（佩戴者自身）就是那个更大的机械系统” [1]。

## 关键概念

### [[GOOS]]
- **定义**：Go 语言的环境变量，用于告知编译器编译出的程序将在哪种操作系统上运行 [1]。
- **视频上下文**：作者在演讲开篇提到它，并介绍了将其设置为 "noose"（无操作系统）的提案 [1-3]。
- **为什么重要**：它是理解 Go 程序如何与底层操作系统交互的基础设置 [1, 3]。
- **关联概念**：扩展 → [[Bare metal]]

### [[Syscall]]
- **定义**：由操作系统提供的系统调用指令，用于执行底层硬件或系统级操作 [4]。
- **视频上下文**：作者通过追踪 `DialTCP` 函数，展示了 Go 代码最终会触发该指令，由操作系统（如 Linux）接管网络连接的建立 [1, 4-6]。
- **为什么重要**：揭示了常规程序在后台对操作系统的深度依赖 [2, 7]。
- **关联概念**：对比 → [[Bare metal]]

### [[Bare metal]]
- **定义**：没有任何操作系统的运行环境，编译后的程序是计算机或微控制器上唯一运行的程序 [3]。
- **视频上下文**：作者介绍了在 Go 中消除操作系统依赖的提案，并讨论了其在提升安全性和消除 CGO 依赖方面的意义 [2, 3, 8]。
- **为什么重要**：它是嵌入式 Go 编程的核心范式，让开发者完全控制硬件限制和功能 [3, 8]。
- **关联概念**：依赖 → [[MCU]]

### [[MCU]]
- **定义**：微控制器单元（Microcontroller unit），本质上是一个 CPU，通常作为更大型机械系统的一部分 [3, 9]。
- **视频上下文**：作者展示了 Raspberry Pi Pico 等硬件上的微小芯片，指出这是程序编译和运行的底层目标平台 [8, 9]。
- **为什么重要**：它是嵌入式编程和 TinyGo 代码实际部署的核心硬件基石 [8, 10]。
- **关联概念**：属于 → [[Bare metal]]

### [[TinyGo]]
- **定义**：一个致力于在极小环境（如微控制器或 WebAssembly）中编译和运行 Go 语言的编译器项目 [11]。
- **视频上下文**：作者将其生成的执行文件大小和运行表现与 Rust、C 和 FreeRTOS 进行了基准测试对比 [10, 12-14]。
- **为什么重要**：它是实现 Go 语言在嵌入式系统上高效运行的关键工具，具有极其轻量且开箱即用的优势 [15]。
- **关联概念**：对比 → [[FreeRTOS]]

### [[Jitter]]
- **定义**：程序运行时间的波动，零抖动意味着程序以恒定的时间运行 [13]。
- **视频上下文**：作者引用论文指出，相较于运行在操作系统上的其他语言，TinyGo 程序表现出零抖动 [13, 14]。
- **为什么重要**：它是衡量嵌入式和实时系统性能的重要指标，证明了无操作系统架构在时间稳定性上的优势 [13, 14]。
- **关联概念**：未明确提及

### [[FreeRTOS]]
- **定义**：一个拥有 20 多年历史、支持众多微控制器架构的传统嵌入式实时操作系统 [14, 16]。
- **视频上下文**：作者将其复杂度和运行表现与免配置的 TinyGo 进行对比，指出基于该系统运行的程序存在运行时的抖动 [14-17]。
- **为什么重要**：作为嵌入式领域的传统标准，它是理解无操作系统 Go 程序轻量化和高效优势的参照物 [15, 16]。
- **关联概念**：对比 → [[TinyGo]]

### [[Reuse slice idiom]]
- **定义**：一种通过检查切片容量并在必要时才分配内存，从而复用底层内存的编程模式 [18, 19]。
- **视频上下文**：作者在分享嵌入式系统编程经验时提出，用于在处理缓冲区对象时避免堆分配 [18, 19]。
- **为什么重要**：在资源极其受限的微控制器环境中，避免不必要的内存分配是至关重要的性能优化技巧 [18, 20]。
- **关联概念**：未明确提及

### [[Generational indices]]
- **定义**：一种用于在对象更改或不再存活时，一次性使该对象的多个相关指针全部失效的模式 [19, 21]。
- **视频上下文**：作者在开发网络协议栈时发现了此模式，指出它非常适合在重用内存（如关闭的 TCP 连接）时保持状态安全 [19, 21]。
- **为什么重要**：解决了底层系统开发（如网络协议栈和文件系统）中内存复用和清理陈旧引用的难题 [21]。
- **关联概念**：未明确提及
