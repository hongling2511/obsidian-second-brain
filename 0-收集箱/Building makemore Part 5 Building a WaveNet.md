# Building makemore Part 5: Building a WaveNet

## 一句话总结
本视频将字符级语言模型重构为类 WaveNet 的层级网络，并演示了对齐 PyTorch API 的底层实操。

## 核心观点
1. **观点概括**：逐步融合上下文信息比单层直接压缩更有效。
   - 为什么重要：将长序列一次性压入单个隐藏层会过快丢失信息，通过类 [[WaveNet]] 的层级树状结构（Progressive Fusion）两两融合，能更高效地利用模型容量。
   - 说服力评估：作者通过控制变量法，在保持参数量（约22k）相同的情况下，对比了单层拉伸输入与层级架构的性能，推演逻辑非常严密，说服力强。
2. **观点概括**：模块化重构代码（PyTorchification）是构建复杂网络的基础。
   - 为什么重要：模仿 `torch.nn` 实现 `Sequential`、`Embedding`、`Flatten` 等模块，能规范化输入输出的接口，隔离网络层的内部状态（如 [[BatchNorm]] 的训练/评估模式），极大减少隐藏 Bug。
   - 说服力评估：通过现场将散乱的面向过程代码封装为面向对象模块，使前向传播代码缩减至只需一行 `model(x)`，直观展现了工程化代码的优势。
3. **观点概括**：开发神经网络本质上是大量“照看张量形状”（Babysitting Shapes）的工作。
   - 为什么重要：张量维度的不匹配在多维矩阵运算中往往不会报错（因自动广播机制），但会导致错误的统计学结果（例如在错误的维度上求平均），毁掉模型性能。
   - 说服力评估：极为有力。作者现场揭示了一个在使用 `BatchNorm1d` 处理 3D 张量时，因维度设定错误导致只在一个维度上计算均值的隐匿 Bug，极具警示意义。

## 步骤拆解
1. **优化 Loss 曲线可视化**
   - 具体操作描述：单批次（batch）的数据量过少会导致 Loss 曲线锯齿状波动剧烈，为了观察到有代表性的下降趋势，将一维的 loss 列表转化为二维的 [[Tensor]]，然后沿行求平均以绘制平滑曲线 [12, 15, 16]。
   - 关键代码/命令：`torch.tensor(lossi).view(-1, 1000).mean(1)` [12, 16]。
   - 注意事项：使用参数 `-1` 可以让 PyTorch 自动推断总行数，这里等于将连续的 1000 个 loss 元素折叠为一行并求均值 [12, 16]。

2. **代码的 PyTorch 化重构 (PyTorchification)**
   - 具体操作描述：将之前零散的操作和层级封装为贴近 `torch.nn` 原生 API 的规范类，包括剥离出单独的 [[Embedding]] 层、[[Flatten]] 层，并编写一个统一的容器 [[Sequential]] 来依次执行网络层的前向传播 [2, 17-19]。
   - 关键代码/命令：定义 `class Sequential:` 并使用循环顺序调用所有层的计算 `out = layer(out)` [19]。
   - 注意事项：保持自定义层的 API 和签名逻辑与官方框架一致，能让网络像“乐高积木”一样自由拼接，且在未来可以零成本切换到官方 `torch.nn` 库 [1, 9, 14]。

3. **构建层级化融合架构 (Hierarchical Architecture)**
   - 具体操作描述：将输入上下文 `block_size` 从 3 增至 8，抛弃一次性把所有输入信息压缩进单层的暴力做法 [20]。把 `Flatten` 层修改为支持动态维度的 `FlattenConsecutive` 层，每次将两个相邻字符（bigram）融合，逐步堆叠构造出树状的深层网络架构 [4, 21-23]。
   - 关键代码/命令：使用 `x.view(B, T // n, C * n)` 进行维度重排 [23]。
   - 注意事项：[[PyTorch]] 中的 [[线性层 (Linear Layer)]] 是支持三维张量运算的，矩阵乘法只针对最后一个维度（特征维度）生效，左侧增加的维度将被视为额外的独立批次（Batch）维度被并行处理 [13, 24]。最后融合到只剩一项时，需要使用 `.squeeze(1)` 移除长度为 1 的冗余维度 [25]。

4. **修复 [[Batch Normalization]] 层的多维张量兼容 Bug**
   - 具体操作描述：模型变深后，送入 BatchNorm 层的变为三维张量（Batch × 组队维度 × 通道数），之前的二维减秩逻辑会导致在各个空间位置独立计算均值。因此需修改层代码，让均值和方差同时涵盖 batch 和 sequence 两层维度 [26-28]。
   - 关键代码/命令：检测维度 `if x.ndim == 3:` 时，执行在元组维度上的规约 `x.mean(dim=(0, 1))` [29, 30]。
   - 注意事项：官方的 `BatchNorm1d` 期望的数据维度格式是 `N x C x L`（通道在中间），而本教程自行实现的结构是 `N x L x C`（通道放在最后），需要警惕这种与官方 API 存在的分歧 [30, 31]。

## 常见坑与解决方案
- **问题描述：测试/评估时产生 NaN（Not a Number）报错或结果异常。**
  - 解决方案：忘记切换 [[Batch Normalization]] 层的状态是一个高频 Bug [7, 32]。如果向处于 `training=True` 状态的 BatchNorm 传入单独一个样本进行评估，由于单个数字的方差不存在（非数值），整个处理流将被污染 [33]。需要在推理前遍历网络将层设定为 `training=False` 以使用全局估算的统计量 [7, 15, 33]。
- **问题描述：网络结构升级为层级化后效果提升不明显，运行无报错但由于 Broadcasting 特性发生了“静默错误 (Silent Bug)”。**
  - 解决方案：三维张量传入旧版 2D BatchNorm 依然能跑通，但实质上是在平行运算多个子统计量（例如维护 4*68 个状态而不是预期的 68 个通道状态）[26-28]。解决方案是通过检查内部变量的 Shape（如打印 `running_mean.shape`）来确认是否真正对多出来的序列维度进行了归一化 [31]。

## 最佳实践
- **拒绝滥用自动维度推断 (`-1`)**：在重塑张量（view）时，最佳做法是显式写出计算结果（如 `T // n`）而不是依赖 `-1`。显式的维度规划能让因代码变更导致的 Shape 不匹配问题立刻暴露并触发错误提示，防止错误在后续流程中放大 [23]。
- **分离“实验原型”与“工程代码”流程**：极力推荐在开发新网络架构时，先利用 [[Jupyter Notebook]] 进行验证，并通过循环逐层打印出张量的具体 Shape 来“照看”数据流转（babysitting the shapes）；确认无误后，再将代码转移粘贴到 [[VS Code]] 中的独立 python 脚本里进行长期训练 [11, 34]。
