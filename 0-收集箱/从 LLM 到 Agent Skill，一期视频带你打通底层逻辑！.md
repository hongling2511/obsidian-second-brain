## 核心观点
1. **观点概括**：[[LLM]] 本质上是一个基于 [[Transformer]] 架构的文字接龙游戏。
   - 为什么重要：打破大模型具备人类思维的拟人化错觉，揭示其逐字预测输出概率的数学函数本质。
   - 依据：作者通过「马克的视频怎么样？」的例子，演示模型如何预测下一个词并将其塞回输入继续预测，直到输出结束标识符。
2. **观点概括**：[[token]] 是大模型处理文本的最基本单元，且与人类理解的「词」非一一对应。
   - 为什么重要：解释了文本输入输出限制的底层逻辑，以及为什么需要 [[tokenizer]] 作为中间人进行编解码。
   - 依据：作者展示了 Open AI 的转换页面，演示「工作坊」、「程序员」甚至特殊字符如何被切分成多个不规则的 [[token]]。
3. **观点概括**：[[tool]] 和 [[MCP]] 协议赋予了模型感知和影响外部环境的能力。
   - 为什么重要：大模型本身无法获取外部实时信息（如天气），通过规范化的工具接入和调用，极大地扩展了其实用边界。
   - 依据：以调用天气查询工具为例，说明模型只输出调用指令，由平台执行后返回给模型归纳；并说明 [[MCP]] 解决了跨平台重复开发工具的代码痛点。
4. **观点概括**：[[agent]] 通过结合工具具备自主规划能力，而 [[agent skill]] 则为其提供个性化的执行规范。
   - 为什么重要：解释了从单一任务到复杂自主任务的演进过程，以及如何通过规范化的文档低成本满足特定业务格式与逻辑需求。
   - 依据：作者演示了在 [[Cloud Code]] 中通过编写包含元数据和指令层（目标、步骤、规则等）的 Markdown 文件，让助手根据天气自动判断出门携带物品并按特定格式输出。

## 详细笔记
- **开场 / 问题定义**：
  - 前段：点出当今 AI 领域新名词繁多（[[LLM]]、[[token]]、[[prompt]]、[[MCP]]、[[agent]]、[[agent skill]] 等），本视频旨在抛开商业概念，从底层工程视角逐一拆解这些概念的确切含义。
- **主体部分 1**：大模型基础运行原理（[[LLM]]、[[token]] 与 [[context]]）
  - 前段：[[LLM]] 的底层引擎是 2017 年提出的 [[Transformer]] 架构。其本质是「文字接龙」，逐字预测下一个概率最高的词并追加到输入中循环。
  - 前段：模型内部只进行数字矩阵运算，需借助中间人 [[tokenizer]] 进行编码（文字转数字/token ID）和解码。切分出的最小片段即为 [[token]]。平均 1 个 token 约等于 0.75 个英文单词或 1.5-2 个汉字。
  - 前段：大模型没有真实记忆，靠每次接收的 [[context]]（包含用户问题、对话历史、系统提示等）进行临时记忆。其容量上限为 [[context window]]。面对超长文档，需使用 [[RAG]] 技术抽取匹配片段以突破窗口限制并控制成本。
- **主体部分 2**：引导大模型输出（[[prompt]] 与 [[tool]]）
  - 中段：[[prompt]] 是给大模型的具体指令，分为用户输入的 [[user prompt]]（说明具体任务）和开发者在后台配置的 [[system prompt]]（说明人设和做事规则）。
  - 中段：大模型无法感知外界环境，需借助 [[tool]]（本质是外部函数）。平台充当传话筒，大模型负责选择工具并生成参数，平台执行调用后，大模型再进行结果归纳总结。
  - 中段：[[MCP]]（Model Context Protocol）是一套统一的工具接入标准协议，让开发者只需开发一次工具，即可在所有支持该协议的平台上运行，避免重复造轮子。
- **主体部分 3**：高阶自主系统（[[agent]] 与 [[agent skill]]）
  - 后段：当任务需要多次调用工具（如先查定位，再查天气，再搜店铺）并不断思考下一步时，这种能自主规划、自主调用工具完成任务的系统称为 [[agent]]。
  - 后段：为解决每次提问都需要重复输入大量私人规则和格式要求的痛点，引入了 [[agent skill]]。它本质是一份给 agent 看的 Markdown 说明文档。
  - 后段：详细演示了 [[agent skill]] 的结构，包含原数据层（name, description）和指令层（目标、步骤、判断规则、输出格式、示例），并在 [[Cloud Code]] 框架中实机演示了根据天气提示出门携带物品的案例。
- **结尾 / 行动建议**：
  - 后段：快速回顾所有的核心概念，指出无论是 [[Cloud Code]] 还是其他主流框架，本质都在此概念体系下运作。

  - 价值：用通俗的比喻清晰定义了复杂数据转换过程的核心角色与处理的最小颗粒度。

## 因果关系链
- 模型只认识数字且本质是数学函数 → 引入 [[Tokenizer]] 进行编解码 → 建立人类文字与模型数字之间的桥梁
- 提问或指令（[[Prompt]]）过于模糊 → 模型无法准确理解意图 → 输出内容发散或不符合用户预期
- 开发者在后台配置 [[System prompt]] → 约束大模型的人设和做事规则 → 引导模型以特定角色或方式（如启发式引导而非直给答案）输出结果
- [[LLM]] 无法感知外界环境且知识库截止于特定时间 → 平台向模型提供外部 [[Tool]]（工具/函数）→ 模型得以获取实时数据（如天气、定位）并解决复杂问题
- 各平台接入工具的规范不一致 → AI 圈推出统一的 [[MCP]]（模型上下文协议）→ 开发者只需编写一次代码，工具即可在所有支持该协议的平台使用
- [[Agent]] 不了解用户的私人习惯与特定的格式要求 → 预先编写包含指令与规则的 [[Agent Skill]]（说明文档）→ [[Agent]] 能够按照特定步骤、判断规则和格式精确完成任务

- 2022年底：OpenAI 发布 GPT-3.5，成为第一个真正达到可用级别的 [[LLM]]，引爆了 AI 浪潮。
- 2023年3月：GPT-4 发布，将 AI 能力天花板拉到了新高度。

## 常见误解澄清
- **关于生成的误解**：认为大模型是一次性输出完整回答。实际上它是按 [[Token]] 一个一个输出，通过不断把新词塞回输入来预测下一个词。
- **关于Token的误解**：认为 [[Token]] 就等于“词”。实际上 [[Token]] 和词并非一一对应，它是模型自己学会的文本切分规则，一个词或字符可能被切分为多个 [[Token]]。
- **关于记忆的误解**：认为大模型像人一样有记忆。实际上模型本身不具备记忆，而是背后的程序在每次提问时，自动将历史对话作为 [[Context]] 一起发送给模型。
- **关于Prompt的误解**：认为 [[Prompt]] 是非常复杂高端的技术。实际上它仅仅是给大模型的一个问题或指令。
- **关于工具调用的误解**：认为调用 [[Tool]] 的是模型本身。实际上大模型只能输出调用指令文本，真正执行调用动作并返回结果的是“平台”。

## 类比与通俗解释
- 用**文字接龙游戏**类比 [[LLM]] 的底层生成原理（循环预测下一个概率最高的词）。
- 用**中间人/翻译**类比 [[Tokenizer]] 的作用（在人类文字与模型数字之间负责编码和解码）。
- 用**临时记忆体**类比 [[Context]]（装载历史记录、系统规则和当前输入等信息）。
- 用**《哈利波特》全集**类比 100万个 [[Token]] 的 [[Context window]] 容量（解释上下文窗口的大小）。
- 用**传话筒**类比平台在用户、[[LLM]] 和 [[Tool]] 之间的职责（负责信息的上传下达与串联）。
- 用**手机的 Type-C 接口**类比 [[MCP]] 协议（解释统一接入标准如何让多平台通用变得方便）。
- 用**说明文档**类比 [[Agent Skill]] 的本质，并将其中的“元数据层”类比为这份文档的**封面**。

## 关键概念

### [[LLM]]
- **定义**：全称 Large Language Model（大模型），是基于 Transformer 架构训练、本质上在进行“文字接龙”概率预测游戏的底层引擎 [1, 2]。
- **视频上下文**：作为所有 AI 技术的最底层基础被首先介绍，它接收输入并逐个输出特定的数据单元以生成完整回答 [1-3]。
- **为什么重要**：它是驱动后续所有 AI 技术（如 Agent、Tool 等）运作的核心算力底座 [4]。
- **关联概念**：依赖 → [[Token]]

### [[Token]]
- **定义**：大模型处理文本的最基本、最小数据单元，是模型内部学会的一套文本切分规则产生的结果 [3-5]。
- **视频上下文**：作者通过拆分“马克的视频怎么样”、“工作坊”和英文单词，说明文字和 Token 并非一一对应，大模型是按 Token 为单位接收和输出信息的 [3, 5]。
- **为什么重要**：打破了人类“以词为单位”的直觉，是理解大模型底层计算原理和计费限制的基础 [3, 5]。
- **关联概念**：依赖 → [[Tokenizer]]

### [[Tokenizer]]
- **定义**：人类与大模型之间的“翻译官”，负责将文字切分并映射为数字的编码过程，以及将数字还原为文字的解码过程 [3, 6]。
- **视频上下文**：因为大模型本质上是一个全量进行矩阵运算的庞大数学函数，只认识数字不懂文字，所以必须通过它作为中间人来处理输入和输出 [6]。
- **为什么重要**：解释了大模型如何“读懂”和“生成”人类语言的关键工程中间件 [6]。
- **关联概念**：未明确提及

### [[Context]]
- **定义**：大模型每次处理任务时所接收到的所有信息总和，相当于大模型的一个临时记忆体 [4, 7, 8]。
- **视频上下文**：作者指出大模型本身没有记忆，之所以能进行多轮对话，是因为背后的程序每次都会将历史对话、用户问题以及系统提示词等信息打包作为 Context 发送给它 [7, 8]。
- **为什么重要**：解释了无记忆状态的数学函数如何实现连贯对话与语境理解的底层机制 [7, 8]。
- **关联概念**：受限于 → [[Context window]]

### [[Context window]]
- **定义**：代表 Context 能够容纳的最大 Token 数量，即大模型单次处理信息的极限容量 [8, 9]。
- **视频上下文**：在讲解 Context 后引入，用以说明模型单次能接收的信息上限，并对比了不同主流模型能装载几十万到上百万 Token（相当于整部《哈利波特》）的容量 [8]。
- **为什么重要**：直接决定了大模型单次能处理的文档长度和复杂任务的边界 [8]。
- **关联概念**：属于 / 依赖 / 对比 / 扩展 → [[Context]]

### [[Prompt]]
- **定义**：用户或系统当前给大模型下达的具体任务指令或问题 [9, 10]。
- **视频上下文**：作者解释说大模型需要指令才运转，Prompt 写得是否清晰具体，直接决定了大模型输出的质量和方向 [10, 11]。
- **为什么重要**：它是人类控制大模型输出意图、角色扮演和行为边界的最直接手段 [10, 11]。
- **关联概念**：扩展 → [[System prompt]]

### [[System prompt]]
- **定义**：开发者在后台配置的提示词，用于向大模型设定其人设和做事的底层规则 [9, 11]。
- **视频上下文**：作者举例说明，为了让辅导数学的 AI 不直接给答案而是引导思考，开发者必须在后台设置约束条件，这部分内容用户不可见但会持续影响模型行为 [11, 12]。
- **为什么重要**：它能在大模型执行具体任务之前建立强制性的角色定位和行为规范框架 [12]。
- **关联概念**：↔ 对比 ↔ [[User prompt]]

### [[Tool]]
- **定义**：本质上是一个可输入和输出的外部函数，用于给大模型提供获取外部数据或执行动作的能力 [9, 13]。
- **视频上下文**：由于大模型无法联网查阅当前天气，作者展示了如何通过平台赋予其天气查询、定位等工具，让模型能感知并影响外部环境 [12-14]。
- **为什么重要**：弥补了大模型仅依靠过往训练数据预测文本的弱点，是系统走向自动化的关键外部延伸 [13, 15]。
- **关联概念**：依赖 → [[MCP]]

### [[MCP]]
- **定义**：全称 Model Context Protocol（模型上下文协议），是一套统一的工具（Tool）接入规范和标准协议 [9, 16]。
- **视频上下文**：因为不同平台接入工具的规范不同会导致重复开发，AI 圈推出了这个统一标准，使开发者只需写一次代码即可在所有支持该协议的平台上使用该工具 [15, 16]。
- **为什么重要**：极大地简化了跨平台开发成本，是构建丰富 AI 工具生态的关键工程统一标准 [16]。
- **关联概念**：依赖 → [[Tool]]

### [[Agent]]
- **定义**：一种能够自主规划步骤、自主多次调用外部工具、持续运作直到完成用户复杂任务的程序系统 [9, 17]。
- **视频上下文**：在单次调用工具无法解决复杂问题时（如根据天气情况寻找附近雨伞店），作者演示了系统如何一步步思考并连续调用定位、天气、店铺搜索来完成目标 [17, 18]。
- **为什么重要**：代表了 AI 从单纯“单次回答问题”向“自主解决复杂多步任务”的智能化范式跃升 [17]。
- **关联概念**：依赖 → [[Tool]]

### [[Agent skill]]
- **定义**：一份提前写好并挂载的 Markdown 格式说明文档，用于向 Agent 规定特定的做事步骤、判断规则和输出格式 [9, 19]。
- **视频上下文**：为了解决通用 Agent 无法掌握用户私人的出差习惯且每次提问要求过长的问题，作者展示了如何编写包含元数据和指令层的文档来定制专属行为 [17, 19, 20]。
- **为什么重要**：赋予了通用智能体高度个性化的定制能力，使其能固化复杂的工作流，免去重复撰写超长提示词的痛苦 [4, 19]。
- **关联概念**：依赖 → [[Agent]]
