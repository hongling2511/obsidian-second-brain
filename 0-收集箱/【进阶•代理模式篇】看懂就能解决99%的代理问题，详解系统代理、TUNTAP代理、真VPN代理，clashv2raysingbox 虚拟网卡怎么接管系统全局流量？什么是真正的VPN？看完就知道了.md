## 一句话总结
详解系统代理、[[TUN]]/[[TAP]]虚拟网卡接管全局流量的原理，并对比科学上网协议与真[[VPN]]的本质区别。

## 目标受众与前置知识
- **适合谁看**：遇到特定软件或游戏无法走代理、节点连通性问题，想了解底层流量接管原理的科学上网用户。（推断）
- **前置知识**：[[TCP/IP]]四层模型、[[NAT]]转换、[[内网穿透]]、操作系统的[[路由表]]（视频提及的讲者过往教程内容）

## 核心观点
1. **系统代理模式具有较大局限性**
   - 为什么重要：解释了为什么部分软件和游戏无法通过常规方式科学上网。
   - 依据：讲者指出绝大部分软件（除浏览器外）不遵循[[系统代理]]，且系统代理通常为[[HTTP]]或[[SOCKS5]]（讲者原话“fsx5”），其中[[HTTP]]不支持[[UDP]]，导致游戏无法使用。
2. **[[TUN]]模式是目前较为完美的客户端代理方式**
   - 为什么重要：解决了系统代理无法接管所有软件流量的痛点。
   - 依据：讲者说明[[TUN]]模式通过在[[网络层]]创建虚拟网卡并修改电脑路由表（涵盖所有IPv4地址），从而强制接管系统所有流量，实现无视软件代理设置的全局代理。
3. **主流科学上网协议不是真正的[[VPN]]**
   - 为什么重要：澄清了翻墙代理与虚拟专用网络的本质技术区别。
   - 依据：讲者演示在使用[[clash]]的[[TUN]]模式时，用[[ping]]命令（[[ICMP]]协议，讲者原话“smp协议”）访问[[谷歌]]会返回“1毫秒的假延迟”（讲者原话），证明[[ss]]、[[vmess]]（讲者原话“vm”）、[[trojan]]（讲者原话“秋卷”）等协议无法封装网络层数据包，而真正的[[VPN]]（如[[WireGuard]]，讲者原话“wg”）可以返回真实延迟。

## 概念关系
- [[系统代理]] ↔ [[TUN]]代理：前者依赖软件主动遵循（应用层设置），后者通过修改路由表在[[网络层]]强制接管全局流量（无视软件设置）。
- [[科学上网协议]] ↔ 真[[VPN]]：前者（如[[ss]]）专为隐藏流量特征翻墙而生，但无法封装网络层数据包；后者（如[[pptp]]、[[IPsec]]、[[OpenVPN]]、[[WireGuard]]）为异地组网/[[内网穿透]]设计，能代理网络层但流量特征明显，不适合科学上网。
- [[gVisor]]（讲者原话“gviser”） ↔ [[System]]协议栈：在处理TUN流量时，前者在用户态解析，性能更好；后者使用操作系统自带协议栈，更稳定但会增加数据重复拷贝。

## 详细笔记

### 代理问题引入与网络通信基础
- **常见痛点**：节点/游戏无法使用、电脑能用手机不能用、能上[[谷歌]]上不了[[YouTube]]等问题，讲者[[布良玲]]（推断为不良林）表示理解本期原理可解决“99%的代理问题”（讲者原话）。
- **家庭网络拓扑与通信流程**：
  - 路径：宽带光猫 → 路由器（通过[[PPPoE]]拨号获取公网IP，如“2.2.2.2”讲者原话） → 电脑等设备（通过[[DHCP]]获取内网IP，如“192.168.0.1”作为网关，讲者原话）。
  - [[TCP/IP]]封装（以访问[[百度]]为例，目标IP假设为“3.3.3.3”讲者原话）：应用层封[[HTTP]] → 传输层封[[TCP]]（目标端口“8连”推断为80端口） → 网络层封IP → 接口层封[[MAC]]。
  - 路由器处理：对数据包进行[[NAT]]转换，将内网IP替换为公网IP后发送至互联网；收到返回数据后再次NAT还原并发给电脑。

### 系统代理模式的执行流程与局限
- **代理工具**：大部分基于[[clash]]（讲者原话“CASH”）、[[v2ray]]（讲者原话“v2ax”/“v2a”）、[[xray]]（讲者原话“瑞”）内核的客户端均支持。如[[v2rayN]]（讲者原话“v2n”）、[[Clash for Windows]]（讲者原话“CASH for Window”）。
- **工作原理**：
  - 软件开启系统代理后，浏览器访问请求交由代理客户端处理。
  - 客户端根据分流规则判断，需代理的数据使用所选节点的协议（如[[ss]]，节点IP假设为“5.5.5.5”讲者原话）进行加密封装。
  - 数据送达节点服务器，解包并代为访问目标网站，返回数据加密传回电脑解密。
- **局限性**：需软件主动遵循，无法接管所有程序（特别是游戏），且[[HTTP]]代理不支持[[UDP]]。

### [[TUN]]/[[TAP]]模式：网络层全局接管
- **核心原理**：创建虚拟网卡，从[[网络层]]拦截并接管系统所有流量（手机全APP翻墙、软路由科学上网均基于此原理）。
- **接管机制（路由表）**：
  - 在[[Windows]]系统通过 `route print`（讲者原话“rootprint”）查看路由表。
  - [[clash]]开启TUN模式后，自动添加覆盖所有IPv4地址的路由条目。数据发往虚拟网卡IP（讲者原话原IP为“198.18.0.1”，网关“198.18.0.2”）。
- **优势与区别**：
  - [[TUN]]相比[[TAP]]更受推荐，因为无需向上封装多余的[[MAC]]地址。
  - 透明代理应用：将虚拟网卡转至路由器（软路由），局域网内设备无需开启任何代理软件即可科学上网。
- **软件支持与配置**：
  - 支持软件：[[clash]]、[[SSTap]]、[[Netch]]（讲者原话“sstapnch”）、[[singbox]]（讲者原话“sbox”）。
  - [[v2ray]]和[[xray]]原生不支持，但[[v2rayN]] 6.0以上版本（讲者原话）通过调用[[singbox]]内核实现了支持，需以管理员权限运行。
  - [[Clash for Windows]]配置：需安装服务模式获取特权。TUN设置中网络协议栈（stack）可选择 [[gVisor]] 或 [[System]]，以及开启自动添加路由（outroot/autodect interface）。

### 真[[VPN]]与代理工具的本质区别
- **概念定义**：VPN全称 Virtual Private Network（虚拟专用网络），核心作用是实现异地私有网络互联（虚拟即物理不直接连接）。
- **技术门槛**：必须能够封装[[网络层]]的数据包。主流翻墙协议（[[ss]]、[[vmess]]、[[trojan]]）做不到这一点，使用代理时的[[ping]]测试（基于[[ICMP]]协议）会返回虚拟网卡直接给出的假延迟/假IP。
- **对比结论**：真VPN（如[[WireGuard]]、[[IPsec]]等）能测出真实延迟，适合内网穿透；但不适合科学上网，因为流量特征未被隐藏，分流也不方便。

## 金句亮点
- “在系统代理模式下软件可以无视系统代理，那么TUN模式做的就是无视软件、无视系统代理，通过配置路由表接管系统所有流量。” — 核心原理解析 / 价值
- “VPN并不是为翻墙而生的，只是因为它能对数据进行加密顺便实现了翻墙功能。相较于专门为隐藏流量特征而生的协议，VPN的流量清晰明了。” — 概念辨析 / 价值

- **关联观点**：2
- **验证标准**：能看到新增的覆盖所有 IPv4 地址且指向虚拟网卡接口（如 198.18.0.1）的路由条目。
- **做什么**：在客户端（如 Clash for Windows）的 TUN 设置中，根据需求将网络协议栈切换为 [[gVisor]]（追求性能）或 [[System]]（追求稳定）。
- **关联观点**：2
- **验证标准**：保存设置并在客户端成功重启 TUN 模式无报错信息。
- **做什么**：在开启代理的TUN模式下，打开命令行使用 `ping` 命令测试谷歌连通性。
- **关联观点**：3
- **验证标准**：观察到返回值为“1毫秒”的假延迟极小值，从而验证当前并非真VPN代理了网络层。

- **前置技能**：掌握 [[TCP/IP]] 四层模型、网络协议栈的数据封装与解封装过程、局域网拓扑结构及路由/NAT 原理 [2-4]。

## 常见坑与解决方案
- **问题描述**：设置系统代理后，部分软件依然无法使用 [5, 11]。
  - 解决方案：使用 [[TUN/TAP]] 模式，通过创建虚拟网卡从网络层接管系统的所有流量 [5]。

- **问题描述**：非常严格的程序或游戏可能会检测出电脑开启了虚拟网卡代理 [6]。
  - 解决方案：将虚拟网卡转移到路由器里作为透明代理（软路由），局域网内的设备无需运行任何代理工具即可被接管流量 [6]。

- **问题描述**：[[v2rayN]] 开启 TUN 模式时，弹出的黑框控制台很碍事 [7]。
  - 解决方案：在 TUN 设置中将“显示控制台”关闭，保存后再尝试开启 [7]。

- **问题描述**：使用 TUN 模式时，用 ping 命令测试网络会返回 1ms 的假延迟，甚至返回假 IP [13]。
  - 解决方案：未明确提及 [13]。

## 最佳实践
- 相比于 TAP 模式，更推荐使用 TUN 模式。因为向下封装 MAC 地址对代理没有实际作用，没有必要使用 TAP 模式 [9, 14]。
- 在配置 [[Clash for Windows]] 协议栈（stack）时，如果电脑兼容性没问题，推荐使用 `gvisor`（在用户空间直接解析数据，性能更好）；若追求极致稳定则使用 `system`（调用操作系统自带协议栈，但会增加数据重复拷贝） [12]。
- 如果平时的需求只是浏览网页和聊天，推荐使用系统代理模式，这是最方便省事的模式 [11, 15]。
- 强烈不推荐使用真正的 [[VPN]] 来进行科学上网，因其流量特征清晰，且分流十分不便；真正的 [[VPN]] 更适合用于实现内网穿透（异地组网） [8, 15, 16]。

## 关键概念

### [[系统代理]]
- **定义**：将数据交给本地 HTTP 或 SOCKS 服务的代理模式 [1, 2]。
- **视频上下文**：作者提到这是大部分电脑用户最常用的方式，通过 Clash 或 v2ray 等设置，浏览器通常会遵循它，但许多软件和游戏并不支持 [2-4]。
- **为什么重要**：它是最方便基础的代理方式，但其“不支持 UDP”和“部分软件不遵循”的缺陷，解释了为什么有时即使开了代理也无法使用特定软件 [2]。
- **关联概念**：对比 → [[TUN/TAP 模式]]

### [[TUN/TAP 模式]]
- **定义**：通过创建虚拟网卡并在网络层接管系统所有流量的代理模式 [5]。
- **视频上下文**：作者重点讲解当软件无视系统代理时，开启此模式可以无视软件的独立行为，强制拦截并处理所有发送到互联网的数据 [5, 6]。
- **为什么重要**：它解决了系统代理无法覆盖游戏和特殊软件的痛点，是目前实现近乎完美科学上网的主流全局代理技术 [5]。
- **关联概念**：依赖 → [[虚拟网卡]]

### [[虚拟网卡]]
- **定义**：由代理软件在操作系统中创建的非物理网络接口，用于在网络层接收所有应用产生的网络数据 [5, 7]。
- **视频上下文**：在开启 TUN 模式后，代理软件会生成一张虚拟网卡，电脑的网络数据会被重定向发给它，从而让代理软件读取和解析数据包 [5-7]。
- **为什么重要**：它是 TUN/TAP 模式和软路由实现底层流量拦截与接管的核心机制 [5, 8]。
- **关联概念**：属于 → [[TUN/TAP 模式]]

### [[路由表]]
- **定义**：操作系统中决定网络数据包应该发送给哪个网卡接口（物理网卡或虚拟网卡）的规则列表 [7]。
- **视频上下文**：作者通过 `route print` 命令展示，TUN 模式会自动在电脑中添加路由条目，使得访问任何 IP 的数据都会匹配并走向虚拟网卡 [7]。
- **为什么重要**：它是成功将系统全局流量劫持、重定向到代理软件的关键底层配置 [6, 7]。
- **关联概念**：未明确提及

### [[透明代理]]
- **定义**：将代理的虚拟网卡转移到局域网网关（路由器）上的技术模式 [8]。
- **视频上下文**：为了防止严格的程序或游戏检测出电脑开启了虚拟网卡，将代理运行在路由器上，使得局域网内所有设备的流量必定经过网关并被接管处理 [8]。
- **为什么重要**：代理过程对终端设备完全透明，既能防止反作弊检测，又能实现全家设备免配置科学上网 [8]。
- **关联概念**：扩展 → [[TUN/TAP 模式]]

### [[真 VPN]]
- **定义**：能够封装网络层数据包（如 ICMP 协议），用于实现异地组网和内网穿透的虚拟专用网络技术 [9, 10]。
- **视频上下文**：作者指出普通翻墙协议（如 SS、Vmess）无法代理 ping 这种网络层工具，会返回假延迟；而真 VPN（如 WireGuard）可以封装网络层，返回真实延迟并实现虚拟内网通信 [9, 11]。
- **为什么重要**：纠正了大众将“翻墙”等同于“VPN”的误区，指出真 VPN 的设计初衷并非为了隐藏流量特征翻墙，而是为了私有网络的互联 [9, 10]。
- **关联概念**：对比 → [[TUN/TAP 模式]]

### [[网络协议栈]]
- **定义**：操作系统中对网络数据包进行逐层封装（从应用层到传输层、网络层、接口层）和解封装的通信处理机制 [3, 12]。
- **视频上下文**：作者以 TCP/IP 四层模型为例，详细拆解了浏览器访问百度的请求是如何一层层加上 HTTP 头部、TCP 端口、IP 地址并最终发出的过程 [3, 12]。
- **为什么重要**：理解数据包在操作系统中的逐层流转，是深入理解不同代理模式（如网络层接管）工作原理的必要前置知识 [5, 12]。
- **关联概念**：未明确提及

### [[分流规则]]
- **定义**：代理软件截获数据后，用于判断该数据是需要被加密代理还是直接转发的依据 [4, 6]。
- **视频上下文**：无论是系统代理还是 TUN 模式，代理客户端（如 Clash）收到数据包后，都会先根据分流规则进行判定，再决定是否将请求交给代理服务器处理 [4, 6]。
