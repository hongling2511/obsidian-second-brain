## 一句话总结
本教程演示如何通过 [[Caddy]] 配合 [[naiveproxy]] 搭建完全隐藏 [[TLS]] 指纹的极高隐蔽性科学上网节点。

## 目标受众与前置知识
- **适合谁看**：需要建立高隐蔽性、防止防火墙封锁的自建节点用户（推断）。
- **前置知识**：[[VPS]] 购买与连接、[[域名]]注册与 [[DNS]] 解析（需参考讲者前置保姆级教程前 10 分钟 25 秒，讲者原话）。

## 核心观点
1. **[[naiveproxy]] 是目前最完美的自建节点隐藏方案**
   - 为什么重要：能够将客户端和服务端一切可能暴露翻墙的特征全部隐藏起来。
   - 依据：讲者原话提及该协议使用 [[Chrome]] 浏览器内核作为网络协议栈，消除客户端 [[TLS]] 指纹和 [[TLS in TLS]] 特征；且基于 [[HTTP/2]] 具有自带多路复用的低延迟优势。
2. **服务端指纹（[[JARM]]）暴露会导致节点被轻易识别**
   - 为什么重要：防火墙不只看伪装网站的表面内容，还能通过网络交互特征精准识别代理服务。
   - 依据：讲者原话演示使用 [[Python]] 探针脚本探测 [[x-ui]](xui) 的 [[Xray]](插瑞) 节点，通过特有指纹仅搜索出 6000 多条匹配结果（讲者原话），暴露出小众代理特征。
3. **真正的伪装不仅看表面，还要从网络行为上伪装成正常网站**
   - 为什么重要：常规的 [[WebSocket]](ws) 回落伪装或 [[Trojan]](秋卷) 默认伪装易被识别。
   - 依据：讲者原话建议使用 [[Caddy]] 或 [[Nginx]](ang) 进行伪装，探测讲者搭建的内务节点指纹在搜索引擎能匹配到 500 多万条（讲者原话）正常网站结果。

## 概念关系
- [[Chrome]] 内核 → [[naiveproxy]](内务)：依赖说明，由于内务基于 Chrome 内核，因此消除了普通代理软件产生的客户端 [[TLS]] 指纹，但代价是不支持 [[v2rayA]](v2a)、[[clash]] 或 [[sing-box]](sbox) 等常规代理客户端。
- [[Caddy]](cadyy) → 服务端伪装：依赖说明，内务的服务端实质上是加了插件的 [[Caddy]]，负责管理网站证书和进行 Web 行为层面的伪装。
- 客户端参数集 → [[JA3]](j3)：因果说明，通过客户端发送的 Client Hello 数据包中的 TLS 版本（如 771）、[[AES-128]](as128)、[[AES-256]](as256) 等加密套件参数，经 [[md5]] 哈希计算得出客户端 TLS 指纹。
- 主动探针 → [[JARM]](jum)：因果说明，通过多次提交不同参数进行探测，根据服务端规律性回应计算出的哈希值即为服务端指纹，用于识别服务端使用的是 [[Nginx]]、[[Caddy]] 还是代理工具。

## 详细笔记
### 痛点引入与 [[naiveproxy]] 的优势
讲者 [[不良林]](布良玲) 提到之前的教程配置繁琐导致新手易错。本期带来基于 [[naiveproxy]](内务) 的终极解决方案。它的原理类似 [[Trojan]](秋卷)，但进一步消除了客户端 [[TLS]] 指纹和 [[TLS in TLS]]（两层 TLS 握手）特征。缺点是因使用 [[Chrome]] 内核，常规软件如 [[v2rayA]](v2a)、[[clash]]、[[sing-box]](sbox) 甚至 [[Shadowsocks]](sswem) 都无法直接兼容。

### 前置准备与服务器推荐
搭建需提前搞定 [[VPS]] 和 [[DNS]] 解析。若不想折腾且预算充足，讲者推荐使用 [[搬瓦工]](班网公) 的三网 [[CN2 GIA]](cn2j) 线路（推断）。讲者演示环境为 [[Ubuntu 22.04 amd64]](w班图22.04amd64位) 系统。

### 服务端编译与搭建配置
[[naiveproxy]] 服务端是带插件的 [[Caddy]](CY/CAD)。步骤如下：
1. 安装 [[Go语言]](购语言) 编译环境。
2. 下载所需文件并编译，约需 5 分钟（讲者原话），生成 `caddy` 文件。
3. 同目录下新建名为 `Caddyfile` 的配置文件（注意首字母大写）。
4. 修改配置文件中的四处：[[域名]]、管理证书邮箱、用空格隔开的用户名和密码、伪装网址。
5. 使用 `caddy run` 启动（前台尝试申请证书，需确保 80 和 443 端口开放），成功后用 `caddy start` 后台运行。

### 客户端配置指南
1. **Windows 电脑**：访问官方下载与当前浏览器内核一致的 [[Windows x64]] 版本（当前演示版本为 105，讲者原话）。因只有命令行，推荐配合 [[v2rayN]](v2n/wn) 使用：将内核文件 `naive.exe` 放入其目录，修改 `config.json`（监听 1080 端口，填写账号密码用逗号隔开）。在客户端添加“自定义配置”，内核选 `naiveproxy`，结合前置 Socks 代理实现分流。
2. **苹果 iOS**：使用 [[Shadowrocket]](小火箭)，协议选 [[HTTPS]] 或 [[HTTP/2]](http2)。
3. **安卓**：使用 [[SagerNet]](cgnet)，需额外安装内务插件。验证节点时可尝试访问 [[Google]](谷歌)。

### [[TLS]] 指纹与网络抓包解析
数据传输时会产生 [[TLS]] 指纹。通过抓包可见，[[naiveproxy]] 发出的 [[JA3]](j3) 指纹与直接用 [[Chrome]] 浏览器访问 [[Baidu]](百度) 时生成的指纹完全一致（如尾号 3da9），而换用 [[Edge]](ag) 浏览器指纹就会改变。服务端的正常回复产生 [[JA3S]](j3s) 指纹，但通常无参考价值。

### 服务端指纹（[[JARM]]）探测对比实战
讲者下载探针脚本并用 [[Python]](pyon) 运行：
- 探测内务节点：得到 [[JARM]](jum/Jam) 指纹，在特定搜索引擎搜索发现 500 多万条结果，与正常 [[Caddy]] 网站特征一致，完美隐藏。
- 探测常规节点：对利用 [[x-ui]](xui) 安装的老版本 1.4.2 的 [[Xray]](插瑞) 内核（未用前端伪装）的 [[WebSocket]](ws/wens) 节点在 4443 端口进行探测。搜索其指纹仅 6000 多条结果，明显暴露出代理服务。部分节点使用 2052 / 2096 端口是为了套用 [[Cloudflare]](CF) CDN 回源。这些明显特征若被防火墙捕捉，IP 很容易被封禁。

## 常见坑与解决方案
- **问题描述**：[[Caddy]] 自动申请证书失败（一直停留在申请阶段）。
  - 解决方案：首先确保域名的 DNS 解析已经生效，另外必须确保 80 和 443 端口正常开放，并且没有被其他进程占用 [9]。
- **问题描述**：断开与 VPS 的连接后，节点服务停止工作。
  - 解决方案：因为服务端处于前台运行模式，应当按 Ctrl+C 关闭后，使用后台启动命令（`caddy start`）重新运行服务 [3, 9]。

## 最佳实践
- 强烈建议将默认的节点用户名和密码修改得复杂一点，以保证安全性 [9]。
- 搭建安全的节点时，务必使用 Web 伪装（如本教程使用带有 [[Caddy]] 或 [[Nginx]] 行为的伪装），使得不仅从表面内容上，而且在 TLS 指纹行为特征上都像是一个正常的网站，以防止被防火墙通过指纹探测精准封锁 [11]。
- 如果不愿意在自建线路上花费太多时间折腾，也不差钱，推荐直接使用高质量线路（例如视频提及的搬瓦工三网 CN2 GIA 线路），以一步到位实现高速稳定的科学上网 [2]。

## 关键概念

### [[naive协议]]
- **定义**：一种使用 Chrome 浏览器内核作为网络协议栈的代理协议，可将客户端和服务端可能暴露翻墙的特征全部隐藏 [1]。
- **视频上下文**：作者将其作为当前自建节点最完美的隐藏方案推荐给观众，并演示了如何从零搭建该节点 [1, 2]。
- **为什么重要**：它能消除客户端的 tls 指纹和 tls in tls 特征，从防火墙角度看就像正常使用谷歌浏览器访问正常网站，极大地提升了隐蔽性 [3]。
- **关联概念**：对比 → [[tls in tls]]

### [[tls指纹]]
- **定义**：在建立 TLS 连接的握手协商过程中，通过客户端支持的加密参数或服务端返回的参数生成的哈希值 [4]。
- **视频上下文**：作者通过实际抓包演示说明不同软件会产生不同的哈希值，以此解释防火墙如何识别代理软件 [4, 5]。
- **为什么重要**：如果没有隐藏服务端的 tls 指纹，节点将会被防火墙精准识别并封锁 [3, 6]。
- **关联概念**：扩展 → [[ja3]]

### [[tls in tls]]
- **定义**：指代理节点设置了 TLS，且通过该节点访问的目标网站也存在 TLS，从而产生两层 TLS 握手特征的现象 [3]。
- **视频上下文**：作者指出常规安全的代理节点仍存在此特征，而 naive 协议通过自带填充消除了这种两层握手的特征 [3]。
- **为什么重要**：它是传统代理协议中容易产生的流量特征，消除它可以进一步保障代理流量的安全 [3]。
- **关联概念**：未明确提及

### [[caddy]]
- **定义**：一个使用 Go 语言编写的类似 Nginx 的中间件，本视频中作为 naive 节点的服务端程序 [7]。
- **视频上下文**：作者演示了如何通过命令编译带有 naive 插件的 caddy，并通过其简短的配置文件实现代理伪装和网站证书的自动管理 [7, 8]。
- **为什么重要**：它的配置比 Nginx 简单很多，是实现节点 Web 伪装和消除服务端指纹特征的核心组件 [2, 7]。
- **关联概念**：扩展 → [[伪装]]

### [[ja3]]
- **定义**：客户端发送 hello 数据包时，通过其包含的 TLS 版本、加密套件等参数计算得出的哈希值，即客户端的 tls 指纹 [4]。
- **视频上下文**：作者通过抓包工具展示，由于 naive 使用 Chrome 网络协议栈，其产生的 ja3 指纹与真实 Chrome 浏览器的指纹是完全相同的 [9]。
- **为什么重要**：它是暴露客户端所用网络协议栈身份的指标，但因为参数重复率高，单靠它较难精准封杀代理软件 [5]。
- **关联概念**：属于 → [[tls指纹]]

### [[ja3s]]
- **定义**：服务器端返回给客户端的 hello 数据包中，通过包含的版本号和选中加密方式等信息计算得到的 tls 指纹 [4]。
- **视频上下文**：作者在抓包演示中指出，由于生成该哈希值的字符串很短，导致哈希值重复率极高 [9]。
- **为什么重要**：在实际的防火墙特征分析中没有太多参考价值，需借助主动探测的 jarm 指纹来代替判断 [9]。
- **关联概念**：属于 → [[tls指纹]]

### [[jarm]]
- **定义**：客户端通过每次提交不同参数对服务器进行 10 次探测后，根据服务器有规律的回应计算出的一个总的哈希值 [5, 6]。
- **视频上下文**：作者使用 Python 脚本获取了 naive 节点的 jarm 指纹，并在搜索引擎中检索出大量正常的 caddy 网站，证明了伪装的有效性 [6, 9]。
- **为什么重要**：这种经过多次探测得到的哈希值比较靠谱，防火墙能借此基本断定服务端使用的是哪种代理程序或 Web 容器 [5]。
- **关联概念**：属于 → [[tls指纹]]

### [[伪装]]
