# Go 1.26's Small But Clever New Function Change!

## 一句话总结
[[Go 1.26]] 允许 `[[new]]` 函数直接接收表达式并一步返回指针，简化了可选值和测试代码的编写。

## 目标受众与前置知识
- **适合谁看**：[[Go]] 工程师（讲者原话）[1]。
- **前置知识**：[[Go]] 基础语法、[[pointer]]（指针）、[[struct]]（结构体）、[[JSON]] 序列化、[[单元测试]]。

## 核心观点
1. **[[Go 1.26]] 的 `[[new]]` 函数支持接收表达式并一步完成初始化**
   - 为什么重要：省去了过去必须先用 `[[new]]` 分配内存再进行解引用（[[dereference]]）赋值的繁琐步骤。
   - 依据：讲者对比了更新前后为一个整型指针赋值为 42 的两段代码（屏幕展示）[1-3]。
2. **使用 [[pointer]] 字段可以有效区分 [[JSON]] 或 [[Protobuf]] 中的“零值”和“缺失值”**
   - 为什么重要：如果没有指针，缺失的字段将默认为零值，这可能会在更新操作中意外擦除用户的真实数据。
   - 依据：讲者通过一个处理 HTTP [[PATCH]] 方法的 `UpdateSettingsParams` 结构体演示了如何校验 `retry limit` 和 `is active` 字段是否为 `nil`（讲者演示）[4, 5]。
3. **新版 `[[new]]` 函数能显著减少复杂单元测试中的视觉噪音和代码输入量**
   - 为什么重要：避免了仅仅为了向结构体参数传递指针地址，而被迫创建无用临时辅助变量的做法。
   - 依据：讲者在表格驱动测试（table-driven tests）的案例中，将原本需要预先声明的 `zero` 和 `falseVal` 变量替换成了更为直接的 `new(0)` 和 `new(false)`（屏幕展示）[6, 7]。

## 概念关系
- [[Go 1.26]] → `[[new]]`：[[Go 1.26]] 对 `[[new]]` 函数的功能进行了扩充，使其能够直接接收表达式 [2]。
- `[[new]]` ↔ [[dereference]]：在旧版本中，使用 `[[new]]` 分配内存后必须配合 [[dereference]] 才能赋初始值；新版本传入表达式后，在初始化阶段消除了显式的解引用需求 [2, 3]。
- [[pointer]] → 可选值：在定义 [[struct]] 时，利用 [[pointer]] 的 `nil` 状态，可以判断调用方是完全没有传递该字段，还是传递了确切的零值 [4, 5]。

## 详细笔记
### [[Go 1.26]] 发布前瞻与旧版 `[[new]]` 函数回顾
- [[Go 1.26]] 将于 2 月份发布，其中包含一项对 `[[new]]` 函数的小巧但优雅的改动 [1]。
- 在 Go 1.26 之前，`[[new]]` 函数（例如 `new(int)`）的作用仅仅是在内存中为某种数据类型寻找一个空闲位置，并用该类型的零值（如整数为 0）进行初始化 [1, 2]。
- 此时变量仅保存内存地址，如果想要赋予特定值（如 42），开发者必须对指针进行解引用（[[dereference]]，即 `*p = 42`）来修改该地址指向的内存数据 [2]。

### [[Go 1.26]] 中 `[[new]]` 函数的新特性
- Go 1.26 允许 `[[new]]` 函数直接接收一个表达式（如 `new(42)`）[2]。
- 内部机制：[[Go compiler]]（编译器）会将 42 识别为整数表达式，随后一步到位地完成内存创建、填充数值 42，并将指针返回给开发者 [3]。

### 应用场景一：API 请求中的局部更新（Partial Updates）
- 在反序列化 [[JSON]] 或 [[Protobuf]] 数据时，经常需要处理可选值 [3]。
- 讲者演示了一个名为 `User` 的数据库记录表示形式，以及用于 HTTP [[PATCH]] 请求的 `UpdateSettingsParams` 结构体 [4, 5]。
- 如果结构体内的 `retry limit`（重试次数）和 `is active`（活跃标识）不使用指针，未传参的字段会变成默认的 0 或 false，从而可能擦除用户原本的数据 [4, 5]。
- 讲者设置了一个名为 [[Alice]] 的假用户（Dummy User）进行演示。在 Go 1.26 之前，如果要测试更新她的 `retry limit`，为了获取合法的内存地址传给指针字段，必须强行声明一个新的临时变量（值为 20）[8, 9]。
- 在 Go 1.26 中，可以直接在初始化结构体时写 `new(20)` [10]。

### 应用场景二：简化表格驱动测试（Table-driven Tests）
- 在编写复杂的 [[单元测试]]（包含 `name`、`initial`、`updates`、`expected` 等字段）时，需要初始化大量的结构体 [11]。
- Go 1.26 之前，为了给指针类型的更新参数赋值，开发者不得不创建名为 `zero` (值为 0) 和 `falseVal` (值为 false) 的辅助临时变量 [6]。
- 使用 Go 1.26，开发者只需在结构体内写入 `new(0)` 和 `new(false)`，测试结果保持一致，但这大大减少了视觉噪音并节省了输入成本 [7]。

### 频道结语与后续推荐
- 讲者总结了 `[[new]]` 函数改动带来的代码清理效果 [12]。
- 频道 [[Flo Woelki]] （推断）在视频末尾推荐了另一期关于 [[Go]] 语言中空结构体（empty struct）含义的视频 [12]。

## 步骤拆解
1. **使用旧版本 Go 演示内存分配与赋值**
   - 具体操作描述：在 Go 1.26 之前，先使用 `new` 函数为指定类型（如 integer）分配一个内存空位并初始化为零值，随后通过解引用该指针来赋实际值 [3, 6]。
   - 关键代码/命令：`new(int)` 分配内存，`*p = 42` 进行解引用赋值 [3, 6]。
   - 注意事项：旧版本的 `new` 函数仅返回指向零值的内存地址，必须额外执行解引用步骤才能完成设值 [3, 6]。

2. **在 Go 1.26 中直接使用表达式初始化指针**
   - 具体操作描述：直接将表达式传递给 `new` 函数，由编译器自动识别类型、分配内存并使用该表达式的值进行初始化，一步返回目标指针 [1, 3]。
   - 关键代码/命令：`new(42)` 或 `new(20)` [1, 2]。
   - 注意事项：编译器会将传入的数值作为表达式处理，省去了单独解引用赋值的步骤 [1]。

3. **构建支持部分更新（PATCH）的 API 结构体**
   - 具体操作描述：声明 `User` 与 `UpdateSettingsParams` 结构体，并将需要通过 [[HTTP]] PATCH 方法更新的可选字段（如重试限制、激活状态）声明为指针类型 [1, 4, 7]。编写 `PatchUser` 函数，通过判断字段指针是否为 `nil` 来决定是否执行更新操作 [7, 8]。
   - 关键代码/命令：在字段后添加 `omitempty` 标签，例如 `json:"retry_limit,omitempty"` [4]；更新判断逻辑 `if updates.RetryLimit != nil` [7]。
   - 注意事项：在更新真实值之前，应添加验证逻辑（例如防止重试次数被更新为负数）[8]。

4. **为 API 更新逻辑编写表格驱动测试**
   - 具体操作描述：在 `main_test.go` 中引入 `testing` 包，定义包含初始输入、更新参数和预期输出的测试用例列表 [2, 5]。使用 `for` 循环与子测试来逐一验证逻辑 [9]。
   - 关键代码/命令：使用 Go 1.26 语法直接初始化测试数据字段，如 `new(0)` 和 `new(false)` [10]。
   - 注意事项：测试中的结构体直接比较（`user != tc.expected`）仅在结构体不包含不可比较类型（如 slices、maps 或 functions）时才是安全的 [10]。

## 常见坑与解决方案
- **问题描述**：在处理网络请求的结构体时，如果不使用指针而使用常规值，缺失的字段会默认回退到零值（如 0 或 false），这可能会意外擦除用户的现有数据 [4, 7]。
  - 解决方案：将结构体中的字段声明为指针类型，以此来严格区分“用户意图设置为零值”和“字段缺失（nil）” [4, 7]。

- **问题描述**：在 Go 1.26 之前的版本中，为了给结构体的指针字段赋值（例如在编写单元测试或大型 API 客户端时），必须被迫创建大量临时/辅助变量来获取内存地址，这增加了显著的代码视觉噪音 [2, 9, 11]。
  - 解决方案：使用 Go 1.26 的特性，直接向 `new` 函数传递原始值表达式（如 `new(20)` 或 `new(false)`），从而免去创建临时变量的需要 [2, 10]。

## 最佳实践
- 在设计需要处理部分更新（partial updates）的网络调用时，推荐使用指针结合 [[JSON]] 标签来区分数据状态 [4, 7]。
- 在编写复杂的单元测试或初始化包含大量字段的大型 API 客户端结构体时，推荐使用 Go 1.26 的 `new` 表达式特性，以减少样板代码和视觉噪音 [2]。
- 编写测试时推荐使用表格驱动测试（table-driven tests）结合子测试，这样即使某一个子测试失败，也不会阻断其他测试用例的运行 [5, 9]。
- 在对数据库记录或结构体执行实际更新赋值之前，推荐先对各个单独字段执行数据校验检查 [8]。

## 关键概念

### [[new 函数]]
- **定义**：用于在内存中为指定类型寻找空闲位置、分配内存并初始化的内置函数 [1]。
- **视频上下文**：在 Go 1.26 之前，它只能接受类型并将其初始化为零值；而在 Go 1.26 中，它可以直接接受一个表达式（如 `new(42)`），在一步内完成内存分配、填入该值并返回指针 [1-3]。
- **为什么重要**：Go 1.26 对此函数的改进有效避免了为获取指针而创建不必要的临时变量，显著减少了视觉噪音和代码量 [4, 5]。
- **关联概念**：依赖 → [[表达式]]

### [[指针]]
- **定义**：指向数字或数据在内存中存储地址的变量，而不是数据数值本身 [2]。
- **视频上下文**：在表示可选值（如解析 JSON 或 Protobuf 时）时作为结构体字段的类型使用，以便区分调用者是明确提供了一个值，还是根本没有提供值 [3, 6]。
- **为什么重要**：如果没有指针，缺失的字段会默认变为零值，这可能会在更新记录时意外清空用户的现有数据 [6, 7]。
- **关联概念**：未明确提及

### [[解引用]]
- **定义**：一种不查看内存地址，而是前往指针指向的内存地址去查看或存储实际数值的操作 [2]。
- **视频上下文**：在 Go 1.26 之前，需要先通过 `new` 分配指针变量，然后使用解引用操作（如赋予 `42`）将值存入该内存地址 [2]。在更新逻辑中，也需要解引用指针以获取实际的整数或布尔值来赋给目标对象 [8]。
- **为什么重要**：在通过指针读写底层数据时，这是必不可少的操作步骤 [2, 8]。
- **关联概念**：依赖 → [[指针]]

### [[表达式]]
- **定义**：一段可以产生一个具体值的代码（例如数字 `42` 或布尔值 `false`） [2, 3]。
- **视频上下文**：Go 1.26 的编译器使得 `new` 函数现在可以接收表达式，编译器识别其类型后，会直接创建内存并填入该表达式的值 [2, 3]。
- **为什么重要**：使得在复杂单元测试或大型 API 客户端初始化结构体时，能够直接内联取址（如 `new(20)`、`new(false)`），彻底清理了不需要的临时变量 [5, 9, 10]。
- **关联概念**：未明确提及

### [[零值]]
- **定义**：数据类型在未被显式赋值时的默认数值（例如整型的 `0`） [1, 2]。
- **视频上下文**：作者强调必须在 API 请求中区分零值（例如明确将重试限制设置为 0）和未传递的缺失值 [6]。
- **为什么重要**：如果无法区分零值和缺失值，默认的零值可能会在数据库更新操作中意外擦除用户的有效数据 [7]。
- **关联概念**：对比 ↔ [[缺失值]]

### [[缺失值]]
- **定义**：表示在请求或操作中未提供特定字段的状态，在代码中通过指针的 `nil` 状态来体现 [6, 7]。
- **视频上下文**：在结构体和部分更新逻辑中，如果传入字段是 `nil`，说明调用者并不想更改或提供该字段 [7, 8]。
- **为什么重要**：利用缺失值的判定配合指针，可以实现安全的 PATCH HTTP 方法，防止误修改未提供的字段 [7]。
- **关联概念**：对比 ↔ [[零值]]

### [[部分更新]]
- **定义**：一种仅更改 HTTP PATCH 请求中提供的（非 `nil`）字段，而不影响其他保留数据的操作模式 [7]。
- **视频上下文**：视频演示了一个 `patch user` 函数，通过检查传入的参数指针是否为 `nil`，来决定是否修改用户的特定字段（如重试限制或激活状态） [7]。
- **为什么重要**：展示了在网络请求和 API 开发中利用指针处理可选参数的安全更新机制 [7]。
- **关联概念**：依赖 → [[缺失值]]

### [[表格驱动测试]]
- **定义**：不为每个测试用例单独写独立代码，而是定义一个包含多个测试用例（含名称、初始输入、更新内容和预期输出）的列表并循环验证的测试方法 [11]。
- **视频上下文**：作者在使用 Go 1.26 新特性编写测试时使用了这种模式，通过循环配合子测试来验证用户的全部更新和部分更新逻辑 [11, 12]。
- **为什么重要**：通过子测试执行，即使某一个测试用例失败，也不会阻止列表中其他测试用例的继续运行，提高了测试的稳健性 [12]。
- **关联概念**：未明确提及
