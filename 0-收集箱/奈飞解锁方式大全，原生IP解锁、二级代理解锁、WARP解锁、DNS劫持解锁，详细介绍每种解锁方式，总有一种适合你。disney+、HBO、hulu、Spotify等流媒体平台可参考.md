## 一句话总结
详细演示了通过 [[原生IP]]、[[二级代理]]、[[WARP]] 和 [[DNS]] 劫持四种方案解锁 [[奈飞]] 非自制剧的原理与配置步骤。

## 目标受众与前置知识
- **适合谁看**：拥有自建 [[VPS]] 节点，但受限于 IP 被封禁而无法观看 [[奈飞]] 非自制剧的流媒体用户。
- **前置知识**：[[VPS]] 基础命令行操作、[[Xray]] / [[v2rayN]] 节点搭建与配置、[[DNS]] 基础通信原理。

## 核心观点
1. **原生IP解锁最简单但成本与风险并存**
   - 为什么重要：是最直接的方案，但具有商业欺骗陷阱。
   - 依据：“有些商家通过DNS解锁来冒充所谓的原生IP”（讲者原话），且被奈飞拉黑后可能不予更换。
2. **二级代理能完美结合速度与解锁能力**
   - 为什么重要：解决了大部分用户“解锁节点速度慢、直连节点不解锁”的两难痛点。
   - 依据：讲者通过演示将流量分流，把不解锁节点的流媒体流量单独转交给速度慢但能解锁的韩国节点，成功实现流畅播放（屏幕展示）。
3. **WARP是目前唯一的免费解锁方案**
   - 为什么重要：零成本获取能够解锁流媒体的纯净 IP。
   - 依据：利用 [[Cloudflare]] 提供的免费 [[VPN]] 服务生成代理 IP，视频中实测该代理 IP 可完整解锁新加坡区奈飞（屏幕展示）。
4. **DNS劫持解锁适合机场主与无缝切换**
   - 为什么重要：节点服务器无需设置复杂的代理分流，像原生解锁一样自然。
   - 依据：讲者原话解释原理并演示通过搭建 [[SNI Proxy]] 服务，配合 DNS 域名劫持直接欺骗流媒体平台。

## 概念关系
- [[WARP]] → [[SOCKS]]：依赖关系（推断）。必须将 WARP 运行模式切换为 SOCKS 代理模式，以防止其接管全局网络导致 [[VPS]] 失联。
- [[分流配置]] → [[二级代理]]：依赖关系。在 [[Xray]] 中通过设置 Routing 规则，让指定流媒体域名的流量走专属的代理出口。
- [[DNS]] ↔ [[SNI Proxy]]：协作关系（推断）。DNS 负责将域名解析导向自定义的解锁机 IP，而 SNI 负责将收到的请求反向代理转发给真实的流媒体服务器。
- [[原生IP]] ↔ [[WARP]]：对比关系（推断）。前者是付费购买商业服务器自带的未封禁 IP，后者是白嫖官方免费分配的动态网络 IP。

## 详细笔记
### 节点搭建与解锁检测
- 讲者 [[不良林]] 演示在 [[W]] 商家购买的新加坡 [[VPS]]（搭载 [[Ubuntu]] 2.04 操作系统，讲者原话），关闭防火墙（指令 `[[UFW]] Disable`）。
- 通过 IP 加上 54321 端口（讲者原话）进入 [[i面板]]（推断为 `x-ui`），创建一个 [[WM]] 加 [[ws]] 的协议节点，并将其导入到 [[v2rayN]]（字幕记为“微 2n”）。
- 网页访问 [[奈飞]]（[[Netflix]] / [[NF]]）直接返回 403 错误（屏幕展示），随后使用 `./nf` 检测工具验证该节点无法解锁非自制剧。

### 方案一：[[原生IP]]
- **概念**：直接使用本身未被奈飞拉黑的 [[VPS]] 服务器。
- **避坑**：价格较贵，不仅不保证网速，且商家可能使用 [[DNS]] 冒充原生 IP，一旦被拉黑未必提供更换服务。

### 方案二：[[二级代理]]（前置代理）
- **原理**：将用户发往不解锁节点的流媒体流量，再转发给另一个支持解锁的节点处理。
- **配置步骤**：在支持代码高亮的编辑器中修改 [[Xray]]（字幕记为“插瑞”）客户端配置。
  1. 修引入站：设置一个本地的 [[SOCKS]]（字幕记为“sox”/“Soxs”）端⼝，设为 30000（讲者原话）。
  2. Routing 分流：插入路由规则，将发往 [[Netflix]]、[[Disney+]]（字幕记为“迪士尼”）、[[HBO]]、[[hulu]]（字幕记为“葫芦”）、[[Spotify]] 等平台的流量导向 Tag 为 `Netflix Proxy` 的出口。
  3. 导入出口：将另一个能解锁的韩国节点配置复制并插入到 `out` 数组中。保存并重启面板，实测解除限制。

### 方案三：[[WARP]] 免费解锁
- **原理**：[[WARP]]（字幕记为“WP”/“WB”）是 [[Cloudflare]]（字幕记为“Cloud Flyer” / [[CF]]）提供的免费 [[VPN]]，将其作为出口代理即可。
- **配置步骤**：
  1. 若官方客户端不支持，可使用 [[WireGuard]]（字幕记为“Wgard” / [[WG]]）网络接口模式（推断需自行查阅教程）。
  2. 官方客户端演示：先执行 `[[GPG]]`（字幕记为“JPJ”）密钥验证，接着安装并注册客户端。
  3. **必须**将其运行模式切换为 [[SOCKS]] 代理，默认端口 40000（讲者原话）。
  4. 测速确认：分配的 IP 可完整解锁 [[IPv4]]，而 [[IPv6]] 只能看自制剧。将 [[Xray]] 配置文件中的出站代理换为 WARP 的 SOCKS 端口即可。

### 方案四：[[DNS]] 劫持解锁
- **原理**：利用类似 [[GFW]] 的劫持机制，将流媒体域名的解析指向自己的解锁机，解锁机上的 [[SNI Proxy]]（[[SNI]]）再进行反向代理通信。
- **配置步骤**：
  1. 在解锁机上使用一键脚本安装 [[SNI Proxy]] 服务（参数 `-gofs`）。
  2. 停用本地默认的 [[DDNS]] 服务占用，接着安装 [[DNS]] 服务（参数 `-id`）。
  3. 将不可解锁的节点服务器的 DNS 指向该解锁机 IP，使用 `[[NSLOOKUP]]`（字幕记为“NSLUP”）命令验证域名已成功指向对应 IP。
  4. 需通过指令禁用 [[IPv6]] 以防优先匹配失败。
- **补充**：网上有商家出售现成的 DNS 解锁服务，每月大概 10 来块钱（讲者原话），适合图方便的用户。

## 金句亮点
- “有些商家通过DNS解锁来冒充所谓的原生IP，这些坑也要注意一下。” — 避坑指南 / 经验总结

## 步骤拆解

1. **步骤名称：VPS 基础节点初始化**
   - 具体操作描述：在新开的 [[VPS]] 上安装节点面板，关闭防火墙，通过 IP+端口 登录面板并创建新节点进行连通性测试 [1, 3]。
   - 关键代码/命令：关闭防火墙命令 `ufw disable`（视频口述为 UFWDable 将防火墙关）；面板默认端口 `54321` [1]。
   - 注意事项：测试连通性后需使用流媒体解锁检测脚本验证，确认当前节点无法解锁奈飞 [3]。

2. **步骤名称：二级代理（前置代理）解锁配置**
   - 具体操作描述：将本地发往奈飞的流量，通过不支持解锁的节点转交给另一台支持解锁的节点。需要从 [[v2rayN]] 导出支持解锁节点的客户端配置，并在 [[Xray]] 面板的模板配置中修改入站 (inbounds)、路由 (routing) 和出站 (outbounds) 设置 [4-6]。
   - 关键代码/命令：
     - 入站：修改配置使 [[SOCKS]] 端口设置为 `30000` [5]。
     - 路由分流：在 routing 位置添加规则，使流媒体域名组流量走特定出站 tag（如 `Netflix Proxy`）[5]。
     - 出站：在 out 出口位置插入解锁节点的配置，并将其 tag 替换为前一步设置的 `Netflix Proxy` [6]。
   - 注意事项：在记事本中复制粘贴代码时，必须看清插入位置的大括号范围，并在结尾加上英文逗号；修改完毕后必须点击“重启面板”使配置生效 [5, 6]。

3. **步骤名称：[[WARP]] 客户端解锁配置**
   - 具体操作描述：在 [[VPS]] 上安装官方 [[WARP]] 客户端，注册账号，将运行模式切换为 [[SOCKS]] 代理模式后进行连接。然后通过修改 [[Xray]] 的出站代理指向 [[WARP]] 提供的本地 [[SOCKS]] 端口 [2, 7, 8]。
   - 关键代码/命令：
     - 更新源：`update` [7]。
     - 默认 [[SOCKS]] 代理端口为 `40000` [8]。
     - 具体的安装、注册、模式切换等完整执行命令：未明确提及（视频中以“执行这条指令”代指） [7, 8]。
   - 注意事项：连接 [[WARP]] 前**必须**先切换为 [[SOCKS]] 代理模式，否则默认模式会接管整个网络流量导致 [[VPS]] 失联 [8]。解锁时只需关注 IPv4，IPv6 状态不用管 [8]。

4. **步骤名称：[[DNS]] 劫持解锁配置**
   - 具体操作描述：在一台能够解锁奈飞的 [[VPS]] 上安装 [[SNI Proxy]] 工具进行反向代理，并在同一台或任意 [[VPS]] 上搭建 [[DNS]] 服务。最后将节点服务器的 [[DNS]] 地址修改为搭建了 [[DNS]] 服务的 IP 地址 [9, 10]。
   - 关键代码/命令：
     - 安装 [[SNI Proxy]] 附加参数：`gofs` [9]。
     - 安装 [[DNS]] 服务附加参数：`-ID` [10]。
     - 验证 [[DNS]] 指向：`nslookup netflix.com` [10]。
     - 完整的安装脚本执行命令：未明确提及 [9]。
   - 注意事项：安装 [[DNS]] 服务前，需停用 [[Ubuntu]] 本地的默认 [[DNS]] 服务以释放 53 号端口；如果是同一台 [[VPS]]，[[DNS]] 和 [[SNI Proxy]] 的 IP 地址一致 [10, 11]。

## 常见坑与解决方案

- **问题描述**：购买的“原生 IP”被奈飞拉黑无法解锁。
  - 解决方案：购买前查看 [[VPS]] 商家服务条款，确认是否有“明确说明保证解锁奈飞”的条款，否则不予更换 IP [3]。
- **问题描述**：修改二级代理配置文件后，节点运行失败/无法使用。
  - 解决方案：仔细检查配置文件大括号结束位置，确认代码段是否漏删或多删，确认是否漏加了英文状态下的逗号，且保存后必须重启面板 [5, 6]。
- **问题描述**：连接 [[WARP]] 后导致 [[VPS]] 服务器失联。
  - 解决方案：在执行连接 [[WARP]] 命令之前，必须先将其运行模式从默认模式切换为 [[SOCKS]] 代理模式 [8]。
- **问题描述**：安装 [[DNS]] 服务时提示 53 号端口被占用。
  - 解决方案：执行相关指令关闭 [[Ubuntu]] 本地的默认 [[DNS]] 服务，再重新运行安装脚本 [10]。
- **问题描述**：完成 [[DNS]] 解锁设置后，访问奈飞依然返回 403 错误（未解锁）。
  - 解决方案：由于系统默认优先使用 IPv6 访问，而测试中 IPv6 并未解锁。可通过指令直接禁用 IPv6，或修改 IPv6 的优先级 [11]。

## 最佳实践

- 修改配置文件时，推荐使用支持“代码高亮”的记事本工具，便于判断代码块结束位置，避免将配置文件改崩 [5, 6]。
- 对于 [[WARP]] 解锁，如果当前 [[VPS]] 无法解锁奈飞，建议直接放弃该方法或换一个地区的 [[VPS]] 尝试，因为现在通过不断断开重连刷 IP 的方法已经失效（分配的都是重复的封禁 IP） [2, 8]。
- 使用脚本一键安装 [[DNS]] 和 [[SNI Proxy]] 服务时，建议将两者分步独立安装，一旦出现问题更方便排查定位 [9]。

## 关键概念

### [[原生IP]]
- **定义**：原本就能直接观看奈飞全部内容的服务器IP地址 [1]。
- **视频上下文**：作为第一种解锁方案介绍，提到其价格比普通IP的VPS贵，且有商家会用DNS解锁冒充原生IP [1, 2]。
- **为什么重要**：是最直接的解锁方式，但存在IP被奈飞拉黑不包换的风险以及真伪难辨的坑 [1]。
- **关联概念**：对比 → [[DNS解锁]] [1, 2]

### [[二级代理]]
- **定义**：将本地发给不支持解锁奈飞节点的流量，转交给另一个支持解锁奈飞的节点，也称为前置代理 [2]。
- **视频上下文**：作为第二种解锁方案被详细演示，用于将“直连速度快但不解锁”的节点与“能解锁但直连速度慢”的节点优势进行结合 [2, 3]。
- **为什么重要**：完美解决了解锁节点直连速度极差的问题，方案非常实用 [2, 3]。
- **关联概念**：依赖 → [[分流配置]] [3]

### [[分流配置]]
- **定义**：在代理软件的路由规则中设定条件，让特定域名组（如奈飞、迪士尼等）的流量走指定的代理出站 [3]。
- **视频上下文**：在配置二级代理时，作者演示了如何修改路由规则（Routing），让奈飞流媒体的流量走名为“Netflix Proxy”的代理节点 [3]。
- **为什么重要**：是实现二级代理和WARP解锁的核心步骤，确保只有特定的流媒体流量被转发，而不影响其他日常网站的访问 [3, 4]。
- **关联概念**：未明确提及

### [[WARP]]
- **定义**：Cloudflare提供的一款免费VPN工具，可分配能够解锁流媒体的代理IP [5, 6]。
- **视频上下文**：作为第三种方案展示，通过安装WARP并在VPS上开启SOCKS代理模式，以此获取解锁奈飞的IP [5, 6]。
- **为什么重要**：解锁原理与二级代理相同，但无需用户自行准备专门的解锁节点，是目前唯一的免费解锁方案 [4, 5]。
- **关联概念**：对比 → [[二级代理]] [4]

### [[DNS解锁]]
- **定义**：通过修改节点服务器的DNS配置，将访问流媒体域名的请求解析并转发到一台提供反向代理服务的服务器上 [4, 7]。
- **视频上下文**：作为第四种解锁方案，作者讲解了其类似DNS劫持的通信原理，并演示了如何在一台解锁机上搭建相关服务 [4, 7, 8]。
- **为什么重要**：使用此方案的节点服务器端无需设置复杂的路由代理规则，且可以无缝切换被封禁的解锁机，常为机场主使用 [4, 9]。
- **关联概念**：依赖 → [[SNI Proxy]] [7, 8]

### [[SNI Proxy]]
- **定义**：一种反向代理工具，用于接收被DNS引导过来的请求并将其正确转发给奈飞官方服务器 [7, 8]。
- **视频上下文**：在搭建DNS解锁实操环节，作者演示使用一键脚本安装SNI Proxy，使得VPS具备反向代理奈飞等流媒体网站的能力 [7, 8]。
- **为什么重要**：它是DNS解锁架构中真正负责与奈飞建立通信、让奈飞误以为正在与该服务器通信的核心组件 [7]。
- **关联概念**：属于 → [[DNS解锁]] [8]

### [[非自制剧]]
- **定义**：奈飞平台上受制于IP地域版权限制的非官方自制影视内容 [10]。
- **视频上下文**：作者在开篇强调，所谓的“解锁奈飞”通常是指解锁“非自制剧”，因为绝大部分普通的VPS默认都能正常观看自制剧 [10]。
- **为什么重要**：界定了“奈飞解锁”的真实检验标准，如果仅能看自制剧（如部分WARP分配的IPv6节点）则不属于完全解锁 [6, 10]。
- **关联概念**：未明确提及
