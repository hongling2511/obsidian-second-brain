## 一句话总结
详解 VPN 与代理协议差异，教学搭建 WireGuard 连接内网。

## 目标受众与前置知识
- **适合谁看**：需要安全实现异地组网访问家庭局域网、希望了解代理协议与 VPN 底层区别的网络技术爱好者。
- **前置知识**：[[ipv6]] 基础与获取、[[软路由]] 基本操作、[[Clash]] TUN模式工作原理（讲者提到懂此模式更易理解）。

## 核心观点
1. **常用代理协议不是真正的 VPN**
   - 为什么重要：澄清技术概念，避免误将代理协议的 TUN 模式等同于真正的网络层通信。
   - 依据：「[[shadowsocks]]、[[vmess]] 等代理协议无法代理网络层，而 [[VPN]] 可以直接封装包含 IP 头部的整个网络层数据包」（讲者原话/推断整合）。
2. **VPN 的核心用途是异地组网而非翻墙**
   - 为什么重要：纠正大众对 VPN 仅用于科学上网的刻板印象，展示其在企业和个人内网访问中的原生价值。
   - 依据：「[[VPN]] 的出现主要是为了解决异地访问内网服务，尤其大公司使用非常广泛，只不过对数据进行了加密顺便实现了翻墙」（讲者原话）。
3. **公网暴露内网服务存在严重安全隐患**
   - 为什么重要：强调采用虚拟局域网（组网）的必要性，替代高风险的直接端口映射。
   - 依据：「上周刚爆的 [[Windows]] [[ipv6]] rc 漏洞就是非常严重的安全问题，可能导致用户成为僵尸网络中的人肉电池」（讲者原话）。

## 概念关系
- [[VPN]] ↔ [[shadowsocks]] / [[vmess]]：对比。[[VPN]] 在网络层工作，直接处理并加密原始 IP 数据包，能保留真实 ping 延迟；而后者无法直接处理 IP 头部，需 [[gvisor]] 解析。
- [[ddns]] → [[WireGuard]]：依赖。由于家庭宽带 [[ipv6]] 地址动态变化，[[WireGuard]] 客户端需依赖 [[ddns]] 域名解析来持续获取正确的服务端 IP 以维持连接（推断）。
- 物理网线 ↔ [[VPN tunnel]]：类比。前者是连接设备的实体物理线，后者是在公网中通过虚拟网卡和加密数据流构建的“虚拟网线”，实现等效的局域网互通。

## 详细笔记
### 暴露内网的风险与真正 VPN 的定义
讲者 [[不良林]] 介绍，许多人误以为 [[shadowsocks]] (原文字幕 sh socks)、[[vmess]] (原文字幕 womens)、[[vless]] (推断，原文字幕 vens)、[[trojan]] (推断，原文字幕 Chen) 是真正的 [[VPN]]。实际上，为避免复杂的 [[tls]] 证书配置，且防止类似 [[Windows]] [[ipv6]] rc 漏洞导致的 [[DDoS]] (推断，原文字幕 DD) 攻击或沦为肉鸡的风险，不应将 [[NAS]] 等内网服务直接暴露在公网。真正的 [[VPN]] 是在公网上建立的 [[VPN tunnel]]（隧道），相当于在异地设备与家里路由器之间拉了一根专用的虚拟网线。

### VPN 与代理协议的工作原理差异
[[VPN]] 常见协议包括 [[WireGuard]] (原文字幕多次识别为 wga/voga/wiog 等)、[[OpenVPN]] (opvpn)、[[IPsec]] (ipsc)、[[L2TP]] (vpnl2tp)。当电脑发起请求（例如从本地随机 9527 端口通过 [[tcp]] 发往内网 192.168.2.2 的 80 端口，讲者原话），[[WireGuard]] 会直接从虚拟网卡读取包含完整 IP 头部的请求数据包，将其加密并封装为自身协议格式，随后通过本机随机端口（如 4134 端口，讲者原话）以 [[udp]] 发往路由器公网 IP（如 2.2.2.2:23456，讲者原话/屏幕展示）。而 [[Clash]] 的 TUN 模式无法直接处理网络层，必须借助 [[gvisor]] 或 [[System]] 协议栈去掉 IP 头部，解析出 payload 后才能交由代理协议处理。由于无需经过复杂的 [[nat]] 转换解析，VPN 能够获取到真实的 ping 延迟（相较于代理返回的 1 毫秒假延迟，讲者原话）。

### 在软路由上部署 WireGuard 教程
基于获取了公网 [[ipv6]] 的 [[Linux]] (原文字幕 ninux) [[软路由]] 操作流程：
1. **安装插件**：安装 `ddns`、`wireguard` 和用于生成二维码的 [[qrencode]] (原文字幕 qrn c) 插件，并重启网络服务使配置生效。
2. **配置 DDNS**：在 [[dynv6]] 网站免费注册避免 [[ipv4]] 污染（讲者原话举例被污染到 127.0.0.1）的动态域名。在路由器绑定获取 [[WAN口]] (原文字幕 万口) 的 [[ipv6]] 地址，计时器设为 600 秒（讲者原话）检测一次变动。
3. **接口与防火墙**：创建 [[WireGuard]] 接口，生成密钥，设置监听端口（如 23456），分配不冲突的 IP 段（如 192.168.5.1）。在防火墙通信规则中放行目标端口 23456 的 [[udp]] 流量，并在高级设置中将协议设为仅 [[ipv6]]。
4. **客户端配置**：添加对端，分配 IP（如 192.168.5.2），生成客户端所需的配置文件与二维码。

### 客户端连接与 MTU 优化
手机端需在 [[APP STORE]] 使用 [[非国区ID]] (原文字幕 飞果去ID) 下载客户端，扫码连接后，便能在外网（如运营商 [[5G]] 网络）像连接家里 [[WiFi]] 一样，通过内网 IP（192.168.2.1，讲者原话）直接访问路由或内网服务。桌面端（[[Mac]]、[[Windows]]）采用导入配置文本的方式操作。讲者强调，若遇到网速远达不到上行带宽、卡顿严重，大概率是 MTU 设置过大导致数据包分片。建议使用 [[Wireshark]] (原文字幕 wsock) 抓包，若观察到大量白色背景数据包，需将 MTU 调小至 1400 或 1380（讲者原话）。注：需开启 [[LAN口]] (原文字幕 L口) 区域内的转发功能才能互访。

### 利用常见代理协议访问内网
演示通过 [[homeep]] (推断为 HomeProxy) 或 [[OpenClash]] 搭建节点（如在 7788 端口搭建，讲者原话），配合 [[小火箭]] (Shadowrocket) 客户端测试。默认情况下，由于绝大多数代理工具的通用设置跳过了内网网段，无法直接访问内网。必须在设置中删除“跳过内网网段”规则后，方可通过代理节点访问内网服务。讲者强烈警告切勿随意公开分享家用节点，以免个人内网隐私服务被外人探清。

## 行动建议
- **做什么**：在软路由防火墙中，添加通信规则放行对应的目标端口（如教程中的 23456）的 [[udp]] 流量，并在高级设置中将协议限定为仅 [[ipv6]]。
- **关联观点**：2
- **验证标准**：保存应用后，异地客户端可以通过该端口成功与局域网建立 VPN 连接，软路由接口状态页面显示出实时流量数据。
- **做什么**：遇到异地访问网速缓慢时，使用 [[Wireshark]] 抓包测速，若出现大量白色背景数据包（分片现象），则在客户端中将 MTU 调小至 1400 或 1380。
- **关联观点**：2
- **验证标准**：调小 MTU 后再次进行测速，分片数据包消失，网络传输速度恢复到接近家庭宽带实际上行带宽。

   - 在接口配置中添加对端，生成预共享密匙，分配允许的 IP 地址，随后生成配置信息与二维码 [7, 9]。

   - 关键代码/命令：源区域设为 WAN，目标区域设为设备，目标端口 `23456`，协议设置为仅 [[IPv6]]，动作设为接受 [7]。

## 常见坑与解决方案
- **问题描述**：将内网服务直接暴露在公网容易导致设备被黑客攻击（如 Windows 的 [[IPv6]] RCE 漏洞） [1]。
  - 解决方案：下载安装对应系统的安全补丁，或者直接关闭 Windows 的 [[IPv6]] 功能 [1]。
- **问题描述**：[[DDNS]] 插件无法正常工作，无法更新域名解析 [9]。
  - 解决方案：检查是否关闭了 [[IPv6]] 的 DNS 解析功能，需重新开启或为部分域名开启 [[IPv6]] 解析，以确保路由器能正确比对 IP [9]。
- **问题描述**：连接 VPN 后，不同内网网段（如 192.168.2.x 与 192.168.5.x）之间无法互相访问 [10]。
  - 解决方案：需要在 LAN 区域内开启转发功能才能成功互访 [10]。
- **问题描述**：连上 [[WireGuard]] 后网络卡顿严重，远达不到家里的上行带宽 [4]。
  - 解决方案：大概率是 MTU 默认值太大导致数据包被分片。可使用抓包工具（如 [[Wireshark]]）观察，若存在分片则需要调小 [[WireGuard]] 的 MTU 值（例如修改为 `1400` 或 `1380`） [4]。
- **问题描述**：使用常规代理节点（如 [[VMess]]）并开启全局模式后，依然无法访问内网 IP [4, 5]。
  - 解决方案：进入代理工具的通用设置，找到默认跳过代理的内网网段，将其删除并保存配置 [5]。

## 最佳实践
- 申请免费的 [[DDNS]] 域名时，尽量选择冷门的后缀，以避免遇到 DNS 污染（如 IPv4 被污染到 127.0.0.1） [8]。
- 个人使用时，防火墙区域直接设置为 LAN 区域即可，没必要单独划分一个区域徒增麻烦 [9]。
- 绝对不要将家里的自建节点随便分享给别人，否则可能会导致内网隐私服务被他人完全访问 [5]。

## 关键概念

### [[VPN]]
- **定义**：一种可以创建虚拟隧道，实现异地访问内网服务并对数据进行加解密的虚拟专用网络 [1]。
- **视频上下文**：作者指出很多人以为常用来翻墙的工具就是VPN，但VPN的主要用途是异地访问内网服务，相当于在公网和内网之间拉了一根虚拟网线 [1, 2]。
- **为什么重要**：作为本视频探讨的核心基石，明确它的原本定义能够破除长久以来将代理等同于VPN的误区 [1, 2]。
- **关联概念**：扩展 → [[VPN tunnel]]

### [[代理协议]]
- **定义**：如Shadowsocks (ss)、vmess等常用于科学上网的协议，无法处理IP头部且无法直接代理网络层数据 [2-4]。
- **视频上下文**：视频开篇即将其与真正的VPN进行区分，并在讲解延迟测试时指出，代理协议在tun模式下无法获取真实的ping延迟 [2, 4]。
- **为什么重要**：它代表了大众常误认为VPN的技术，澄清两者的能力边界是理解网络底层代理的前提 [2, 4]。
- **关联概念**：对比 ↔ [[VPN]]

### [[VPN tunnel]]
- **定义**：运行VPN后在公网上创建的一条虚拟连接通道，通过它连接两端的虚拟网卡接口 [1, 3]。
- **视频上下文**：作者将其形象地比作一根虚拟网线，一头连接异地电脑，一头连接老家路由器，使得两端可以通过这条隧道直接进行数据通信 [1, 3]。
- **为什么重要**：它是理解VPN如何跨越公网实现内网互通的核心原理模型 [3, 5]。
- **关联概念**：依赖 → [[VPN]]

### [[WireGuard]]
- **定义**：一种被Linux之父称为艺术品的VPN技术协议实现，支持点对点和远程访问两种配置方式 [6]。
- **视频上下文**：视频以它为例，详细演示了如何在软路由上通过配置接口、通信端口和虚拟网卡来搭建真正的VPN服务 [4, 6, 7]。
- **为什么重要**：作为视频中手把手教学的实操对象，掌握它即可实现异地公网安全访问家庭局域网的目标 [2, 4]。
- **关联概念**：依赖 → [[MTU]]

### [[网络层]]
- **定义**：包含IP头部的网络数据层级，是ping等工具以及真实VPN运作的通信层级 [3, 4]。
- **视频上下文**：视频在解释通信原理时提到，VPN能够直接封装包含IP头部的整个网络层数据包，因此能获取真实的ping延迟 [3, 4]。
- **为什么重要**：它是区分真假VPN的最关键技术指标，决定了代理工具能否处理底层的网络协议包 [4]。
- **关联概念**：未明确提及

### [[点对点 VPN]]
- **定义**：VPN隧道部署在两台路由器上，实现两个局域网内部网络直接互通的类型 [6]。
- **视频上下文**：视频按使用场景划分VPN分类时提到，这种组网方式在企业中应用广泛，也适合用户拥有“两个家”的互联需求 [6]。
- **为什么重要**：展示了VPN在解决多地局域网整体物理互联时的架构方案 [6]。
- **关联概念**：对比 ↔ [[远程访问 VPN]]

### [[远程访问 VPN]]
- **定义**：虚拟网线一头连在路由器上充当服务端，另一头直接连在用户电脑客户端上的VPN类型 [6]。
- **视频上下文**：视频指出这不仅是大部分个人用户以及外地出差连公司网络使用的方式，也是本期实操主要讲解的配置模式 [6]。
- **为什么重要**：它是对个人用户价值最大的应用模式，理解它就能完成单台设备到异地局域网的安全接入 [6]。
- **关联概念**：对比 ↔ [[点对点 VPN]]

### [[DDNS]]
- **定义**：用于自动检测接口公网IP变化并实时更新域名解析的动态DNS服务 [8]。
