# Let's build the GPT Tokenizer

## 一句话总结
深入剖析 [[大语言模型]] 中 [[Tokenization]] 原理及其导致的各类反直觉故障现象 [1, 2]。

## 内容类型
tech_tutorial

## 目标受众与前置知识
- **适合谁看**：AI 开发者、研究人员，以及希望深入理解 [[大语言模型]] 底层运作机制与各类“怪异”模型行为根源的技术从业者。
- **前置知识**：需要具备基础的 [[Python]] 编程能力，了解基础的 [[神经网络]] 与 [[Transformer]] 架构模型概念，以及对 [[字符编码]]（如 [[Unicode]] 和 [[UTF-8]]）有基本认知 [3-6]。

## 核心观点
1. **观点概括**：[[Tokenization]] 是导致 [[大语言模型]] 诸多怪异缺陷的根本原因 [2, 7]。
   - 为什么重要：这打破了人们习惯将问题归咎于模型架构本身的迷思，指出了数据预处理环节才是算术差、无法处理复杂拼写、非英语表现不佳等问题的罪魁祸首 [2, 8, 9]。
   - 说服力评估：极强。作者展示了 `egg` 的不同大小写对应不同 Token、算术被任意切分、以及 Python 缩进使得 GPT-2 效率极低等直观测试案例 [10-13]。
2. **观点概括**：[[Byte Pair Encoding]] (BPE) 是当前压缩文本并生成模型词表的核心算法 [14, 15]。
   - 为什么重要：不理解 BPE 算法就无法理解文本是如何被转换成 [[Transformer]] 能够处理的输入向量的 [16, 17]。
   - 说服力评估：强。作者从底层逻辑出发，逐行编写 Python 代码实现了 BPE 的词频统计、合并和编解码过程，极具实操说服力 [18-21]。
3. **观点概括**：分词器的训练与语言模型本身的训练是完全独立的两个阶段 [22, 23]。
   - 为什么重要：揭示了分词器拥有自己独立的训练集和词表，这解释了为何会出现因数据不匹配而导致的极端安全漏洞或崩溃 [24, 25]。
   - 说服力评估：强。通过详细剖析 Reddit 社区词汇 "Solid Gold Magikarp" 引发模型崩溃的案例，完美佐证了这一观点 [25-27]。

## 副观点与隐含立场
- **副观点**：尽管当前行业离不开 [[Tokenization]]，但这是一个需要被警惕的“历史包袱”，理想的未来应是端到端的直接处理 [14, 28]。
   - 如何为核心观点服务：强化了该环节充满了“陷阱（footguns）”，凸显了开发者掌握其原理以避坑的必要性 [1, 28]。
   - 隐含前提：作者假设观众认同目前使用复杂的 [[正则表达式]] 分词规则以及处理乱码字符是丑陋且非最优的工程妥协 [29, 30]。
- **副观点**：分词策略直接关系到模型调用的经济性与上下文窗口的使用效率 [31, 32]。
   - 如何为核心观点服务：从成本和效率的现实角度，论证了理解 Tokenizer 对于实际应用场景的巨大价值 [32]。
   - 隐含前提：受众关注 API Token 计费成本以及 [[上下文窗口]] 长度受限的现实约束 [13, 32]。

## 详细笔记
- **[概念引入]** 介绍 Tokenization 导致的各类“反直觉”现象。通过网页端分词器演示，解释为何拼写任务难、多语言模型效果差、算术运算不稳定等现象本质都是 Token 切分问题 [1, 2, 7, 10, 11, 31, 33-35]。
- **[基础理论]** 探讨从文本到向量的映射。复习 [[Unicode]] 字符集与 [[UTF-8]] 编码规则。解释为何不直接将字节串喂给模型，并引出通过 [[Byte Pair Encoding]] (BPE) 进行序列压缩的需求 [5, 6, 14, 36]。
- **[代码实战]** 从零手撕 BPE 算法。演示如何寻找最常见的字节对（Pairs），如何不断合并（Merges）扩展词表（Vocab），以及实现健壮的 Encode 和 Decode 函数处理失效的 UTF-8 字节 [19-21, 37]。
- **[工业级实现分析]** 剖析 OpenAI 的 `tiktoken` 与 GPT-2/GPT-4 分词器源码。讲解如何使用 [[正则表达式]] 强制阻断跨类别的字符合并。引入特殊标记（如 `<|endoftext|>`）处理多轮对话等微调格式 [29, 38-42]。
- **[工具包对比]** 介绍 Google 开源的 [[SentencePiece]]。对比其基于字符操作以及添加 Dummy 前缀等复杂历史设置，与基于字节的 `tiktoken` 做法的差异（以 [[Llama 2]] 为例） [43-45]。
- **[深层原理解析与收尾]** 探讨词表大小带来的优缺点对模型推断的影响。深挖 Solid Gold Magikarp 故障的产生逻辑，指出它是分词数据与模型预训练数据脱节造成的“未初始化内存”问题。给出格式选择与最佳实践建议 [25, 28, 32, 46-49]。

## 技术栈与环境要求
- **核心技术**：`[[Tokenization]]` (分词/词元化)、`[[Large Language Models (LLMs)]]`、`[[Byte Pair Encoding (BPE)]]`、`[[Unicode]]`、`[[UTF-8]]`、`[[Transformer]]`。
- **涉及框架与库**：`[[tiktoken]]` (OpenAI 官方库) [1]、`[[sentencepiece]]` (Google 开源库) [2]、`[[regex]]` (Python 正则表达式增强包) [3]。
- **环境要求**：Python 运行环境（如 Google Colab 或 Jupyter Notebook），Python 3.7+（依赖字典的有序性）[4]。
- **前置技能**：
  - Python 基础（字符串与字节类型转换、字典操作、列表处理）。
  - 字符编码基础（理解 Unicode Code Points 与 bytes 字节流的映射关系）。
  - 神经网络基本架构（了解 Transformer 中的 Embedding 层和最后的线性分类层 LM head 作用）。

## 步骤拆解
1. **理解文本与字节的底层表示**
   - **具体操作描述**：大语言模型无法直接处理字符串文本，需要将其转化为离散的整数（Tokens）。不能直接使用 Unicode Code Points，因为其词表太大且包含多达 15 万个字符 [5-7]。需要先将字符串统一编码为 `[[UTF-8]]` 字节流 [8]。
   - **关键代码/命令**：`text.encode("utf-8")` [9]
   - **注意事项**：原始的单字节流由于序列过长，会严重消耗 `[[Transformer]]` 极为有限的上下文窗口（Context Length），因此必须进行数据压缩 [10]。

2. **训练分词器：实现基础的 [[Byte Pair Encoding (BPE)]] 算法**
   - **具体操作描述**：遍历字节序列，统计连续出现频次最高的相邻字节对，将其合并并铸造（Mint）为一个新的 Token ID 追加到词表中 [11]。以此迭代，直到词表达到预设的大小（Vocabulary Size）[12]。
   - **关键代码/命令**：利用字典记录相邻元组的频次，并通过 `max(stats, key=stats.get)` 找出最常出现的 Pair [13, 14]。
   - **注意事项**：Tokenizer 分词器的训练与 LLM 模型的训练是两个完全独立的预处理阶段，通常它们拥有各自独立的训练数据集 [15]。

3. **实现解码 (Decoding)**
   - **具体操作描述**：将 Token ID 序列还原为 Python 字符串。首先利用映射表将 ID 转回字节，拼接后再通过 UTF-8 解码成文本 [4, 16, 17]。
   - **注意事项**：并非所有的字节序列都是合法的 UTF-8（例如模型乱序输出时），直接解码可能会抛出异常 [18]。

4. **实现编码 (Encoding)**
   - **具体操作描述**：将输入字符串转换为 Token ID 序列。先转为 UTF-8 字节串，然后按照训练阶段记录的合并规则树（merges dictionary）自顶向下进行递归合并替换 [19-24]。
   - **注意事项**：处理较短字符串或单字符时需要编写防御性代码，避免空序列导致 `min()` 函数报错 [25]。

5. **使用 Regex 进行强制预切分 (GPT 系列优化)**
   - **具体操作描述**：纯粹的 BPE 可能会把字母和相连的标点符号跨类合并在一起（例如 "dog." 变成一个 Token），从而把语义和标点混淆。通过引入 `[[regex]]` 预先将文本切分为字母、数字、标点等独立区块，强制阻止跨块合并 [3, 26-31]。
   - **关键代码/命令**：`import regex as re`，使用 `re.findall(pattern, text)` 进行正则匹配切分 [3, 27, 28]。
   - **注意事项**：GPT-2 的正则设计有缺陷（大小写处理不一致、未合并多余空格），导致其在 Python 代码任务上表现糟糕；GPT-4 修复了这一问题，通过高效打包连续空格大幅提升了代码能力 [32-34]。

6. **引入特殊 Token (Special Tokens)**
   - **具体操作描述**：在基础词表上，添加如 `<|endoftext|>`、FIM (Fill-in-the-Middle) 等特殊用途标记，用于划分对话轮次或文档边界 [35-40]。
   - **注意事项**：这些 Token 不参与常规的 BPE 编码，而是由专门的逻辑捕捉。加入新的特殊 Token 后，需要对 Transformer 进行“模型手术”（Model Surgery），即扩充 Embedding 矩阵与最后一层 Linear 层的维度 [41, 42]。

## 常见坑与解决方案
- **问题描述：解码出无效的 UTF-8 字节流导致程序崩溃** [17, 43]。
  - **解决方案**：在调用 `.decode()` 方法时，添加参数 `errors="replace"`，将无法解析的字节强制替换为 Unicode 替代符（通常是 `` 字符）[18]。

- **问题描述：Trailing Whitespace（尾随空格）导致模型补全逻辑崩溃** [44]。
  - 在分词规则中，空格通常是作为下一个单词的“前缀”存在（例如 ` token` 而不是单独的空格加 `token`）[45]。如果 prompt 末尾留有空格，会导致生成的下一个字符与平日训练分布不符，引发性能退化或幻觉。
  - **解决方案**：在提交推理请求前，应当注意清理 prompt 尾部的空格，或者依赖 API 层面的警告提示 [44, 46]。

- **问题描述：Solid Gold Magikarp 故障（遇到特定罕见词模型发疯/骂人）** [47-49]。
  - **解决方案**：这通常是因为 Tokenizer 的训练数据集（如包含了大量 Reddit 用户名数据）和 LLM 本身的训练集出现了未对齐 [50, 51]。这些专属词汇虽然在词表里拥有 ID，但其对应的 Embedding 向量从未被更新过，等同于未初始化内存 [52, 53]。解决此问题的核心是确保分词器与 LLM 的训练数据分布高度对齐。

## 最佳实践
- **避免过度标准化**：如果使用 `[[sentencepiece]]` 训练大语言模型，建议关掉大写转小写、多余空格清理等传统的 NLP 标准化设置，保持原始训练数据的原貌（Keep raw text raw） [54, 55]。
- **放弃“句子”概念**：现代 LLM 中不应强加“Sentence”边界限定，直接将所有预训练文档视作一段无尽的连续字节流进行处理更为合理 [56]。
- **偏好 YAML 胜过 JSON**：由于 JSON 具有大量的括号和引号格式，在 BPE 分词下往往被切得很碎。相较而言，YAML 可以节省非常多 Token。为了节约上下文窗口和 API 计费成本，在结构化数据格式选择上推荐使用 YAML [57]。
- **直接复用现成词表**：除非有极特殊的需求需要从零训练分词器，业务应用中建议直接复用 `[[tiktoken]]` 库与 GPT-4 的 `cl100k_base` 词表，它速度极快并且历经实践检验 [58]。

## 关键概念

### [[Tokenization]]
- **定义**：将字符串或文本翻译为大型语言模型（LLM）可处理的离散整数序列（Tokens）的预处理过程 [1, 2]。
- **视频上下文**：视频开篇指出，Tokenization 常常是使用大型语言模型时最令人头疼的部分，模型不擅长处理非英语语言、拼写任务表现差、难以进行简单的算术运算或出现 Python 代码缩进问题，根本原因往往都可以追溯到 Tokenization 层面 [3-6]。
- **关联概念**：依赖 → [[Byte Pair Encoding (BPE)]]

### [[Byte Pair Encoding (BPE)]]
- **定义**：一种被广泛用于分词器的算法，通过迭代查找并在序列中合并最常出现的字符对或字节对，从而铸造（mint）出新的 Token [7, 8]。
- **视频上下文**：作者详细演示了如何从头手写代码实现 BPE 算法：先统计文本序列中连续对的频率，然后用新的整数 ID 替换最高频的对，借此不断压缩序列并扩充词表 [9-12]。
- **关联概念**：依赖 → [[UTF-8 Encoding]]

### [[Unicode Code Points (Unicode 码点)]]
- **定义**：Unicode 联盟为全球所有支持的字符、脚本以及表情符号分配的唯一标准整数标识 [13, 14]。
- **视频上下文**：视频解释了 Python 中的字符串实际上是不可变的 Unicode 码点序列，但因为码点数量高达 15 万且标准在不断更新，直接将它们作为模型的输入会导致词表过大且不稳定，因此不建议直接对码点进行 BPE 操作 [13, 15]。
- **关联概念**：转换至 → [[UTF-8 Encoding]]

### [[UTF-8 Encoding]]
- **定义**：一种将 Unicode 码点转换为 1 到 4 个字节的变长字节流编码方案 [16]。
- **视频上下文**：视频强调，合理的做法是先将原始文本转换为 UTF-8 字节流，产生 256 个基础字节构成的初始词表，然后在这 256 个单字节的基础上应用 BPE 算法，因为 UTF-8 结构紧凑且完全向后兼容 ASCII [16-19]。
- **关联概念**：处理于 → [[Byte Pair Encoding (BPE)]]

### [[Vocabulary Size (词表大小)]]
- **定义**：分词器最终能够识别并支持的离散 Token 的总数量，是一个需要通过实验优化的超参数 [20, 21]。
- **视频上下文**：视频指出词表大小的权衡：词表增加能将序列压缩得更短，让模型关注更长的上下文；但过大的词表会导致模型中的嵌入表和分类层变得极为庞大，且部分罕见 Token 可能因为在训练中出现次数太少而面临“欠训练”问题 [20, 22-24]。
- **关联概念**：决定 → [[Embedding Table]]

### [[Embedding Table (嵌入表)]]
- **定义**：Transformer 语言模型中的一个二维查找表，用于将代表 Token 的离散整数映射为可通过反向传播训练的连续稠密向量 [25]。
- **视频上下文**：视频中解释了输入模型的 Token ID 会作为索引，从嵌入表中提取出对应行的向量，如果想要给模型增加新的词汇（例如特殊 Token），就必须对模型动“手术”，增加嵌入表的行数并初始化新的向量 [25, 26]。
- **关联概念**：依赖 → [[Vocabulary Size (词表大小)]]

### [[Regex Chunking (正则分块)]]
- **定义**：在运行 BPE 算法前，利用正则表达式将原始文本预先切分为相互独立的数据块（如字母、数字、标点分开）的机制 [27, 28]。
- **视频上下文**：视频解析了 GPT-2 和 GPT-4 的分词机制，指出 OpenAI 刻意用正则表达式将标点符号、字母等强制隔离，以防止 BPE 算法产生像 "dog." 这样把单词和标点毫无道理地融合在一起的冗余 Token [28-30]。
- **关联概念**：优化 → [[Byte Pair Encoding (BPE)]]

### [[Compression Ratio (压缩率)]]
- **定义**：原始字节流序列长度与经过 BPE 合并压缩后的 Token 序列最终长度的比值 [31]。
- **视频上下文**：作者通过代码展示，随着 BPE 合并循环次数的增加，相同的文本被表示成了更少的 Token（例如演示中执行 20 次合并后达到了约 1.27 的压缩率），这有效提高了 Transformer 消耗文本内容的密度和效率 [24, 31]。
- **关联概念**：度量 → [[Byte Pair Encoding (BPE)]]

### [[TikToken]]
- **定义**：由 OpenAI 发布的官方开源分词库，专门用于快速执行 Token 序列和字符串之间的推断转换（Encode/Decode），但不含训练代码 [32, 33]。
- **视频上下文**：视频通过 TikToken 库来演示各种字符串在 GPT-2 和 GPT-4 下被分词的差异，特别是展示了 GPT-4 对连续空格的合并效率更高，极大地改善了 Python 代码的分词效率 [32, 34, 35]。
- **关联概念**：对比 ↔ [[SentencePiece]]

### [[SentencePiece]]
- **定义**：谷歌开源的另一个主流分词库，广泛应用于 Llama 和 Mistral 等模型，同时支持分词器的训练与推理 [36]。
- **视频上下文**：视频指出 SentencePiece 与 TikToken 在底层理念上不同，它直接在 Unicode 码点层面上运行 BPE 算法（而不是字节层面），并会在遇到未见过的罕见字符时采用字节回退（byte fallback）机制 [36-38]。
- **关联概念**：对比 ↔ [[TikToken]]

### [[Special Tokens (特殊 Token)]]
- **定义**：为了提供结构化信息而人为人为插入词表的额外标记，它们并不通过 BPE 算法的自然合并产生 [39, 40]。
- **视频上下文**：视频介绍了 `<|endoftext|>` 这一特殊的占位标记，模型利用它来了解训练集中不同无关文档之间的边界，在微调阶段（如 ChatGPT）还会大量引入此类特殊 Token 来分隔对话角色和信息元数据 [40-42]。
- **关联概念**：属于 → [[Vocabulary Size (词表大小)]]

### [[Glitch Tokens (异常 Token)]]
- **定义**：由于分词器训练集和语言模型实际训练集的数据分布不一致，导致模型在其权重矩阵中完全没有学习过的一类极端罕见的 Token [43-45]。
- **视频上下文**：视频以著名的 "Solid Gold Magikarp"（一个 Reddit 用户名）为例，解释了该词由于在分词训练集中高频出现而获得专属的 Token ID，但在语言模型训练集中却从未出现，导致对应的嵌入向量处于未初始化（未分配内存）的随机状态；一旦用户输入触发该 Token，模型便会输出完全混乱、受损甚至是具有安全风险的内容 [44-47]。
- **关联概念**：副作用依赖 → [[Embedding Table]]
