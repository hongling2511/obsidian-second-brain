## 一句话总结
盘点电脑、手机和软路由端的主流去广告原理与方法，重点解析基于 [[DNS]]、[[host]] 以及 [[mitm]] 的技术实现与优劣。

## 目标受众与前置知识
- **适合谁看**：希望在各大终端设备上去除网页或APP广告的互联网用户及代理节点使用者。
- **前置知识**：[[DNS]]、[[IP]]、[[http]]/[[HTTPS]]、[[代理]]、[[抓包]]（推断）

## 核心观点
1. **基于 [[DNS]] 或 [[host]] 的去广告方法简单但有局限性**
   - 为什么重要：指出了常规屏蔽域名的痛点，即无法处理同域名下的广告。
   - 依据：如果广告与网站内容放置在同一域名下（如 [[B站]] ），单纯屏蔽域名会导致正常内容也被屏蔽（讲者原话）。
2. **[[mitm]]（中间人攻击）能精准过滤同域名加密广告，但性能开销大且门槛高**
   - 为什么重要：揭示了目前解决高难度广告拦截的核心技术及其附带代价。
   - 依据：需进行加解密操作并运行 [[JS]] 脚本，访问量大时会导致手机发烫掉电；且对于有 [[TLS pinning]] 的APP（如 [[tiktok]] ）需反编译老版本才能抓包（讲者原话）。
3. **桌面端最优的去广告方案是直接使用浏览器插件**
   - 为什么重要：为电脑端用户提供了一条低门槛且最高效的去广告路径。
   - 依据：浏览器插件直接工作在解密流量之后的层级，能够轻易修改 [[HTML]] 等网页数据并隐藏广告元素，无需配置 [[CA证书]] 实施 [[mitm]]（讲者原话）。

## 概念关系
- [[DNS]] ↔ [[mitm]]：对比。[[DNS]] 仅能在请求起始阶段拦截特定域名，而 [[mitm]] 能够解密并查看 [[HTTPS]] 流量的深层路径以实现精细化过滤。
- [[HTTPS]] → [[CA证书]]：依赖。要通过 [[mitm]] 解密并修改由 [[HTTPS]] 加密的广告数据包，设备必须安装并信任自定义的 [[CA证书]]。
- [[广告屏蔽]] ↔ [[反广告屏蔽]]：对抗（推断）。网站为生存会检测去广告插件并要求加白名单，两者技术不断对抗螺旋上升。

## 详细笔记
### 广告加载基础与 DNS 去广告原理
讲者 [[不良林]] 首先以其博客 [[不良林.com]] 为例，演示通过 `F12` 开发者面板中的“网络”标签可以发现，广告是通过向特定网址请求插入 [[JS]]（视频音译 gs）代码加载的。访问网站的第一步是发 [[DNS]] 请求获取 [[IP]]，因此只要对广告域名进行 DNS 污染，分配无效 IP（如 0.0.0.0，讲者演示推断），广告即无法加载。此类功能可通过自带模块的代理工具实现，也可使用 [[软路由]] 上的 [[AdGuard Home]]（视频称 adgard home/adg home）、[[smartdns]]、[[MosDNS]]（M DNS）、[[dnsmasq]]（dnsmc/dnsmk/dns mask）实现。讲者以 [[openwrt]]（openw）自带的 [[dnsmasq]] 演示了屏蔽过程，并指出使用 `ipconfig/flushdns` 可清除电脑 DNS 缓存。由于这和 [[GFW]] 的工作原理之一相同，讲者不推荐专门加装 [[AdGuard Home]] 徒增网络复杂性。

### 代理工具中的 host 规则屏蔽
相比于纯 DNS 拦截，讲者更推荐直接使用自带 DNS 模块的代理工具（如 [[sing-box]]/sbox、[[clash]]），在访问目标 [[host]] 的阶段通过分流规则进行屏蔽。
- **[[软路由]]**：在 [[sing-box]] 的 [[home proxy]] 插件中，通过添加网上开源的二进制格式广告规则集进行拦截出站（封锁）。
- **桌面/通用端**：在 [[open clash]] 中添加附加规则集（包含 750 条规则，讲者原话/屏幕展示），策略选为 reject。此类方式也可配置在 [[fakeip]] 模式下。
- **安卓端**：在 [[v2rayN]]（v2ng）中利用其默认的 [[geosite]]（geos）数据库直接配置阻止网址。
- **苹果端（iOS）**：在 [[小火箭]] 中添加规则集并将策略设置为 reject。
- **服务端节点**：在自建节点如 [[x-ui]]（x油安）面板的 [[Xray]]（插瑞）设置中直接植入屏蔽代码，客户端可免配置，但无法屏蔽国内 [[直连]] 广告。

### 同域名广告困境与 mitm (中间人攻击) 原理
基于域名的屏蔽对于像 [[B站]]（哔哩哔哩）、[[京东]] 这种将正常内容和广告数据放置在同一个域名中的应用无效。对于明文的 [[http]] 请求，可以通过编写 [[正则]] 匹配请求路径屏蔽。但在 [[HTTPS]] 普及的今天，必须利用 [[mitm]] 才能看到加密数据包内的具体内容。
原理为：以 [[小火箭]]（同时提及 [[Surge]]/sarch 和 [[Loon]]/loom）为例充当中间人，解密 APP 与服务器通信数据，利用 [[JS]]（gs）脚本剔除广告数据后再重新加密返回给 APP。除了去广告，该技术还可破解本地 [[VIP]] 功能或通过修改地区代码解锁 [[tiktok]]。

### 移动端与软路由 mitm 实施及局限
- **iOS配置演示**：在 [[小火箭]] 中生成并安装、信任 [[CA证书]]，开启 HTTPS 解密，并导入第三方去广告模块（含 URL rewrite 和 script 等操作指令）。
- **弊端与门槛**：增加计算开销引起发热掉电；规则编写需 [[抓包]] 分析；遇到 [[TLS pinning]]（tls 拼 ING）需反编译 APP 获取老版本。
- **安卓限制**：由于安卓 7.0（讲者原话）后默认不信任用户证书，配置 [[mitm]] 必须 [[ROOT]]，讲者推荐非 ROOT 方案可使用 [[李跳跳]]（理跳跳）跳过开屏广告。
- **软路由与服务端**：虽然有 [[cproxy]] 等插件支持，但因极吃性能，讲者强烈不建议在 [[软路由]] 或服务器节点（除早前插入 [[cookie]] 伪装看 [[奈飞]] 等演示外）部署 [[mitm]]。

### 桌面端浏览器插件与特殊去广告法
- **Windows / 桌面系统**：系统级去广告工具多收费且易导致网络冲突。桌面端绝大多数流量通过浏览器，故最推荐使用浏览器插件。
- **插件工具**：演示在 [[Chrome]] 中安装 [[AdGuard]]（adcard/adg）。因插在浏览器解密后工作，无需 [[CA证书]] 即达到 [[mitm]] 级别效果，能拦截第三方请求并基于 [[HTML]]/[[CSS]]/[[JS]]（前端三剑客）隐藏网页元素。讲者以 [[itdog]] 网站为例展示了元素隐藏。此外提到解除网页限制可使用 [[Tampermonkey]]（T monkey/油和件）。
- **局限与特殊广告**：插件有误杀风险（如拦截验证码导致无法登录）。视频中直接剪辑进画面的赞助商广告（讲者提及自己的专线 [[机场]] 和 [[VPS]] 广告）无法拦截。
- **送中节点**：针对 [[YouTube]]（某管），利用 IP 归属地被 [[谷歌]] 判定为中国的节点（送中 IP）看视频无广告，但会导致失去部分谷歌服务且影响同 IP 段他人，讲者不推荐。

## 金句亮点
- 有限度的广告是可以接受的，但影响正常使用的恶心广告必须将其消灭之。 — 价值观判断 / 动机说明
- 广告屏蔽和反广告屏蔽技术是不断对抗螺旋上升的，目前来看广告屏蔽更胜一筹。 — 行业洞察 / 技术演进总结

## 步骤拆解
1. **通过 [[DNS]] 屏蔽广告（以软路由 [[dnsmasq]] 为例）**
   - 具体操作描述：在 [[dnsmasq]] 中指定广告域名对应无效 IP（如 0.0.0.0），并保存应用 [1]。
   - 关键代码/命令：`ipconfig /flushdns` [1]。
   - 注意事项：系统与浏览器均存在缓存，需清除 DNS 缓存和浏览器缓存后才会生效 [1, 2]。

2. **通过 Host 屏蔽广告（以 [[sing-box]] 的 homeproxy 为例）**
   - 具体操作描述：添加广告域名的规则集并选择二进制文件格式，随后在路由规则中添加一条新规则，使用该规则集并将出站改为“封锁”，最后将该规则拖到上方并应用 [3]。
   - 关键代码/命令：未明确提及。
   - 注意事项：由于在访问阶段进行阻断，屏蔽生效后的错误代码为连接被关闭，与 DNS 解析错误不同 [4]。

3. **通过 Host 屏蔽广告（以 [[OpenClash]] 为例）**
   - 具体操作描述：进入规则附加页面添加规则，填入广告域名的规则集网址，保证匹配顺序为优先，策略组选择 reject，应用配置并等待重启 [4, 13]。
   - 关键代码/命令：未明确提及。
   - 注意事项：未明确提及。

4. **安卓端屏蔽广告（以 [[v2rayNG]] 为例）**
   - 具体操作描述：编辑自定义规则，在阻止的网址处添加代码，直接调用自带的 geosite 数据库获取广告域名，保存后重启 VPN 服务 [13]。

   - 注意事项：页面可能会残留广告位（内容加载失败），需清空浏览器缓存刷新 [13]。

5. **自建节点端屏蔽广告（以 [[x-ui]] 面板的 [[Xray]] 为例）**
   - 具体操作描述：在面板的相关设置中插入 Block 代码，阻断所有请求到广告域名的流量，保存并重启面板 [5]。

## 常见坑与解决方案
- **问题描述**：设置 [[DNS]] 去广告后，网页依然加载广告 [1, 2]。
  - 解决方案：清除电脑 DNS 缓存并清空浏览器缓存 [1, 2]。
- **问题描述**：同域名下的广告和正常内容混杂（如 B 站开屏广告），单纯屏蔽域名会导致正常内容也无法访问 [6]。
  - 解决方案：使用 [[mitm]] 配合正则表达式，拦截获取广告的请求或重写数据包剔除广告数据 [6-8]。
- **问题描述**：由于 [[HTTPS]] 加密，代理工具无法查看数据包内容，过滤规则失效 [7]。
  - 解决方案：配置 [[mitm]]，通过安装 CA 证书进行中间人攻击以解密流量 [7]。
- **问题描述**：iOS 端点击安装证书后提示错误 [8]。
  - 解决方案：将 [[Shadowrocket]] 升级到最新版 [8]。
- **问题描述**：某些 APP（如 TikTok）采用 [[TLS pinning]] 防范中间人抓包，导致无法进行 [[mitm]] [9]。
  - 解决方案：下载老版本 APP（未做 TLS pinning）或对软件进行反编译 [9]。
- **问题描述**：从 Android 7.0 开始，系统默认不信任用户自行安装的 CA 证书，导致无法直接使用 [[mitm]] [9]。
  - 解决方案：将手机 ROOT 后把 CA 证书安装到系统根证书目录；未 ROOT 用户只能使用拦截域名去广告，或安装“李跳跳”等 APP 自动跳过开屏广告 [9, 10]。
- **问题描述**：出现误杀情况，例如网页验证码被当成广告屏蔽，导致无法正常登录或注册 [15]。
  - 解决方案：暂时关闭广告过滤插件，观察网页是否恢复正常 [15]。
- **问题描述**：网站检测到广告加载失败，弹窗限制用户浏览 [12]。
  - 解决方案：添加拦截此弹窗的过滤规则，或将该网站加入白名单 [12]。
- **问题描述**：直接剪辑进视频正片中的广告 [15]。
  - 解决方案：未明确提及 [15]。

## 最佳实践
- 在 [[DNS]] 阶段去广告时，更推荐直接使用自带 DNS 模块的代理工具（如 [[sing-box]]），不推荐单独安装 [[AdGuard Home]]，因其效果有限且会徒增网络复杂性 [2-4]。
- 桌面系统更推荐使用浏览器插件进行去广告，不推荐使用系统级去广告工具，因为后者配置繁琐且容易导致网络问题 [10]。
- 极度不推荐在软路由（如 [[OpenWrt]]）上执行 [[mitm]] 去广告，因为该操作极其消耗性能，且手机端仍需单独安装信任证书 [10]。
- 极度不推荐单纯为了去广告将节点配置为“送中”节点，这会导致部分未在中国大陆提供的服务失效，殃及同 IP 段的其他用户，且该操作难以恢复 [16, 17]。

## 关键概念

### [[DNS污染]]
- **定义**：对特定广告域名的DNS解析进行干预，使其被分配到无效的IP地址（如0.0.0.0）[1]。
- **视频上下文**：作者将其作为最基础的去广告原理解释，通过让设备获取不到广告域名的正确IP来阻止广告加载，但也指出其局限性是无法处理与正常内容共用域名的广告[1, 2]。
- **为什么重要**：它是软路由（如AdGuard Home、DNSmasq）和代理工具中最常见、最底层的去广告工作机制[1, 3]。
- **关联概念**：对比 → [[mitm]]

### [[host]]
- **定义**：http请求中包含的访问目标网站域名信息[4]。
- **视频上下文**：作者说明代理工具在获取到正确IP之后，可以通过嗅探数据包中的host来判断其是否属于广告域名，进而根据分流规则将其屏蔽[4]。
