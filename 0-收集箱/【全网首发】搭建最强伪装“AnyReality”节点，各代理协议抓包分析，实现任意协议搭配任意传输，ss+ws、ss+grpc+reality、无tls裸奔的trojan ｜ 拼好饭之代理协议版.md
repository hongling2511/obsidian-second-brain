## 一句话总结
通过抓包分析不同代理协议与传输方式组合的数据特征，并演示如何搭建兼具隐蔽性与免证书优势的 AnyReality 节点。

## 目标受众与前置知识
- **适合谁看**：想要深入理解代理协议底层数据传输特征，或需要搭建个性化免证书代理节点的技术用户（推断）。
- **前置知识**：[[代理协议]]、[[抓包工具]]、[[TCP]]/[[UDP]]、[[TLS]]/[[reality]]。

## 核心观点
1. **代理处理过程可拆解为三层架构**
   - 为什么重要：理解这三层结构是实现任意协议搭配任意传输方式的理论基础。
   - 依据：讲者将数据处理明确划分为代理协议层（如 [[SS]]）、传输方式层（如 [[WS]]）和传输安全层（如 [[TLS]]）（讲者原话）。
2. **底层协议加解密直接改变数据包的大小与内容可见性**
   - 为什么重要：直观展示了未加密协议的明文特征与加密带来的体积开销。
   - 依据：通过抓包对比，未加密的 [[SS]] 访问 [[Google]] 的数据包大小为 87 个字节（屏幕展示/讲者原话），而使用 [[AES]] 加密后膨胀到了 171 个字节（屏幕展示/讲者原话）。
3. **SS 协议的 UDP 数据默认不经过传输安全层加密**
   - 为什么重要：影响节点的实际安全性和防阻断策略，对游戏和突破封锁有不同影响。
   - 依据：抓包显示本地通过 SS+reality 给 [[VPS]] 发送 UDP 数据时，并未通过 [[reality]] 加密，而是直接快速发送（讲者原话/演示）。
4. **[[AnyReality]]（ANTLS + Reality）能兼顾防特征识别与零证书配置**
   - 为什么重要：解决了 TLS 中嵌套 TLS（讲者称 tos 因 tos）的问题，同时大幅降低了搭建门槛。
   - 依据：讲者指出该组合既解决了协议嵌套问题，又因为使用 reality 而可以直接使用其他网站的证书（讲者原话）。

## 概念关系
- [[ANTLS]] + [[reality]] → [[AnyReality]]：前者的个性化填充能力解决特定封锁问题，后者解决免证书配置问题，组合构建出兼顾两者优势的新节点。
- 未加密 [[SS]] ↔ [[AES]] 加密 [[SS]]：前者抓包能直接看到域名长度和端口等明文协议头信息，后者抓包全为乱码且数据包体积变大。
- [[TCP]] ↔ [[UDP]]：在代理底层表现不同，TCP 需要三次握手建连，而 UDP 直接发送不需建连。

## 详细笔记
### 代理协议的三层结构
讲者 [[不良林]] 提出代理处理的数据可分为三个部分：
- **代理协议（最上层）**：负责初步处理数据，例如 [[SS]]、[[S Women's Willness]]（推断为 VMess/VLESS）、[[Children]]（推断为 Trojan）、[[SOC]]、[[NTLS]]、[[ANTLS]] 等。
- **传输方式（第二层）**：上层数据传输的载体，如 [[ROW]]（原始状态）、[[WS]]、[[KP]]、[[GRPCX]]、[[HTTP]]、[[HTTP upgrade]]、[[MIC]] 等。例如 [[WM]] 加 WS 就是上层数据通过 WS 传输。
- **传输安全（第三层）**：负责底层加密，如 [[TLS]] 以及 [[Rreality]]（reality）。

### SS 协议抓包分析（无加密 vs AES 加密）
- **无加密的 SS 数据流**：打开 [[ck]]（抓包工具），监听 [[V2N]] 的 10808 端口，使用 [[C]] 指令访问 [[Google]]。抓包显示：经过 [[TCP]] 三次握手后，第 4 个数据包大小为 87 个字节（讲者原话）。通过分析 16 进制数据可清晰看到协议结构（如 03 代表域名，长度 10 个字节，访问 google.com 的 80 端口）。[[Wesock]]（抓包工具识别名）甚至将其错误解析为 HTTP 协议，最后通过 [[fination]] 结束连接。
- **AES 加密的 SS 数据流**：数包变成乱码，无法解析出协议特征，且在访问相同内容下，数据大小从 87 个字节膨胀到了 171 个字节（讲者原话）。大部分情况下加密会让数据变大。

### 不同传输方式与协议搭配测试
- **[[SS]] + [[WS]]**：Host 设置为 [[百度]]。抓包显示在三次握手后发起 HTTP 请求升级到 WS（返回 101 码表示接受升级），随后在 WS 连接里传输 SS 加密乱码数据。WS 仅负责转交数据。
- **[[SS]] + [[reality]]**：非主流组合，大多数工具不支持。在本地使用 [[xay]] 运行内核心启动。抓包看起来像通过 HTTP 访问 [[雅虎]]，但里面包裹的是未加密的 SS 数据。
- **SS 代理 [[UDP]] 数据的差异**：
  - 使用 [[NC]] 监听并发送 UDP 数据至 [[VPS]]：UDP 并没有走底层 reality 传输（即未被 reality 加密），而是以原生的 SS 协议直接发送。
  - 讲者认为这对延迟敏感的游戏党是好事（数据发送快速直接，延迟低）；但如果 VPS 线路太烂，不如使用 [[oT]]（推断为 QUIC/TUIC 协议）。若用此方式访问被墙网站可能被阻断。
  - 验证加密：将协议的加密方式改为 [[ais]]（即 AES），UDP 数据包才被加密成乱码。

### 裸奔的秋卷（Trojan）抓包
- 正常使用的 [[秋卷]]（推断为 Trojan）必须套 TLS，代理工具如 [[小火箭]] 和 V2N 不允许关闭传输安全。
- 通过跑内核拖掉 TLS 后抓包发现：前段有 56 个字节是经过下 224（推断为 [[SHA224]]）哈希计算的密码 123456（讲者原话），后续协议头和发送数据均为明文，证明协议本身不加密。秋卷是通过 [[UDPTCP]] 的方式传输数据（在 TCP 里面传输 UDP）。

### AnyReality 节点搭建步骤
- **工具选择**：由于 SS 不支持 ANTLS，讲者使用 [[Sinbox]] 搭建。
- **服务端配置**：在 VPS 上安装 1.12 版以前的 Sinbox，清空配置并粘贴 AnyReality 参考配置（包含监听 43 端口、设置密码、自定义填充字节以及传输安全 reality 配置），随后重启。
- **客户端配置**：小火箭虽支持 ANTLS 但不能把 TLS 改成 reality。目前只能使用 Sinbox 客户端（[[Mac]] 版的 [[SFM]] 和 [[安卓]] 版的 [[sfa]]）。[[iPhone]] 用户需等待正式版发布或成为 [[testflight]] 用户。配置后无须申请域名证书，抓包显示为正常访问雅虎的流量，内部传输了个性化字节填充的数据。

## 金句亮点
- “由于通过传输安全进行加密，所以里面的 SS 就不需要重复加密，可以看到同样的 SS 协议，使用不同的方式传输就会产生不一样的流量特征。” — 知识解释 / 揭示多层协议组合的流量特征原理
- “对延迟敏感的游戏党而言，或许是一个好事，原身数据发送快速直接延迟低，不过前提是你使用线路不错的 VPS。” — 经验总结 / 提供了特定代理场景（游戏加速）的优化思路

## 步骤拆解
1. **配置服务端 [[Xray]]**
   - 具体操作描述：在 VPS 上安装 `[[Xray]]`，使用 `[[Vim]]` 编辑配置文件位置，粘贴演示用的配置文件并保存退出，随后重启服务 [2]。
   - 关键代码/命令：未明确提及（视频中仅指代为“这条指令”）[2]。
   - 注意事项：即使配置不加密的 SS 节点，`[[Xray]]` 内核仍要求必须保留 password 参数；此类实验性操作可能会导致 IP 被墙 [2]。

2. **抓包分析未加密与加密 [[Shadowsocks]]**
   - 具体操作描述：在 `[[v2rayN]]` 中添加加密方式为 None 的 SS 节点（10000 端口）；打开抓包工具设置捕获过滤；在命令行使用 `[[cURL]]` 通过该代理访问 Google 并观察明文数据；随后将节点改为 `[[AES]]` 加密重试并对比抓包结果 [2-4]。
   - 关键代码/命令：未明确提及（抓包与过滤仅用“这条指令”代指）[3]。
   - 注意事项：未加密状态下可直接从 10808 端口抓取到域名和协议头；加密后数据包大小会膨胀（如由 87 字节膨胀至 171 字节）[3, 4]。

3. **抓包分析 SS+[[WebSocket]]**
   - 具体操作描述：将节点传输方式改为 `[[WebSocket]]`，Host 设为 baidu.com，路径设为 A，抓取分析 HTTP 升级为 WS（返回 101 状态码）的建连过程 [4]。
   - 关键代码/命令：未明确提及 [4]。
   - 注意事项：`[[WebSocket]]` 层只负责传输转交，不关心内部传输的加密数据，内部处理由上层代理协议完成 [4]。

4. **配置并抓包 SS+[[Reality]] 及 [[UDP]] 测试**
   - 具体操作描述：在本地直接运行 `[[Xray]]` 加载 SS+`[[Reality]]` 配置（监听 11111111 端口）；在 VPS 使用 `[[Netcat]]` 监听 `[[UDP]]` 12345 端口；通过代理发送 `[[UDP]]` 数据并分析抓包结果 [4, 5]。
   - 关键代码/命令：未明确提及 [5]。
   - 注意事项：`[[Shadowsocks]]` 的 `[[UDP]]` 数据不走底层传输，不经过 `[[Reality]]` 处理，会以未加密的明文形式直接发送 [5, 8]。

5. **抓包分析无 TLS 裸奔 [[Trojan]]**
   - 具体操作描述：直接使用内核加载不包含任何传输方式配置的 `[[Trojan]]` 文件（密码 123456），通过代理访问并抓取裸奔数据包 [6, 8]。
   - 关键代码/命令：未明确提及 [8]。
   - 注意事项：常规代理工具不允许关闭传输安全，需手写配置；`[[Trojan]]` 协议本身无法加密，必须依赖 `[[TLS]]` 或 `[[Reality]]` 等传输层安全机制 [6, 8]。

6. **搭建 AnyReality 节点**
   - 具体操作描述：在 VPS 端安装 `[[Sing-box]]`，清空默认配置并粘贴包含自定义填充字节与 `[[Reality]]` 传输的配置文件（监听 43 端口），重启服务；在客户端导入配置文件并启用 [6, 7]。
   - 关键代码/命令：未明确提及 [6, 7]。
   - 注意事项：因 `[[Xray]]` 不支持 ANTLS，且 `[[Shadowrocket]]` 无法将传输安全修改为 `[[Reality]]`，此方案必须使用 `[[Sing-box]]` 搭建与连接 [6, 7]。

## 常见坑与解决方案
- **问题描述**：使用未加密的协议节点进行测速时，数据可能会被防火墙拦截，导致测不出延迟 [2]。
  - 解决方案：未明确提及 [2]。

- **问题描述**：部分代理协议的非常规组合（如 SS+`[[Reality]]`）以及去除 TLS 的 `[[Trojan]]`，在 `[[v2rayN]]` 或主流代理工具中不提供配置选项，或不允许关闭传输安全 [4, 8]。
  - 解决方案：手动编写配置文件，直接在命令行使用内核（如 `[[Xray]]`）启动 [4, 8]。

- **问题描述**：使用代理发送 `[[UDP]]` 数据时，数据不经过底层 `[[Reality]]` 加密，如果是访问被墙网站则可能被阻断 [5, 8]。
  - 解决方案：需在协议本身设置加密方式，将本地与服务端的 SS 协议加密方式同步修改为 `[[AES]]` 或其他支持的加密方式 [8]。

- **问题描述**：搭建 AnyReality 时，iPhone 用户因 `[[Sing-box]]` 1.12 正式版未发布，可能无法正常配置客户端 [7]。
  - 解决方案：等待正式版发布，或申请成为 `[[Sing-box]]` 的 TestFlight 用户 [7]。

## 最佳实践
- 进行裸奔协议抓包实验有封锁风险，仅作为分析学习使用 [2]。
- 对延迟敏感的游戏场景，如果 VPS 线路质量好且游戏无敏感数据，直接使用未加密的原生 `[[UDP]]`（不经过传输层加密）延迟会更低；但如果线路烂，原生的效果可能还不如套用代理 [8]。
- 配置 AnyReality 服务端时，切忌直接使用参考配置，必须将各项参数（如密码、端口等）修改为个人的专属设置 [7]。

## 关键概念

### [[代理协议]]
- **定义**：处理需要代理的数据的最上层协议。[1]
- **视频上下文**：作者将其作为代理节点数据处理的第一部分，常见的如 SS、Vmess、Trojan 等均属于这一层，负责初始的数据封装或加密。[1]
- **为什么重要**：它是代理节点配置的基础核心，决定了客户端与服务端之间数据交互的底层结构。
- **关联概念**：未明确提及

### [[传输方式]]
- **定义**：将上层代理协议处理后的数据进行实际网络传输的方法。[1]
- **视频上下文**：作为代理数据处理的第二部分，作者列举了 TCP/RAW、WS、gRPC 等传输方式，并强调它通常不对数据进行加密，只负责把数据转交给通信双方。[1, 2]
- **为什么重要**：决定了数据在网络中流动的具体握手形态和通信信道类型。
- **关联概念**：扩展 → [[WS]]

### [[传输安全]]
- **定义**：负责对底层数据传输进行加密的第三部分协议。[1]
- **视频上下文**：作者解释如果第一层代理协议未加密，可以交由第三层的传输安全（如 TLS、Reality）来处理加密和流量伪装。[1]
- **为什么重要**：是保护代理数据免遭防火墙深度包检测（DPI）识别和拦截的关键防线。
- **关联概念**：扩展 → [[TLS]] / [[Reality]]

### [[Reality]]
- **定义**：一种无需自行配置域名和证书，可直接借用其他网站有效证书的传输安全协议。[1]
- **视频上下文**：作者通过抓包展示，将协议套上 Reality 后，流量特征会伪装成正常通过 HTTP 访问其他白名单网站（如雅虎），且由于 Reality 提供了加密，内部代理协议无需重复加密。[1, 3]
- **为什么重要**：大幅降低了安全代理的搭建门槛（免域名证书），同时提供了强有力的抗封锁伪装特性。
- **关联概念**：对比 ↔ [[TLS]]

### [[TLS]]
- **定义**：一种常用于代理节点中加密通信和伪装流量的传输安全协议，通常依赖私有域名和证书。[1]
- **视频上下文**：作者指出 Trojan 等协议默认被设计为必须套用 TLS，但也演示了如何通过直接跑内核来脱掉 TLS 实现“裸奔”。[4, 5]
- **为什么重要**：是现代安全代理协议的主流底层加密依赖，但也常常引发“TLS in TLS”的特征痛点。[1, 5]
- **关联概念**：对比 ↔ [[Reality]]

### [[SS]]
- **定义**：一种支持自定义加密方式（或不加密）且结构清晰的代理协议。[6, 7]
- **视频上下文**：作者通过抓包演示了不加密的 SS（可直接看到目标域名和端口等协议头结构）和 AES 加密的 SS，并演示了 SS 与 WS 及 Reality 搭配时的不同流量特征。[2, 3, 6, 7]
- **为什么重要**：作为经典的代理协议，其灵活性使其成为演示任意协议搭配任意传输/安全层组合的绝佳样本。
- **关联概念**：属于 → [[代理协议]]

### [[WS]]
- **定义**：一种通过 HTTP 协议升级建立的传输方式。[1, 2]
- **视频上下文**：作者在演示 SS+WS 组合抓包时指出，连接建立初期会发送 HTTP 请求进行升级（服务端返回 101 状态码），随后在此连接内传输代理数据，且 WS 本身不关心传输内容的具体结构。[2]
- **为什么重要**：能够将代理流量有效伪装成正常的 Web 实时通信流量，是目前最常用的传输方式之一。
- **关联概念**：属于 → [[传输方式]]

### [[Trojan]]
- **定义**：一种默认设计为必须套用 TLS 传输安全的代理协议。[4, 5]
- **视频上下文**：作者提到常规代理工具（如 V2rayN）不允许关闭其传输安全层，但通过直接运行底层内核，作者成功抓包演示了未套 TLS 的“裸奔” Trojan 的原始数据结构（如包含哈希计算的密码验证等）。[4, 5]
- **为什么重要**：打破了该协议必须绑定 TLS 的固定认知，展示了代理架构分层解耦的原理。
- **关联概念**：依赖 → [[TLS]]

### [[AnyReality]]
- **定义**：指打破协议默认限制，将任意代理协议（或剥离了默认 TLS 的协议）与 Reality 传输安全层强行组合的代理搭建模式。[1, 5, 6]
- **视频上下文**：作者提出并演示了这一概念，通过使用 Sing-box 核心，将原本需要套 TLS 的协议换成 Reality，从而实现了既免除证书配置又规避特定痛点的自组节点。[5, 6, 8]
- **为什么重要**：代表了一种突破常规客户端操作面板限制、利用底层内核实现高度定制化伪装的高阶节点搭建思路。
