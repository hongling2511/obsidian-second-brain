## 一句话总结
讲者[[不良林]]讲解了从 [[IPv4]] 枯竭到 [[IPv6]] 普及的背景，并演示了如何通过配置网络设备获取公网 IPv6 地址。

## 目标受众与前置知识
- **适合谁看**：受困于无公网 IP、想要实现外网访问内网设备或解决内网穿透问题的网络用户。
- **前置知识**：[[IPv4]] 和 [[私有IP]] 基础概念、[[光猫]]与[[路由器]]的基本设置操作、[[二进制]]与[[十六进制]]换算（推断）。

## 核心观点
1. **[[NAT]] 技术只是缓解 [[IPv4]] 地址枯竭的缓兵之计，且破坏了网络对等连接。**
   - 为什么重要：解释了为什么大多数用户的电脑需要依靠多级内网穿透才能对外提供服务。
   - 依据：讲者指出美国有 3 亿人口拥有 15 亿多 IPv4 地址，而我国 14 亿人口仅有 3 亿多（讲者原话）。由于地址不够，家庭网络处于运营商的大内网中，必须通过多层 [[NAT]] 转换，无法被外部直接访问。
2. **[[IPv6]] 拥有极其庞大的地址池，能为每台设备分配公网 IP，消除对 [[NAT]] 的依赖。**
   - 为什么重要：拥有公网 IP 后，设备能直接互相访问，降低通信复杂性，实现真正的万物互联。
   - 依据：讲者表明 [[IPv6]] 的地址总数量是 2 的 128 次方（讲者原话），数量大到足以让所有网络设备分配到一个或多个公网 IP 直连互维。
3. **当前处于 [[IPv4]] 向 [[IPv6]] 过渡期，存在兼容性与安全性问题，不建议无特殊需求者开启。**
   - 为什么重要：提醒用户盲目开启 IPv6 可能导致部分网页打不开或带来网络被黑客入侵的风险。
   - 依据：讲者举例 [[百度]] 目前不支持 IPv6，[[Cloudflare]] 的防火墙现阶段按 /128 封锁 IPv6 并不合理；同时因为去除了 [[NAT]]，设备公网裸奔，需要依靠自身防火墙和鉴权来避免类似 [[clash]] 未鉴权导致的安全问题。
4. **[[光猫]]采用路由模式会大概率阻碍 [[IPv6]] 前缀的下发，必须改为桥接模式。**
   - 为什么重要：这是普通家庭成功获取和分配公网 IPv6 地址的核心操作前提。
   - 依据：讲者演示发现，当光猫负责拨号时（路由模式），路由器下接的电脑很难获取公网 IPv6 地址，必须让光猫转为桥接模式，由 [[路由器]] 负责 [[pppoe]] 拨号以获取前缀委派。

## 概念关系
- [[IPv4]] 枯竭 → [[NAT]]：因为公网 IPv4 数量不足，不得不采用网络地址转换技术供局域网设备上网。
- [[IPv6]] ↔ [[IPv4]]：IPv6 具有 128 个比特位，地址空间远大于 32 比特位的 IPv4，两者目前互不兼容，需通过双栈环境共存。
- [[光猫]]桥接模式 → [[路由器]]获取 [[IPv6]] 前缀：光猫桥接后交由路由器 [[pppoe]] 拨号，方可成功获得运营商下发的 IPv6 前缀（PD）。
- 移除 [[NAT]] → 设备安全风险升高：因为有了公网 [[IPv6]] 地址，设备可被直连，失去了 NAT 的遮掩，需自主强化防火墙与鉴权。

## 详细笔记
### [[IPv4]] 现状与 [[NAT]] 的痛点
- **基础原理**：常见的 192.168.0.x 属于私有 IP，无法直接访问互联网。通常设备向 [[路由器]] 请求访问 [[百度]]（假设 IP 为 6666）时，路由器通过 [[NAT]] 技术将私有 IP 转换为运营商分配的公网 IP（如 3333）并记录，再发往互联网。
- **痛点**：因为中国 14 亿人口仅有 3 亿多 IPv4 地址（讲者原话），用户大多身处运营商的大内网。不仅做端⼝转发、[[udp]] 打洞、[[dmz]] 等穿透行为困难，且破坏了对等连通性。如果别人想访问你建的网站或私服，会被运营商的多层 [[NAT]] 阻挡。

### [[IPv6]] 基础与通信流程
- **地址规范**：[[IPv6]] 拥有 2 的 128 次方（讲者原话）的地址量。用冒号分 [[十六进制]] 表示，总计 128 个比特。连续的零可省略，多段连续的零可用双冒号代替（仅限一处以防歧义）。
- **结构定义**：地址分为前缀和接口 ID（如 /64、/56、/48 表示前缀长度）。虽然规范接口 ID 为 64 比特，但有部分 [[vps]] 商家会划分为 96 或 112。以 2 开头的多为公网 IP，以 f 开头（推断：指 fe80 等）类似私有 IP。
- **通信流程**：[[路由器]]进行 [[pppoe]] 拨号不仅获取 [[IPv4]]，还获取 [[IPv6]] 公网 IP 及一段前缀（PD）。电脑会自动生成同前缀的 IPv6 地址并获取 [[DNS]]。客户端发起请求时会向 DNS 查询，若优先拿到 [[IPv6]] 解析，将跳过 [[NAT]] 直接进行端到端通信。

### 开启 [[IPv6]] 面临的挑战与阻碍
- **生态不兼容**：由于 [[IPv6]] 和 [[IPv4]] 不兼容，当前网络多使用双栈策略。一些纯 [[IPv4]] 网站（如 [[百度]]）或混有 IPv4 资源（如 JS 脚本）的页面在纯 IPv6 环境中会加载失败。
- **网络封锁**：部分网站如 [[Cloudflare]] 防火墙采取了一刀切的 /128 封锁策略；[[小米]] 等部分品牌路由器默认开启防火墙且无法关闭，阻断所有 IPv6 入站；运营商可能屏蔽 80、443 等常用端口。
- **安全与工具问题**：暴露在公网会导致攻击风险提升，不配置鉴权（如未鉴权的 [[clash]]）极度危险。同时在透明代理环境下可能导致 DNS 劫持或分流规则失效。

### 设备 [[IPv6]] 设置与验证教程
- **环境检查**：在电脑打开 [[cmd]]，输入 `ipconfig`，如果有 2 开头的 IPv6 地址代表已启用；若为空白需在网卡属性勾选开启；若只有 f 开头的本地地址则需要配置上游设备。
- **[[光猫]]与网络修改**：
  - 用超级管理员账号（如 [[电信]] 宽带）进入光猫后台。
  - 记下宽带账号密码，将负责拨号的路由模式修改为“桥接模式”（Bridge），并将 IP 模式设置为 IPv4 和 IPv6 并行。
- **[[路由器]]拨号配置**：
  - 对于 [[tplink]] 或 [[小米]]：改为 [[pppoe]] 拨号并在设置中开启 IPv6，方式选 Native。
  - 对于 [[openwrt]] 路由器：将 WAN 口协议改为 [[pppoe]] 拨号；新建名为 WAN6 的接口，协议选 [[dhcpv6客户端]]，自定义接口填 `@wan` 以获取 PD 前缀；在 LAN 接口中将 IPv6 分配长度设为 64 即可下发公网 IP。

## 常见坑与解决方案
- **问题描述**：由于 IPv6 与 IPv4 不兼容，纯 IPv6 环境会导致不支持 IPv6 的网站打不开，或网页内包含的 IPv4 资源（如图片、JS 脚本）加载失败 [11]。
  - 解决方案：使用双栈模式，同时使用 IPv4 和 IPv6 访问网站，支持的优先走 IPv6，不支持的使用 IPv4 [11]。
- **问题描述**：光猫采用路由模式负责拨号时，IPv6 前缀委托（PD）大概率无法下发到路由器 LAN 口，导致下级设备获取不到公网 IP [12]。
  - 解决方案：将光猫改为桥接模式；若因特殊原因必须用光猫拨号，尝试自行搜索 IPv6 中继或使用 NAT6 解决 [9, 12]。
- **问题描述**：运营商屏蔽家庭网络的常见端口（如 80、443），导致搭站或特定服务（如免流节点）不可用 [4]。
  - 解决方案：搭建服务需避开这些端口（视当地运营商情况，个别地区可能未封 53 端口） [4]。
- **问题描述**：获取公网 IP 后失去 [[NAT]] 保护，设备直接对外提供服务面临安全问题 [4]。
  - 解决方案：自己做好防护，如打开电脑防火墙、关闭不需要的服务、敏感服务开启鉴权（如 [[clash]] 需开鉴权防止被滥用） [4]。
- **问题描述**：透明代理工具对 IPv6 支持不好，可能导致 DNS 劫持失败、分流规则失效等不兼容状况 [5]。
  - 解决方案：若无特殊需求不建议开启 IPv6；若已开启并出现奇怪问题，可通过反向操作关闭 IPv6 [5]。

## 最佳实践
- 光猫强烈建议使用调解（桥接）模式并由下级路由器拨号，以确保 IPv6-PD 能够正确下发并为各网络设备分配公网 IP 地址 [9, 12]。
- 现阶段由于处于 IPv4 向 IPv6 过渡的初期，最佳网络配置为双栈使用（支持 IPv6 的优先使用，不兼容的退回 IPv4）以保证正常访问 [11]。
- 建立对外的敏感服务时，必须抛弃以往依赖 [[NAT]] 阻挡外部访问的习惯，主动开启服务鉴权并合理配置防火墙以保障安全 [4]。

## 关键概念

### [[IPv6]]
- **定义**：一个不需要 NAT 的互联网世界协议，能为每一台网络设备分配一个或多个公网 IP，实现万物互联 [1]。
- **视频上下文**：作者指出它的出现主要是为了解决 IPv4 地址不够用的问题，其地址数量极大，使用冒号分16进制表示 [2, 3]。
- **为什么重要**：它是解决当前网络因为 NAT 破坏对等连接、导致设备无法被外部直接访问的核心技术方案 [2-4]。
- **关联概念**：对比 → [[IPv4]]

### [[IPv4]]
- **定义**：目前广泛使用但总地址数量仅有40多亿、目前已经面临分配耗尽的网络通信协议 [2]。
- **视频上下文**：视频提及由于 IPv4 地址不够给每台设备分配公网 IP，导致大部分用户家里没有公网 IP，不得不依赖 NAT 技术 [2]。
- **为什么重要**：理解其地址枯竭的局限性是理解互联网为什么必须向 IPv6 过渡的根本原因 [2]。
- **关联概念**：对比 → [[IPv6]]

### [[NAT]]
- **定义**：网络地址转换技术，用于在设备访问互联网时将私有 IP 修改转换为路由器或运营商的公网 IP [1, 5]。
- **视频上下文**：作者详细解释了路由器如何通过 NAT 修改数据包的源 IP 来实现内网设备上网，以及返回数据时再转回私有 IP 的过程 [5, 6]。
- **为什么重要**：它虽然作为“缓兵之计”解决了 IPv4 地址不足的问题，但破坏了网络的对等连接，增加了通信复杂性和设备负担 [1, 2]。
- **关联概念**：依赖 → [[私有 IP]]

### [[公网 IP]]
- **定义**：可以在互联网上直接进行相互对等访问，而不需要经过 NAT 处理的 IP 地址 [1, 3, 6]。
- **视频上下文**：视频提到如果要访问互联网就必须使用公网 IP，有了它设备就可以直接转发数据给互联网，直接对外部提供服务 [1, 6, 7]。
- **为什么重要**：拥有它是实现设备对外提供服务（如建网站、开私服）而不需内网穿透的必要条件 [4, 6, 7]。
- **关联概念**：对比 → [[私有 IP]]

### [[私有 IP]]
- **定义**：只能在局域网设备之间使用的 IP 地址，无法直接用于互联网通信 [1]。
- **视频上下文**：作者举例 192.168.0.2 就是一个常见的私有 IP，外网其他人根本无法通过这个地址访问到你的设备 [6, 8]。
- **为什么重要**：解释了为什么多数家庭设备默认处在内网中，且被屏蔽在直接的互联网访问之外 [6]。
- **关联概念**：对比 → [[公网 IP]]

### [[接口 ID]]
- **定义**：IPv6 地址中后缀部分（按照规范应固定为后64个比特位）的设备标识 [9]。
- **视频上下文**：作者提到在运营商分配的 IPv6 前缀不变的情况下，接口 ID 可以随意修改，从而为电脑网卡手动添加任意数量的公网 IPv6 地址 [9, 10]。
- **为什么重要**：它是实现“单设备分配无限个公网 IPv6 地址”这一操作的核心机制 [10, 11]。
- **关联概念**：未明确提及

### [[PD]]
- **定义**：即前缀委托（Prefix Delegation），是运营商下发给路由器的一段 IPv6 地址前缀 [10, 12]。
- **视频上下文**：视频在演示 OpenWrt 路由器配置时提到了 ipv6-pd 选项，并说明路由器会根据这个前缀向下级设备（如电脑）自动生成同前缀的公网 IPv6 地址 [10, 12]。
- **为什么重要**：如果没有成功下发 PD，连接路由器的下级设备就无法获取到公网 IPv6 地址 [12, 13]。
- **关联概念**：依赖 → [[桥接模式]]

### [[双栈]]
- **定义**：电脑或设备同时使用 IPv4 和 IPv6 两种协议访问网站的运行状态 [14]。
- **视频上下文**：作者指出因为 IPv6 和 IPv4 不兼容，且很多网站仍依赖 IPv4 资源，所以现在都是双栈使用，支持 IPv6 就优先使用，不支持则使用 IPv4 访问 [14]。
- **为什么重要**：它是解决当前 IPv4 到 IPv6 过渡初期阶段协议不兼容、资源加载失败等问题的实际应用方案 [14]。
