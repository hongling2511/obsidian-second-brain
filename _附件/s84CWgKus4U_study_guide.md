# OpenClash Mastery and Networking Architecture Study Guide

This study guide provides a comprehensive overview of OpenClash, a powerful OpenWrt plugin designed for high-performance transparent proxying. It covers installation procedures, core networking modes, DNS optimization, and the technical mechanics of traffic routing based on the provided source context.

---

## I. Glossary of Key Terms

| Term | Definition |
| :--- | :--- |
| **OpenClash** | A specialized proxy client for OpenWrt routers used to manage network traffic, reduce latency, and provide "no-pollution" DNS resolution. |
| **Kernel (Clash)** | The core engine of OpenClash. Variants include the "Premium" version and the "Meta" version (which includes enhanced DNS features). |
| **Fake-IP Mode** | A DNS resolution method where the router returns a placeholder (fake) IP to the client immediately, resolving the real IP only when the traffic reaches the proxy core. |
| **Redirect** | A firewall-based method that intercepts TCP traffic and redirects it to a specific local port (e.g., 7892) for processing. |
| **TProxy** | Transparent Proxy; a module that can handle both TCP and UDP traffic without modifying the packet's destination IP, relying instead on packet marking and routing tables. |
| **TUN Mode** | A mode that creates a virtual network interface (`utun`) to intercept traffic at the routing level, bypassing the need for complex firewall redirection rules. |
| **nftables / iptables** | Firewall management tools in Linux. OpenWrt (version 23.05+) uses `nftables` by default to handle packet filtering and redirection. |
| **IP Forwarding** | A kernel setting (usually set to `1`) that allows a device to pass network traffic from one interface to another, essential for router functionality. |
| **Subscription Conversion** | The process of converting a provider’s node list into a Clash-compatible configuration format, often performed via online or self-hosted API services. |

## II. Key Concepts and Technical Architectures

### 1. Initial Setup and Kernel Management
Successful OpenClash deployment requires a matching kernel for the router's architecture.
*   **Automatic Update:** Users can check for and update kernels directly via the "Version Update" tab. If the default download URL is blocked, a GitHub mirror or alternative URL must be configured.
*   **Manual Upload:** For environments with no initial internet access, kernels can be downloaded to a local PC and uploaded via the "Configuration Management" tab. Filenames must not contain parentheses.

### 2. DNS Optimization and Leakage Prevention
DNS leakage occurs when requests are sent to domestic DNS servers, potentially exposing browsing habits.
*   **Standard Configuration:** Disabling all default DNS servers and using "Custom Upstream DNS" within OpenClash settings.
*   **Whitelist Mode:** Advanced users can use specific configuration templates to ensure only domestic traffic uses direct connection rules, while all other traffic is routed through the proxy to prevent leaks.
*   **UDP Handling:** Some UDP-based requests (like Windows Time Sync) may fail if the proxy node does not support UDP. These domains should be added to a "Direct" rule list.

### 3. Comparison of Transparent Proxy Modes

| Mode | Primary Mechanism | Traffic Support | Key Characteristic |
| :--- | :--- | :--- | :--- |
| **Redirect** | Firewall DNAT | TCP Only | Efficient for TCP; fails for UDP because it cannot retrieve the original destination IP. |
| **TProxy** | Packet Marking (`mark 354`) | TCP + UDP | Does not modify packet headers; requires Table 100 routing rules. |
| **TUN** | Virtual Network Card (`utun`) | TCP + UDP | Uses routing rules instead of firewall hooks; supports `system` or `gvisor` stacks. |
| **Mixed** | Hybrid (Redir + TProxy/TUN) | TCP + UDP | Uses Redirect for TCP performance and TProxy/TUN for UDP compatibility. |

### 4. Firewall Chain Logic
OpenClash manipulates various chains within the firewall to manage data packets:
*   **PREROUTING:** Where DNS hijacking (port 53) and initial traffic redirection occur.
*   **INPUT:** Handles traffic destined for the router itself (e.g., traffic redirected to Clash's listening port).
*   **OUTPUT:** Intercepts traffic generated by the router itself to ensure it is also proxied.
*   **FORWARD:** Manages traffic passing through the router from local network devices to the internet.

## III. Short-Answer Practice Questions

**1. Why is the "Redirect" mode insufficient for handling UDP traffic?**
*Answer:* UDP is connectionless. When the `Redirect` rule modifies the destination IP to the local router’s IP, the original destination information is lost, making it impossible for the proxy to know where to send the packet.

**2. What is the significance of "Table 100" in TProxy configuration?**
*Answer:* TProxy marks packets with a specific identifier (e.g., `mark 354`). Table 100 is a dedicated routing table that instructs the system to route any packet with that mark to the local loopback interface so the Clash core can process it.

**3. What should a user do if the kernel download fails during the initial setup?**
*Answer:* The user should navigate to "Developer Options" or "Overriding Settings" to change the download URL to a mirror site, or manually download the kernel and upload it through the "Configuration Management" interface.

**4. How does TUN mode differ from firewall-based redirection?**
*Answer:* TUN mode creates a virtual network interface (`utun`). Instead of using firewall rules to "hijack" ports, it uses high-priority routing rules to direct traffic into the virtual interface, where Clash reads the raw IP packets directly.

**5. How can a user verify if OpenClash is successfully running?**
*Answer:* The main interface will display a green "Running" status. Additionally, the user can check the "Logs" page for a "successful start" message or access the web control panel (Yacd or Dashboard) to see active connections.
