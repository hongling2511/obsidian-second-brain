# NaiveProxy Node Construction and Security Analysis Study Guide

This study guide provides a comprehensive overview of NaiveProxy (referred to as "Naive"), an advanced solution for constructing private network nodes designed to achieve maximum stealth. It covers the technical principles of Naive, the importance of TLS fingerprinting, implementation procedures for both server and client, and the security advantages over traditional proxy protocols.

---

## 1. Technical Principles and Core Concepts

### The NaiveProxy Philosophy
NaiveProxy is characterized as an "ultimate" solution for node construction because it prioritizes the complete concealment of proxy characteristics. While common proxy protocols are often identifiable by firewalls, Naive aims to make traffic indistinguishable from standard web browsing.

### Key Security Features
*   **Chrome Network Stack:** The Naive client utilizes the Chromium network stack. From the perspective of a firewall, the traffic appears identical to a user browsing the web via a standard Google Chrome browser.
*   **Elimination of TLS Fingerprints:** By using the Chrome kernel, Naive eliminates unique client-side TLS fingerprints (JA3) that often reveal the use of proxy software like v2ray or Clash.
*   **Mitigation of TLS-in-TLS Features:** Standard nodes often suffer from "TLS-in-TLS" handshake characteristics (where a TLS-encrypted proxy tunnel carries TLS-encrypted website data). Naive uses self-padding to eliminate these identifiable double-handshake features.
*   **HTTP/2 Multiplexing:** Based on the HTTP/2 protocol, Naive supports native multiplexing. This results in lower latency compared to WebSocket (WS) protocols, which require frequent handshakes.
*   **Automated Web Camouflage:** The server-side implementation uses Caddy, which automatically manages SSL certificates and serves a legitimate website to any unauthorized probe, making the server behave like a standard web server.

## 2. Server-Side Construction and Configuration

### Environment and Prerequisites
Construction requires a VPS (typically running Ubuntu 22.04), a registered domain name, and DNS resolution pointing the domain to the VPS IP address.

### The Role of Caddy
The server-side component is essentially a version of **Caddy** (a web server similar to Nginx) compiled with a specific Naive plugin. Caddy is preferred for its simplicity and automated certificate management.

### Construction Steps
1.  **Install Go Language:** Required to compile the Caddy environment.
2.  **Download and Compile:** Fetch the necessary files and compile Caddy with the NaiveProxy plugin.
3.  **Configure the Caddyfile:** Create a configuration file (named `Caddyfile`) with the following components:
    *   **Domain Name:** The resolved domain for the node.
    *   **Email:** For managing SSL certificates.
    *   **Credentials:** A username and password (separated by a space).
    *   **Forward Proxy Target:** A legitimate website URL to serve as camouflage.
4.  **Execution:** Run Caddy. Use the `run` command initially to monitor certificate acquisition, then use the `start` command for background operation.

## 3. Client-Side Implementation

Naive lacks a native graphical user interface (GUI), operating primarily via command line. To improve usability, it can be integrated into existing proxy managers.

### Integration with v2rayN
*   Download the Naive kernel (matched to the OS version).
*   Place the `naive.exe` kernel into the v2rayN directory.
*   Configure a JSON file specifying the local SOCKS proxy port (e.g., 1080) and the server details (domain, username, and password).
*   Add a "Custom Configuration" server in v2rayN, selecting the `naiveproxy` core.

### Cross-Platform Support
*   **iOS:** Supports Naive via Shadowrocket (selecting HTTPS or HTTP/2 protocol types).
*   **Android:** Supports Naive via SagerNet with an additional NaiveProxy plugin.

## 4. Deep Dive: TLS Fingerprinting (JA3, JA3S, and JARM)

A significant portion of Naive's security relies on manipulating TLS fingerprints to avoid detection.

### Understanding TLS Handshakes
When establishing a TLS connection, the client sends a `Client Hello` packet containing supported versions and encryption suites. The server responds with a `Server Hello`. This negotiation generates unique identifiers.

| Fingerprint Type | Definition | Significance |
| :--- | :--- | :--- |
| **JA3** | Client-side TLS fingerprint generated from the `Client Hello` packet. | Can identify specific software (e.g., Chrome vs. a proxy tool). |
| **JA3S** | Server-side TLS fingerprint generated from the `Server Hello` response. | Less reliable due to high repetition rates among different servers. |
| **JARM** | An active probing fingerprint generated by sending multiple specific TLS probes to a server. | Highly precise; can distinguish between web servers (Caddy/Nginx) and proxy cores (Xray/v2ray). |

### The "Behavioral" Defense
Naive nodes use Caddy to ensure that the JARM fingerprint matches that of a standard web server. Probing a Naive node yields a JARM value associated with thousands of legitimate Caddy-hosted websites, whereas standard Xray or v2ray nodes often return JARM values that are statistically rare and concentrated in specific regions or hosting providers (e.g., Bandwagon Host), making them easy targets for firewalls.

## 6. Glossary of Terms

| Term | Definition |
| :--- | :--- |
| **Caddy** | A modern web server written in Go that features automatic HTTPS and serves as the server-side host for Naive. |
| **Chromium Stack** | The network communication code used by the Chrome browser; used by Naive to mimic legitimate browsing traffic. |
| **HTTP/2 Multiplexing** | A feature allowing multiple requests and responses to be sent concurrently over a single TCP connection, reducing latency. |
| **JARM** | An active TLS server fingerprinting tool that identifies the specific implementation of a server's TLS stack. |
| **Multiplexing** | The method by which multiple signals or streams of information are combined into one signal over a shared medium. |
| **Self-Padding** | The process of adding extra data to a packet to obscure the length and characteristics of the underlying encrypted traffic. |
| **TLS Fingerprint** | A unique hash value derived from the parameters exchanged during a TLS handshake, used to identify the software involved. |
| **TLS-in-TLS** | A detectable pattern where one layer of TLS encryption is wrapped inside another, common in traditional proxy setups. |
